## 需求
### 需求：执行完整素材整理 Pipeline
系统必须执行完整的素材整理流程：扫描 → 处理 → 录入 → 向量化。

#### 场景：成功执行完整 Pipeline
- **当** 素材整理任务触发（定时/手动）
- **那么** 系统依次执行：扫描目录、处理新素材、录入数据库、生成向量

#### 场景：并发控制
- **当** 素材整理任务正在运行
- **那么** 系统跳过本次触发，记录警告日志

#### 场景：错误恢复
- **当** Pipeline 中某个素材处理失败
- **那么** 系统记录错误，继续处理其他素材，不中断整个流程

### 需求：统计和报告
系统必须生成素材整理执行统计，包括成功数、失败数、跳过数。

#### 场景：生成执行报告
- **当** 完成一次素材整理
- **那么** 系统输出统计：扫描总数、新增数、更新数、失败数、总耗时

#### 场景：详细错误日志
- **当** 有素材处理失败
- **那么** 系统记录每个失败素材的文件路径和错误原因

### 需求：支持定时和手动触发
系统必须支持多种触发方式：定时任务、Web 界面手动触发、API 调用。

#### 场景：定时任务触发
- **当** 到达配置的调度时间
- **那么** 系统自动执行素材整理

#### 场景：Web 界面手动触发
- **当** 用户在 Web 界面点击"素材整理"按钮
- **那么** 系统立即执行一次素材整理

#### 场景：API 调用触发
- **当** 调用 POST /api/materials/organize 接口
- **那么** 系统执行素材整理并返回执行结果

### 需求：配置支持
系统必须支持通过配置文件自定义素材整理行为。

#### 场景：自定义扫描路径
- **当** 配置 materials.rawPath
- **那么** 系统扫描指定路径而非默认路径

#### 场景：启用/禁用整理
- **当** scheduler.materialProcessing.enabled = false
- **那么** 系统跳过素材整理任务

#### 场景：间隔模式配置
- **当** scheduler.materialProcessing.intervalMinutes = 30
- **那么** 系统每隔 30 分钟执行一次素材整理
