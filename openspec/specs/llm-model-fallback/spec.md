## 新增需求

### 需求:系统必须支持配置多个有序的 LLM 提供商
配置文件必须支持 `ai.providers` 数组字段，每个元素声明一个 LLM 提供商的完整连接参数，数组顺序即为调用优先级。

#### 场景:配置多提供商列表
- **当** 配置文件中存在 `ai.providers` 数组
- **那么** 系统必须按数组顺序将第一个元素作为主力提供商，后续元素作为降级备选

#### 场景:兼容旧版单一配置格式
- **当** 配置文件中不存在 `ai.providers` 数组，但存在顶层 `ai.apiKey`、`ai.baseUrl`、`ai.model` 字段
- **那么** 系统必须将这些字段自动归一化为单元素的 `providers` 数组，确保调用逻辑不受影响

#### 场景:提供商配置参数完整性
- **当** 声明一个提供商条目时
- **那么** 该条目必须包含 `name`、`apiKey`、`baseUrl`、`model` 字段，`temperature`、`maxTokens` 为可选字段，缺省时继承全局默认值；由于所有提供商均为标准 OpenAI 兼容接口（如 HiGPT 通过网关代理），不需要 `queryParams` 等额外连接参数

### 需求:AI 客户端必须按优先级自动 Fallback 到备用提供商
当主力 LLM 提供商调用失败时，系统必须自动切换到下一个提供商重试，无需人工介入。

#### 场景:主力提供商调用成功
- **当** 主力提供商（`providers[0]`）在重试范围内返回有效内容
- **那么** 系统必须使用该响应，不触发 fallback

#### 场景:主力提供商所有重试均失败时切换备用
- **当** 主力提供商经过最多 2 次重试后仍失败（网络错误、HTTP 错误或空响应）
- **那么** 系统必须自动切换到 `providers[1]`（若存在）重新发起调用，并在日志中记录 fallback 事件，包含失败提供商的 baseUrl 和错误信息

#### 场景:所有提供商均失败
- **当** `providers` 数组中所有提供商均已尝试并失败
- **那么** 系统必须抛出包含所有提供商错误信息的聚合错误，并在日志中以 `error` 级别记录"所有 LLM 提供商均不可用"

#### 场景:仅有单一提供商时的行为
- **当** `providers` 数组仅有一个元素
- **那么** 系统行为必须与当前单提供商模式完全一致（最多重试 2 次后抛出错误）

### 需求:系统必须记录 LLM 提供商使用和切换事件
每次 LLM 调用必须在日志中标明所使用的提供商，切换事件必须被显式记录。

#### 场景:正常调用日志
- **当** 成功从某提供商获取响应
- **那么** 日志必须以 `info` 级别记录：使用的提供商 baseUrl、模型名称、生成内容长度

#### 场景:Fallback 切换日志
- **当** 发生提供商切换
- **那么** 日志必须以 `warn` 级别记录：原提供商 baseUrl、失败原因、切换目标提供商 baseUrl
