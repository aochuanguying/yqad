## 需求
### 需求：批量同步历史素材到 ChromaDB
系统必须支持批量将 MySQL 中的历史素材同步到 ChromaDB，生成向量嵌入。

#### 场景：成功批量同步
- **当** 执行向量同步工具
- **那么** 系统遍历所有未向量化的素材，批量生成向量并添加到 ChromaDB

#### 场景：分批处理
- **当** 同步超过 100 个素材
- **那么** 系统分批次处理，每批最多 100 个，避免内存溢出

#### 场景：跳过已存在的向量
- **当** 素材在 ChromaDB 中已存在向量
- **那么** 系统跳过该素材，不重复生成

### 需求：向量化文本生成
系统必须为每个素材构建用于生成向量的文本，包含描述、标签、文件名。

#### 场景：构建向量文本
- **当** 为素材生成向量
- **那么** 系统组合 description + tags + filename 作为向量文本

#### 场景：空描述处理
- **当** 素材描述为空
- **那么** 系统仅使用文件名和标签生成向量

### 需求：同步进度跟踪
系统必须记录向量同步进度，支持中断后继续。

#### 场景：记录同步进度
- **当** 完成一批同步
- **那么** 系统记录已处理数量和当前批次

#### 场景：中断后继续
- **当** 同步过程中断后重新启动
- **那么** 系统从上次中断点继续，不重复处理

### 需求：向量更新支持
系统必须支持更新现有素材的向量表示。

#### 场景：更新描述后同步向量
- **当** 素材描述被手动更新
- **那么** 系统重新生成向量并更新 ChromaDB 记录

#### 场景：强制重新向量化
- **当** 用户触发强制同步
- **那么** 系统忽略现有向量，为所有素材重新生成向量
