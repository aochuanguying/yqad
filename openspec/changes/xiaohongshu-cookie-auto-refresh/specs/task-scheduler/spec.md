## 新增需求

### 需求：统一任务调度器
系统必须提供统一的定时任务调度器，管理所有定时任务。

#### 场景：系统启动
- **当** 应用启动时
- **那么** 系统初始化调度器
- **那么** 系统从数据库加载所有启用的定时任务
- **那么** 系统注册每个任务的 Cron 表达式

#### 场景：注册 Cookie 刷新任务
- **当** 调度器初始化
- **那么** 系统注册 Cookie 刷新任务（默认每天凌晨 2 点）
- **那么** 系统设置任务类型为"cookie-refresh"

#### 场景：注册评论任务
- **当** 调度器初始化
- **那么** 系统注册评论任务（如果启用）
- **那么** 系统设置任务类型为"auto-comment"

#### 场景：注册素材整理任务
- **当** 调度器初始化
- **那么** 系统注册素材整理任务（如果启用）
- **那么** 系统设置任务类型为"material-organize"

### 需求：任务并发控制
系统必须防止相同类型的任务并发执行。

#### 场景：任务正在运行
- **当** 定时任务触发时已有同类型任务在运行
- **那么** 系统跳过本次执行
- **那么** 系统记录"任务跳过：已有同类型任务运行中"

#### 场景：不同类型任务并行
- **当** Cookie 刷新任务和评论任务同时触发
- **那么** 系统允许两个任务并行执行
- **那么** 系统互不干扰

### 需求：任务管理 API
系统必须提供 API 接口管理定时任务。

#### 场景：查看所有任务状态
- **当** 用户调用 `GET /api/scheduler/tasks`
- **那么** 系统返回所有定时任务列表
- **那么** 每个任务包含：名称、类型、Cron 表达式、上次执行时间、下次执行时间、状态

#### 场景：手动触发任务
- **当** 用户调用 `POST /api/scheduler/tasks/:taskId/trigger`
- **那么** 系统立即触发指定任务
- **那么** 系统返回任务执行状态

#### 场景：启用/禁用任务
- **当** 用户调用 `POST /api/scheduler/tasks/:taskId/toggle`
- **那么** 系统切换任务的启用/禁用状态
- **那么** 系统更新数据库中的任务配置
