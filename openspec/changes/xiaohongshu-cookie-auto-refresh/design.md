## 上下文

**当前状态**:
- 小红书 Cookie 手动从浏览器复制，有效期 1-3 天
- Cookie 存储在环境变量中，Docker 重启后失效
- 无自动刷新机制，失效后需人工干预
- 部署环境：群晖 NAS218+ (x86_64, 2 核 2GB), Docker 容器化部署

**约束条件**:
- Docker 无头模式（无图形界面）
- 群晖资源有限（CPU/内存）
- 需要支持多容器共享 Cookie
- 首次部署需人工扫码一次

**利益相关者**:
- 运维人员：负责部署和维护
- 开发人员：使用小红书搜索功能
- 系统：依赖 Cookie 的搜索服务

## 目标 / 非目标

**目标：**
1. 实现扫码登录自动获取 Cookie（二维码图片方式，支持两次扫码）
2. 实现每 24 小时自动刷新 Cookie（与评论/素材整理任务统一调度）
3. Cookie 明文存储到 MySQL 数据库
4. 支持 Docker 无头环境部署
5. 保留前端页面手工配置 Cookie 功能
6. 提供 API 接口手动触发刷新和查看状态

**非目标：**
1. 多账号轮换（当前只需单账号）
2. 代理 IP 支持（当前无此需求）
3. 分布式 Cookie 管理（单数据库实例）
4. 其他平台（抖音/快手）的 Cookie 管理
5. Cookie 加密存储（内网环境，明文即可）

## 决策

### 1. 扫码方式：二维码图片 vs 有界面浏览器

**决策**: 二维码图片方式

**理由**:
- Docker 无头环境无法显示浏览器界面
- 二维码图片可通过文件共享访问
- 用户扫码体验相似（都是用手机扫）

**替代方案**:
- ❌ VNC 远程桌面：复杂，资源消耗大
- ❌ 有头浏览器 + Xvfb：配置复杂，不稳定

### 2. Cookie 存储：数据库 vs 文件 vs 环境变量

**决策**: MySQL 数据库明文存储

**理由**:
- 支持多容器共享（搜索服务、刷新任务）
- 支持版本管理和历史记录
- 配置简单，无需加密解密逻辑
- 便于调试和监控

**替代方案**:
- ❌ 环境变量：Docker 重启后失效，无法动态更新
- ❌ 本地文件：多容器共享复杂，权限管理麻烦
- ❌ 加密存储：增加复杂度，收益有限（内网环境）

### 3. 刷新策略：定期刷新 vs 失效前刷新 vs 失败后刷新

**决策**: 定期刷新（每 24 小时）

**理由**:
- 简单可靠，易于实现和调试
- 保持登录态活跃，延长 Cookie 有效期
- 避免失效后刷新导致的业务中断

**替代方案**:
- ❌ 失效前刷新：需要检测有效期，小红书 Cookie 无明确过期时间
- ❌ 失败后刷新：业务已中断，体验差

### 4. 浏览器自动化：Playwright vs Puppeteer vs Selenium

**决策**: Playwright

**理由**:
- 团队已有 Playwright 使用经验
- 支持无头模式，资源消耗低
- API 简洁，易于维护
- 自动等待机制，稳定性高

**替代方案**:
- ❌ Puppeteer: 只支持 Chrome，功能类似
- ❌ Selenium: API 复杂，速度慢

### 5. 加密方案：AES-256-CBC vs bcrypt vs 明文

**决策**: AES-256-CBC 加密

**理由**:
- 可逆加密（需要解密使用）
- 256 位密钥，安全性高
- Node.js 原生支持，性能好

**替代方案**:
- ❌ bcrypt: 不可逆，不适合此场景
- ❌ 明文：安全风险高

### 6. 调度器：独立调度 vs 统一调度

**决策**: 与现有定时任务统一调度（评论、素材整理）

**理由**:
- 统一管理，便于监控和维护
- 共享调度逻辑，减少代码重复
- 统一日志输出，便于排查问题
- 避免多个调度器冲突

**替代方案**:
- ❌ 独立调度器：代码重复，管理复杂
- ❌ 系统 crontab: 需要额外容器，复杂

## 风险 / 权衡

### 风险 1: 二维码被未授权访问

**风险**: 二维码图片存储在共享目录，可能被未授权访问

**缓解措施**:
- 设置目录权限（chmod 700）
- 定期清理旧二维码
- 二维码有效期设置为 5 分钟

### 风险 2: 浏览器数据卷损坏

**风险**: 挂载的浏览器数据卷��坏，导致需要重新扫码

**缓解措施**:
- 定期备份浏览器数据卷
- 使用可靠的存储（NAS RAID）
- 提供重新扫码的文档

### 风险 3: 群晖资源不足

**风险**: Playwright + Chromium 可能占用较多内存（200-300MB）

**缓解措施**:
- 设置 Docker 内存限制（512MB）
- 使用后立即关闭浏览器
- 监控内存使用，必要时升级硬件

### 风险 4: Cookie 刷新失败

**风险**: 自动刷新失败，导致服务中断

**缓解措施**:
- 失败后发送告警（日志/邮件）
- 保留旧 Cookie 继续使用
- 提供手动刷新 API

### 风险 5: 小红书风控升级

**风险**: 小红书升级风控，导致扫码登录失败

**缓解措施**:
- 使用真实用户数据目录（模拟真实用户）
- 控制登录频率（每天最多 1 次）
- 监控登录成功率

## Open Questions

1. **加密密钥管理**: 是否使用环境变量存储加密密钥？如何保证密钥安全？
   - 暂定：使用环境变量 `COOKIE_ENCRYPTION_KEY`，32 字节随机字符串

2. **刷新时间选择**: 凌晨 2 点是否合适？是否需要根据业务高峰期调整？
   - 暂定：凌晨 2 点，业务低峰期

3. **日志保留**: Cookie 刷新日志保留多久？是否需要单独建表？
   - 暂定：保留 30 天，不单独建表，使用 JSON 字段存储最近日志

4. **监控告警**: 是否需要集成现有监控系统？使用什么告警方式？
   - 暂定：先记录日志，后续集成邮件告警
