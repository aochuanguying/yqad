## 上下文

当前发帖日志系统已实现基础功能，记录了发帖的基本信息（标题、内容、状态、错误信息等）。但通过复盘发现以下问题：

**当前状态**：
- 日志记录较为基础，仅包含最终结果
- 缺少 Pipeline 各步骤的执行详情
- 错误信息不够详细，难以快速定位根因
- 没有性能指标统计，无法评估系统健康度
- 数据库表结构较简单，缺少扩展性字段

**约束**：
- 不能影响发帖主流程的性能
- 日志记录应该异步、非阻塞
- 需要保持向后兼容
- 数据库迁移需要可回滚

**利益相关者**：
- 开发团队：需要详细日志进行故障排查
- 运维团队：需要监控指标评估系统健康度
- 最终用户：间接受益于更快的故障恢复

## 目标 / 非目标

**目标：**
1. 完善日志记录维度，覆盖 Pipeline 全流程
2. 增强错误追踪能力，支持快速定位根因
3. 添加性能监控指标，支持系统健康度评估
4. 优化数据库查询性能，支持高效日志检索
5. 改进异常处理机制，提高系统鲁棒性

**非目标：**
1. 不改变发帖主流程的业务逻辑
2. 不引入新的外部依赖（如 APM 工具）
3. 不重构现有日志架构（渐进式优化）
4. 不改变 API 接口的基本使用方式

## 决策

### 1. 日志记录策略：异步 + 批量

**决策**：采用异步非阻塞方式记录日志，避免影响发帖主流程性能。

**替代方案**：
- 同步记录：简单但影响性能 ❌
- 完全移除日志：不可接受 ❌
- 异步 + 批量：平衡性能和完整性 ✅

**理由**：
- 发帖流程对延迟敏感，同步记录日志会增加 100-300ms 延迟
- 使用 `setImmediate` 或 `process.nextTick` 异步执行
- 批量写入可减少数据库连接次数

### 2. 日志数据结构：扩展字段 vs 新增表

**决策**：在现有 `post_logs` 表中添加扩展字段，保持数据结构扁平化。

**替代方案**：
- 新增 `post_log_details` 表：规范化但查询复杂 ❌
- JSON 字段存储所有扩展信息：灵活但难以索引 ❌
- 扩展字段：平衡灵活性和查询效率 ✅

**理由**：
- 发帖日志查询主要是按条件过滤，扁平结构更高效
- 新增字段可以添加索引，提升查询性能
- 保持与现有代码的兼容性

### 3. 错误堆栈存储：完整堆栈 vs 简化信息

**决策**：存储完整的错误堆栈信息，便于深度排查。

**替代方案**：
- 仅存储错误消息：信息不足 ❌
- 存储完整堆栈：信息完整但占用空间 ✅
- 分级存储：严重错误存堆栈，普通错误存消息 ⚠️

**理由**：
- 完整堆栈对排查复杂问题至关重要
- MySQL TEXT 字段可存储 64KB 数据，足够使用
- 可通过日志清理策略控制存储成本

### 4. 监控指标：实时计算 vs 预聚合

**决策**：采用实时计算方式，在查询时动态统计指标。

**替代方案**：
- 预聚合到 Redis：查询快但增加复杂度 ❌
- 实时计算：简单直接，数据量可承受 ✅
- 定时任务聚合：折中方案 ⚠️

**理由**：
- 发帖日志数据量不大（30 天滚动），实时计算性能可接受
- 避免引入额外的缓存层，降低系统复杂度
- 如需优化，可后续添加物化视图

### 5. Pipeline 埋点：手动埋点 vs AOP 切面

**决策**：采用手动埋点方式，在每个 Pipeline 步骤中显式记录。

**替代方案**：
- AOP 切面拦截：自动化但不够灵活 ❌
- 装饰器模式：优雅但需要重构代码 ❌
- 手动埋点：直接、可控、易调试 ✅

**理由**：
- Pipeline ���骤不多（8 个），手动埋点工作量可控
- 可以精确控制记录的时机和内容
- 便于添加业务特定的上下文信息

## 风险 / 权衡

### 风险 1：日志记录影响性能

**风险**：异步日志记录仍会占用 CPU 和数据库资源。

**缓解措施**：
- 使用 `setImmediate` 延迟到事件队列空闲时执行
- 批量写入减少数据库交互次数
- 设置日志记录超时（5 秒），避免阻塞过久
- 生产环境可配置日志级别，仅记录关键信息

### 风险 2：数据库表结构变更风险

**风险**：添加新字段可能导致现有查询失败或性能下降。

**缓解措施**：
- 所有新字段设置为 NULLABLE，提供默认值
- 迁移脚本包含回滚 SQL
- 在低峰期执行迁移
- 迁移前备份数据

### 风险 3：日志数据量激增

**风险**：详细日志可能导致数据量大幅增长。

**缓解措施**：
- 保持 30 天清理策略
- 错误堆栈等大字段的日志单独设置更短保留期（7 天）
- 监控数据量增长，必要时调整策略

### 风险 4：错误信息泄露敏感数据

**风险**：详细错误堆栈可能包含敏感信息（API Token、用户数据等）。

**缓解措施**：
- 实现敏感数据过滤机制
- 在记录前对错误堆栈进行脱敏处���
- 限制日志查询权限
