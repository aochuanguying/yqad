## 新增需求

### 需求：系统必须记录完整的错误堆栈

系统必须记录错误发生时的完整堆栈信息，便于深度排查问题根因。

#### 场景：记录错误堆栈轨迹

- **当** 发帖过程中发生错误
- **那么** 系统必须记录完整的错误堆栈轨迹（包括文件名、行号、函数调用链）

#### 场景：记录错误上下文参数

- **当** 发生错误时
- **那么** 系统必须记录错误发生时的上下文参数（如：当前处理的帖子标题、图片数量、配置参数等）

#### 场景：记录错误来源组件

- **当** 记录错误时
- **那么** 系统必须标识错误的来源组件（如：auto-post、post-logging-service、hybrid-material-service 等）

### 需求：系统必须对错误信息进行分类

系统必须对错误进行分类管理，便于统计分析和针对性优化。

#### 场景：按错误类型分类

- **当** 记录错误时
- **那么** 系统必须标识错误类型（网络错误、数据库错误、API 错误、合规检查错误等）

#### 场景：按错误严重程度分级

- **��** 记录错误时
- **那么** 系统必须标识错误严重程度（致命、严重、警告、提示）

#### 场景：记录错误根本原因

- **当** 发生连锁错误时
- **那么** 系统必须记录根本原因（Root Cause），而不仅是表面错误

### 需求：系统必须记录错误恢复历史

系统必须记录错误恢复的过程和结果，便于分析系统的容错能力。

#### 场景：记录自动重试历史

- **当** 系统自动重试错误操作时
- **那么** 系统必须记录每次重试的时间、结果、重试间隔

#### 场景：记录降级处理历史

- **当** 系统执行降级处理时（如精华帖降级为普通帖）
- **那么** 系统必须记录降级原因、降级前后的状态对比

#### 场景：记录人工干预历史

- **当** 需要人工干预处理错误时
- **那么** 系统必须记录干预时间、干预措施、处理结果

### 需求：系统必须对敏感信息进行脱敏

系统必须对错误信息中的敏感内容进行脱敏处理，防止信息泄露。

#### 场景：脱敏 API Token

- **当** 错误信息中包含 API Token 时
- **那么** 系统必须将其脱敏为 `Bearer ***` 或类似形式

#### 场景：脱敏用户隐私数据

- **当** 错误信息中包含用户隐私数据时
- **那么** 系统必须进行脱敏处理（如：手机号、邮箱等）

#### 场景：脱敏数据库连接信息

- **当** 错误信息中包含数据库连接字符串时
- **那么** 系统必须隐藏密码等敏感信息
