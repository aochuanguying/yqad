## 新增需求

### 需求:系统必须持久化存储待确认发帖记录
系统必须将已生成但未确认的发帖记录持久化到文件，支持容器重启后恢复和防重检查。

#### 场景：生成内容时持久化记录
- **当** 远程 API 调用 `POST /api/posts/generate` 或 `POST /api/posts/batch` 生成内容成功
- **那么** 系统将 `PendingPost` 记录写入 `data/pending-posts.json` 文件，并返回 `taskId`

#### 场景：服务启动时恢复记录
- **当** 服务启动或容器重启
- **那么** 系统读取 `pending-posts.json`，过滤过期记录（>30 分钟），加载未过期记录到内存

#### 场景：回调成功后删除记录
- **当** 客户端调用回调端点且 `success: true`
- **那么** 系统扣减次数后，从 `pending-posts.json` 中删除该记录

#### 场景：回调失败或删除记录
- **当** 客户端调用回调端点且 `success: false`，或记录过期
- **那么** 系统从 `pending-posts.json` 中删除该记录，不扣减次数

### 需求:系统必须支持发帖防重检查
系统必须基于持久化的发帖记录进行防重检查，避免发布重复内容。

#### 场景：待确认记录防重
- **当** 生成新发帖内容时
- **那么** 系统检查 `pending-posts.json` 中是否存在相同标题或内容相似度超过阈值的记录，若存在则拒绝生成

#### 场景：历史发帖防重
- **当** 生成新发帖内容时
- **那么** 系统检查最近 7 天内的 `post-history.json` 和主题 `postHistory`，若发现重复则拒绝生成

#### 场景：防重检查维度
- **当** 执行防重检查时
- **那么** 系统检查标题完全匹配、内容相似度、同一主题发帖间隔三个维度

## 修改需求

### 需求：存储策略从内存改为持久化
**修改说明：** 原设计使用内存存储待确认记录，现改为持久化文件存储，支持容器重启恢复。

#### 场景：记录存储方式
- **当** 生成发帖内容时
- **那么** 系统将记录同时保存在内存 Map 和 `pending-posts.json` 文件中，确保读写性能和持久化
