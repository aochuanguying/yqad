## 新增需求

### 需求:系统必须均衡使用多个主题
系统必须基于主题剩余额度进行加权随机选择，优先使用剩余可用次数多的主题，避免顺序使用导致的机械模式。

#### 场景:有多个可用主题时的选择
- **当** 存在多个可用主题（`useCount < maxUseCount`）
- **那么** 系统按剩余额度降序排序，使用加权随机算法选择主题，剩余额度越多的主题被选中的概率越大

#### 场景:所有主题使用次数相同
- **当** 所有可用主题的剩余额度相同
- **那么** 系统随机选择一个主题（概率均等）

#### 场景:只有一个可用主题
- **当** 只有一个主题的剩余额度大于 0
- **那么** 系统选择该主题

#### 场景:所有主题都已耗尽
- **当** 所有主题的 `useCount >= maxUseCount`
- **那么** 系统返回 null，触发自由生成模式

### 需求：远程 API 必须使用均衡选择策略
远程发帖 API 在生成内容时必须使用新的均衡选择策略，而非原有的顺序选择。

#### 场景：远程 API 生成内容选择主题（单个）
- **当** 远程 API 调用 `POST /api/posts/generate` 且 `useTopic: true`
- **那么** 系统使用 `getBalancedTopic()` 函数选择主题

#### 场景：远程 API 生成内容选择主题（批量）
- **当** 远程 API 调用 `POST /api/posts/batch` 且 `useTopic: true`
- **那么** 系统为每篇内容独立调用 `getBalancedTopic()` 函数选择主题，可能选择不同主题

#### 场景：本地自动发帖保持原逻辑
- **当** 本地定时任务调用 `performDailyPosts()`
- **那么** 系统继续使用 `getNextAvailableTopic()` 按顺序选择主题（保持向后兼容）

## 修改需求

### 需求:主题选择函数行为变更
**修改说明：** 新增 `getBalancedTopic()` 函数实现均衡选择，原有 `getNextAvailableTopic()` 保持不变用于向后兼容。

#### 场景:调用均衡选择函数
- **当** 调用 `getBalancedTopic()` 函数
- **那么** 系统执行加权随机算法，返回选中的主题或 null
