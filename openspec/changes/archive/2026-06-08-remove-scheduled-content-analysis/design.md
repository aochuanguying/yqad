## 上下文

系统当前在启动时创建 `ContentAnalysisService`，并把 `analysis` handler 注入调度器。调度器会注册“内容分析”任务，按 `scheduler.analysis` 配置定时执行 `analysisService.analyze()`，启动后的遗漏任务检查也可能补执行该任务。

内容分析结果仍被自动发帖链路使用：自动发帖在生成内容前会读取分析摘要，并在摘要缺失或过期时按需刷新。因此本变更只移除调度入口，不移除内容分析服务、分析结果文件、帖子解析和图片识别能力。

## 目标 / 非目标

**目标：**
- 系统启动后不再注册“内容分析”定时任务。
- 调度配置热重载不再处理“内容分析”任务配置。
- 启动后的遗漏任务检查不再执行“内容分析”。
- 自动发帖等业务仍可按需使用内容分析摘要。
- 签到、评论、发帖、素材梳理等其他调度任务行为保持不变。

**非目标：**
- 不删除 `ContentAnalysisService`。
- 不删除 `data/analysis.json` 或变更分析结果结构。
- 不重构自动评论和自动发帖的内容生成策略。
- 不移除 `PostParser`、OCR、图片描述或素材识别能力。

## 决策

1. 仅从调度层移除“内容分析”任务。

   这样可以直接消除周期性执行和遗漏任务补执行，同时保留按需分析能力。相比删除整个服务，此方案风险更低，不会破坏自动发帖依赖的摘要兜底逻辑。

2. 从 `createScheduler` 的 handler 契约中移除 `analysis`。

   调度器不再需要接收内容分析 handler，启动入口也不再需要提供对应函数。这可以避免未来误注册“内容分析”任务，也让类型约束反映真实调度任务集合。

3. 移除 `scheduler.analysis` 默认配置。

   定时入口被删除后，该配置不再有消费者。保留配置会造成误导，让用户以为仍可配置内容分析定时任务。

4. 保留顶层 `analysis` 配置。

   `analysis.postCount`、`analysis.maxCacheCount`、`analysis.storagePath` 仍属于内容分析服务的运行参数，自动发帖按需刷新摘要时仍需要这些配置。

5. 不迁移历史分析数据。

   `data/analysis.json` 仍可被现有链路读取。删除定时入口不会改变文件格式或读写语义，因此不需要数据迁移。

## 风险 / 权衡

- 分析摘要可能更久不更新 → 自动发帖在需要摘要时仍必须按内容分析规范检查时效并按需刷新。
- 管理界面或配置保存逻辑可能仍展示 `scheduler.analysis` → 实现时需要搜索 Web 配置相关代码，确保没有残留配置入口或保存后重新写回该字段。
- 外部用户已有 `scheduler.analysis` 自定义配置 → 删除默认配置后，旧配置应被忽略，不应导致启动或热重载失败。
- 任务数量变化可能影响日志和测试断言 → 实现后需要更新调度器相关测试或快照。

## 迁移计划

- 移除调度器中的“内容分析”注册和配置键映射。
- 移除启动入口传入的 `analysis` handler。
- 移除默认配置中的 `scheduler.analysis`。
- 检查配置 UI、配置类型和测试中是否引用 `scheduler.analysis`，按“定时入口已移除、按需分析保留”的语义同步更新。
- 通过类型检查和相关测试验证其他定时任务仍正常注册。

## 未决问题

- 是否需要在 Web 管理界面显式隐藏历史用户配置中的 `scheduler.analysis` 字段，取决于当前配置界面的字段生成方式。
