## 上下文

项目根目录不是 Git 仓库，无法通过版本历史恢复源码。当前可靠恢复来源主要有三类：`dist` 中的 CommonJS 构建产物、对应 `.d.ts` 类型声明、`.js.map` 中保留的原始源文件路径。调查结果显示 `dist` 中共有 57 个非测试 JavaScript 模块，其中 42 个在 `src` 中没有对应源文件，直接导致 `npm run build` 失败。

配置侧存在两个问题：一是根目录同步冲突文件中保存了更完整的 package/test 配置；二是默认配置文件直接包含 API Key 等敏感信息，不能继续作为仓库基线配置。部署侧的 HiGPT 网关目录说明与 `docker-compose.yml` 都引用 `app/Dockerfile`，但实际文件缺失。

## 目标 / 非目标

**目标：**
- 恢复缺失源文件，使主项目能够从 `src` 重新完成 TypeScript 构建。
- 将构建脚本恢复为可复制 Web 静态资源的版本，避免 `dist/web/public` 丢失。
- 整理默认配置，去除可直接复用的敏感凭据，并保留本地覆盖配置入口。
- 补齐 HiGPT 网关 Docker 构建入口，使部署目录自洽。

**非目标：**
- 不重写业务逻辑，不改变自动评论、自动发帖、素材处理、AI 调用等运行时语义。
- 不删除用户本地 token、素材、日志或冲突备份文件。
- 不保证从构建产物恢复出的源码完全等同于最初手写 TypeScript，只保证结构、导出和可构建性恢复。

## 决策

- 以 `dist` 为事实恢复源：由于没有 Git 历史，`dist` 是当前唯一覆盖完整运行逻辑的来源；`.js.map` 可用于确认目标源路径，但没有 `sourcesContent`，不能直接还原原始源码文本。
- 采用渐进恢复策略：优先恢复缺失模块并让构建通过，再在后续变更中按模块重构为更干净的 TypeScript。这样可以先消除“源码不可构建”的阻断问题。
- 保留冲突文件作为证据：冲突文件不直接删除，先将其中明确有效的信息合并到当前配置，例如构建脚本、测试脚本和依赖声明。
- 配置安全基线采用占位符：默认配置中保留字段结构，但用空值或环境变量覆盖方式替代真实密钥，避免把敏感信息继续固化在仓库默认配置中。

## 风险 / 权衡

- 构建产物恢复出的源码可读性较差 → 后续可以按模块逐步重构，但当前优先恢复可构建状态。
- 使用 `allowJs` 接纳恢复出的 JavaScript 源文件会降低类型严格性 → 保留现有 TypeScript 文件和声明生成，后续再逐步迁移恢复文件。
- 默认配置移除真实密钥后，本地运行需要通过 `config/local.yaml` 或环境变量补齐 → 这是安全收益与便利性的权衡。
- Dockerfile 只能按现有网关 package 推断补齐 → 通过 `npm run build` 与启动命令验证基础可用性。
