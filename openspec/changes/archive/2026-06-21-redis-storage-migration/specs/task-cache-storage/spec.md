## 新增需求

### 需求：系统必须提供任务缓存的 Redis 存储功能

系统必须从零开始使用 Redis Hash 存储任务缓存，支持字段级操作、TTL 自动过期和批量查询。

**约束**：
- 项目从零开始，没有历史数据需要迁移
- 只使用 Redis 存储，不涉及内存 Map 持久化
- 使用 Hash 数据结构（结构化存储，字段级操作）

#### 场景：保存任务
- **当** 调用 `saveTask(taskId, task)` 方法时
- **那么** 系统必须使用 `HSET` 命令将任务字段存储到 Hash
- **并且** 必须设置 30 分钟 TTL（使用 `EXPIRE` 命令）

#### 场景：获取任务
- **当** 调用 `getTask(taskId)` 方法时
- **那么** 系统必须使用 `HGETALL` 命令获取所有字段
- **并且** 必须反序列化为 AsyncTask 对象
- **并且** 如果任务不存在必须返回 null

#### 场景：更新任务状态
- **当** 调用 `updateTaskStatus(taskId, status)` 方法时
- **那么** 系统必须使用 `HSET` 命令只更新 status 字段
- **并且** 必须保留其他字段不变

#### 场景：删除任务
- **当** 调用 `deleteTask(taskId)` 方法时
- **那么** 系统必须使用 `DEL` 命令删除整个 Hash
- **并且** 必须返回删除是否成功

#### 场景：批量获取任务
- **当** 调用 `getTasks(taskIds)` 方法时
- **那么** 系统必须使用 Pipeline 批量执行 `HGETALL` 命令
- **并且** 必须返回 Map<string, AsyncTask> 格式的结果

#### 场景：检查任务是否存在
- **当** 调用 `hasTask(taskId)` 方法时
- **那么** 系统必须使用 `EXISTS` 命令检查
- **并且** 必须返回布尔值结果

### 需求：系统必须支持任务自动过期清理

系统必须确保任务在 30 分钟后自动过期并从 Redis 中删除。

#### 场景：设置 TTL
- **当** 保存或更新任务时
- **那么** 系统必须刷新 30 分钟 TTL
- **并且** 必须使用 `EXPIRE` 命令

#### 场景：过期清理
- **当** 任务超过 30 分钟未访问时
- **那么** Redis 必须自动删除该键
- **并且** 不需要应用层手动清理
