## 新增需求

### 需求：系统必须提供 Redis 连接管理功能

系统必须从零开始实现统一的 Redis 连接管理器，支持连接池管理、健康检查、优雅关闭和自动重连。

**约束**：
- 项目从零开始，没有现有数据库连接
- 支持测试环境（IP 通信）和生产环境（容器内通信）
- 必须配置持久化确保重启后数据不丢失

#### 场景：初始化连接
- **当** 应用启动时
- **那么** 系统必须根据环境变量（NODE_ENV）加载对应的 Redis 配置（测试环境使用 IP，生产环境使用容器名）
- **并且** 必须建立连接池（最小连接数 5，最大连接数 20）

#### 场景：健康检查
- **当** 调用 `healthCheck()` 方法时
- **那么** 系统必须执行 PING 命令测试连接
- **并且** 必须返回连接状态（connected/disconnected）和延迟（毫秒）

#### 场景：自动重连
- **当** Redis 连接断开时
- **那么** 系统必须自动尝试重连（最多 3 次，间隔 1 秒）
- **并且** 必须在重连失败时抛出错误并记录日志

#### 场景：优雅关闭
- **当** 应用关闭时调用 `disconnect()` 方法
- **那么** 系统必须等待所有进行中的命令完成
- **并且** 必须在 5 秒超时后强制关闭连接

### 需求：系统必须支持多数据库隔离

系统必须支持通过配置切换不同的 Redis 数据库编号，实现测试和生产环境的数据隔离。

#### 场景：切换数据库
- **当** 配置中指定 `db` 参数时
- **那么** 系统必须连接到指定的数据库编号（测试环境 DB 0，生产环境 DB 1）
- **并且** 必须使用 `SELECT` 命令切换数据库

#### 场景：键前缀隔离
- **当** 配置中指定 `keyPrefix` 时
- **那么** 系统必须在所有键操作时自动添加前缀（如 `test:` 或 `prod:`）
- **并且** 必须防止不同环境的键冲突
