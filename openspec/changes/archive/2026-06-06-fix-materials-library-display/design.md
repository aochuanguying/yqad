## 上下文

当前 Web 管理端素材库页面通过 `GET /api/materials` 拉取 `items` 并直接按返回顺序渲染缩略图网格（不做二次排序/分组）。后端素材索引来自文件系统递归扫描（`fs.readdirSync`），其返回顺序在不同文件系统/复制方式/刷新时可能不稳定，导致前端展示出现“顺序随机、目录与图片混排、刷新后顺序变化”等错乱观感。

用户期望素材库默认按“分类（目录）”分组展示，同时目录内展示顺序稳定，便于在大量素材中快速定位。

## 目标 / 非目标

**目标：**
- Web 管理端素材库在“全部目录”模式下按 directory 分组展示（根目录单独分组）
- `GET /api/materials` 返回的 `items` 顺序稳定（目录升序、文件名升序），避免前端依赖扫描顺序
- 不改变 `GET /api/materials` 的响应字段结构，仅调整顺序与前端渲染方式

**非目标：**
- 不新增素材的元数据体系（例如标签、评分、拍摄时间等）
- 不引入数据库或持久化索引（仍基于文件系统扫描与内存缓存）
- 不改造素材文件浏览器接口（`/api/materials/browser`）的既有排序与交互

## 决策

1) **排序在后端完成，分组在前端完成**
- 方案 A：仅前端分组 + 前端排序  
  - 优点：后端改动最小  
  - 缺点：API 仍不稳定；其他潜在消费者仍可能受影响；前端需要实现两套排序（含筛选模式）
- 方案 B：后端提供稳定排序（本次选择）+ 前端根据 directory 分组渲染  
  - 优点：稳定性从源头保证；不改变字段结构；前端逻辑更简单（只负责分组展示）  
  - 缺点：扫描阶段需要对目录项排序，有少量 CPU 开销
- 方案 C：后端新增“分组结构”字段（例如 `groupedItems`）  
  - 优点：前端实现最省  
  - 缺点：增加 API 形态与维护成本；需要兼容字段缺失；与现有简洁 API 风格不一致

2) **扫描阶段对目录项排序以保证缓存顺序稳定**
- 在递归扫描时对每层 `fs.readdirSync` 的结果按 “目录优先 + 名称升序” 排序后再递归与收集文件，从而使 `cachedMaterials` 的顺序天然稳定。

3) **目录展开（用于素材选择）保持确定性**
- 对目录展开时（取直属图片文件）同样按文件名升序排序，避免同一目录下的素材选择顺序随文件系统变化。

## 风险 / 权衡

- **[大量文件时排序开销]** → 仅在扫描/刷新时发生；保持现有内存缓存策略；必要时可通过刷新按钮/定时刷新控制频率
- **[localeCompare 在不同环境排序差异]** → 目录与文件名使用同一排序方式；对主要场景（中文/英文/数字）保持稳定；避免依赖文件系统默认顺序
- **[前端分组导致 DOM 变多]** → 仅在素材库页面发生；可复用现有网格结构，每个分组一个网格容器，控制分组标题样式简洁
