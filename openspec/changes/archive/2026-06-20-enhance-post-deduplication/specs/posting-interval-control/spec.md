## 新增需求

### 需求：系统必须控制同一主题的发帖间隔

系统必须检查同一主题的最后发帖时间，确保两次发帖间隔不少于配置的最短间隔天数。

#### 场景：发帖间隔符合要求
- **当** 同一主题的最后发帖时间距离当前时间超过配置的最短间隔天数（默认 3 天）
- **那么** 系统必须判定内容通过发帖间隔检查，允许继续发布流程
- **并且** 必须在合规性检查报告中记录该主题的最后发帖时间和间隔天数

#### 场景：发帖间隔不足
- **当** 同一主题的最后发帖时间距离当前时间少于配置的最短间隔天数
- **那么** 系统必须拒绝发布该内容
- **并且** 返回错误信息必须提示"同一主题发帖间隔不足 X 天，建议 Y 天后再发布"
- **并且** 错误信息必须包含最后发帖时间和建议的下次发帖时间

#### 场景：主题首次发帖
- **当** 待发布内容使用的主题从未发布过帖子
- **那么** 系统必须判定内容通过发帖间隔检查，允许发布
- **并且** 必须在主题数据中记录首次发帖时间

#### 场景：自由生成模式不受间隔控制
- **当** 发帖使用自由生成模式（无预配置主题）
- **那么** 系统必须跳过发帖间隔检查
- **并且** 必须在合规性检查报告中标记为"自由生成模式，无间隔控制"

#### 场景：可配置的最短间隔天数
- **当** 配置文件中设置了 `postingIntervalControl.minIntervalDays` 参数
- **那么** 系统必须使用配置的天数作为最短发帖间隔（默认 3 天，范围 1-14 天）

#### 场景：可配置的最长间隔天数
- **当** 配置文件中设置了 `postingIntervalControl.maxIntervalDays` 参数
- **那么** 系统必须使用配置的天数作为建议的最长发贴间隔（默认 7 天，用于提醒）
- **并且** 当间隔超过最长间隔时，必须在合规性检查报告中记录提醒信息

### 需求：系统必须更新主题的最后发帖时间

系统必须在成功发帖后更新所使用主题的最后发帖时间。

#### 场景：使用预配置主题发帖成功
- **当** 使用预配置主题成功发布帖子
- **那么** 系统必须更新该主题的 `lastPostDate` 字段为当前时间
- **并且** 必须将变更持久化到主题数据存储

#### 场景：自由生成模式不更新时间
- **当** 使用自由生成模式发布帖子
- **那么** 系统不得更新任何主题的 `lastPostDate` 字段

#### 场景：发帖失败不更新时间
- **当** 发帖失败（无论何种原因）
- **那么** 系统不得更新主题的 `lastPostDate` 字段

### 需求：系统必须提供发帖间隔查询 API

系统必须提供 HTTP API 供外部查询主题的发帖间隔状态。

#### 场景：查询主题发帖间隔状态
- **当** 客户端发送 GET 请求到 `/api/compliance/posting-interval/:topicId`
- **那么** 系统必须返回该主题的发帖间隔状态
- **并且** 响应必须包含：最后发帖时间、已间隔天数、最短间隔要求、是否可发帖、建议的下次发帖时间

#### 场景：查询所有主题发帖间隔状态
- **当** 客户端发送 GET 请求到 `/api/compliance/posting-interval`
- **那么** 系统必须返回所有预配置主题的发帖间隔状态列表
- **并且** 响应必须支持筛选（如：`?status=blocked` 仅返回不可发帖的主题）

### 需求：系统必须支持发帖间隔白名单

系统必须支持配置发帖间隔白名单，允许特定情况下绕过间隔控制。

#### 场景：白名单主题豁免间隔控制
- **当** 待发布内容使用的主题在白名单中
- **那么** 系统必须跳��发帖间隔检查，允许立即发布
- **并且** 必须在合规性检查报告中标记为"白名单豁免"

#### 场景：紧急发帖豁免
- **当** 发帖请求包含 `bypassIntervalControl: true` 标志且调用方有相应权限
- **那么** 系统必须跳过发帖间隔检查，允许发布
- **并且** 必须在合规性检查报告和发帖日志中记录"紧急发帖豁免"
- **并且** 必须记录豁免操作的责任人和原因
