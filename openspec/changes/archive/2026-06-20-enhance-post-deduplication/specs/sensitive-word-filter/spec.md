## 新增需求

### 需求：系统必须检测和过滤帖子内容中的敏感词

系统必须在发帖前检测标题和正文中的敏感词，并根据敏感词级别执行相应的处理策略。

#### 场景：内容不包含敏感词
- **当** 待发布内容的标题和正文均不包含敏感词库中的任何词汇
- **那么** 系统必须判定内容通过敏感词检查，允许继续发布流程

#### 场景：内容包含禁止级敏感词
- **当** 待发布内容包含敏感词库中标记为"禁止"级别的敏感词
- **那么** 系统必须拒绝发布该内容
- **并且** 返回错误信息必须提示"内容包含敏感词汇"，但不暴露具体词汇

#### 场景：内容包含替换级敏感词
- **当** 待发布内容包含敏感词库中标记为"替换"级别的敏感词
- **那么** 系统必须自动将敏感词替换为同义词或 `*` 号屏蔽
- **并且** 必须在合规性检查报告中记录替换的词汇和位置

#### 场景：内容包含警告级敏感词
- **当** 待发布内容包含敏感词库中标记为"警告"级别的敏感词
- **那么** 系统必须允许发布该内容
- **并且** 必须在合规性检查报告中记录警告信息
- **并且** 必须在发帖日志中标记该内容需要人工复核

#### 场景：敏感词白名单豁免
- **当** 待发布内容包含的词汇同时在白名单中
- **那么** 系统必须跳过该词汇的敏感词检测
- **并且** 白名单词汇必须优先于敏感词库匹配

### 需求：系统必须使用 AC 自动机算法进行敏感词匹配

系统必须使用基于 Trie 树的 AC 自动机算法实现高效的多模式敏感词匹配。

#### 场景：单次扫描检测所有敏感词
- **当** 系统需要检测一段内容中的敏感词
- **那么** 系统必须通过一次扫描识别出所有匹配的敏感词
- **并且** 检测时间复杂度必须为 O(n)，其中 n 为内容长度

#### 场景：支持中文敏感词检测
- **当** 敏感词库包含中文词汇
- **那么** 系统必须正确识别和匹配中文敏感词
- **并且** 必须支持中英文混合内容的检测

#### 场景：敏感词库动态加载
- **当** 敏感词库文件发生变化
- **那么** 系统必须支持热更新，无需重启服务即可加载新词库
- **并且** 必须在 1 秒内完成词库重新加载

### 需求：系统必须提供敏感词库管理 API

系统必须提供 HTTP API 支持敏感词库的增删改查操作。

#### 场景：查询敏感词列表
- **当** 管理员发送 GET 请求到 `/api/compliance/sensitive-words?page=1&level=禁止`
- **那么** 系统必须返回分页的敏感词列表，支持按级别筛选
- **并且** 响应必须包含词汇、级别、分类和创建时间

#### 场景：添加敏感词
- **当** 管理员发送 POST 请求到 `/api/compliance/sensitive-words`
- **并且** 请求体包含 `word`、`level`、`category` 字段
- **那么** 系统必须将新敏感词添加到词库
- **并且** 必须立即生效，后续检测必须包含新添加的词汇

#### 场景：删除敏感词
- **当** 管理员发送 DELETE 请求到 `/api/compliance/sensitive-words/:id`
- **那么** 系统必须从词库中删除指定 ID 的敏感词
- **并且** 必须立即生效，后续检测不再包含已删除的词汇

#### 场景：批量导入敏感词
- **当** 管理员发送 POST 请求到 `/api/compliance/sensitive-words/batch-import`
- **并且** 请求体包含敏感词数组（JSON 格式）
- **那么** 系统必须批量导入敏感词到词库
- **并且** 必须返回导入成功和失败的数量统计

### 需求：系统必须存储敏感词库到本地文件

系统必须将敏感词库存储在 `data/sensitive-words.json` 文件中。

#### 场景：敏感词库文件结构
- **当** 系统初始化加载敏感词库
- **那么** 系统必须从 `data/sensitive-words.json` 读取数据
- **并且** 文件结构必须包含：词汇、级别（禁止/替换/警告）、分类、创建时间、更新时间

#### 场景：敏感词库持久化
- **当** 管理员通过 API 添加、删除或修改敏感词
- **那么** 系统必须立即将变更持久化到 `data/sensitive-words.json` 文件
- **并且** 必须保证文件的 JSON 格式有效性
