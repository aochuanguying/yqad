## 新增需求

### 需求：主题内容池结构
系统必须为每个支持多次复用的主题维护一个内容池，内容池包含多个子方向（sub-directions）和内容提纲（outlines），每个子方向代表一个独立的发帖角度。

#### 场景：创建主题时初始化内容池
- **当** 创建主题且 maxUseCount > 1 时
- **那么** 系统必须要求提供至少 maxUseCount 个子方向，确保每次复用都有独立的内容角度

#### 场景：内容池数据结构
- **当** 主题支持多次复用时
- **那么** 内容池必须包含以下字段：
  - subDirections: 子方向数组，每个元素包含 { title, direction, outline }
  - usedSubDirections: 已使用的子方向索引数组
  - lastUsedIndex: 上次使用的子方向索引

### 需求：内容池选取策略
系统必须实现智能的内容池选取策略，确保同一主题的多次发帖内容均衡分配，避免重复使用相同的内容组合。

#### 场景：首次发帖选取
- **当** 主题首次发帖（useCount = 0）时
- **那么** 必须选取索引为 0 的子方向，即内容池中的第一个子方向

#### 场景：后续发帖选取
- **当** 主题再次发帖（useCount > 0）时
- **那么** 必须选取未使用过的子方向中索引最小的一个，确保按顺序使用

#### 场景：内容池耗尽处理
- **当** 所有子方向都已使用但 useCount < maxUseCount 时
- **那么** 必须允许选取已使用过的子方向，但需要在 Prompt 中告知 AI 避免与历史发帖内容重复

### 需求：发帖历史增强记录
系统必须在发帖历史中记录每次使用的子方向信息，以便后续复用时的去重和差异化处理。

#### 场景：记录子方向使用
- **当** 发帖成功后
- **那么** 必须在 postHistory 中记录：
  - usedSubDirectionIndex: 使用的子方向索引
  - subDirectionTitle: 使用的子方向标题
  - subDirectionDirection: 使用的子方向描述

#### 场景：历史记录查询
- **当** 为该主题生成新帖子时
- **那么** 必须能够查询到该主题所有历史发帖使用的子方向信息
