## 新增需求

### 需求：系统必须提供认证中间件保护 API 和 Web 资源
系统应当实现 Express 中间件，验证请求的 Session 状态，阻止未授权访问。

#### 场景：已认证用户访问受保护 API
- **当** 已登录用户请求受保护的 API 端点
- **那么** 中间件允许请求通过，继续处理业务逻辑

#### 场景：未认证用户访问受保护 API
- **当** 未登录用户请求受保护的 API 端点
- **那么** 中间件返回 401 状态码和 JSON 错误响应 `{"error": "未授权访问"}`

#### 场景：未认证用户访问受保护页面
- **当** 未登录用户请求受保护的页面（如 index.html）
- **那么** 中间件重定向到登录页 `/login.html`

#### 场景：访问公开 API
- **当** 用户请求公开的 API 端点（如 `/api/auth/login`）
- **那么** 中间件跳过认证检查，允许直接访问

### 需求：认证中间件必须支持白名单机制
系统应当提供配置方式，指定不需要认证的公开 API 路径。

#### 场景：配置公开 API 白名单
- **当** 管理员在配置中设置 `web.auth.publicRoutes` 数组
- **那么** 中间件对这些路径跳过认证检查

#### 场景：默认公开路由
- **当** 未配置公开路由白名单
- **那么** 系统默认公开以下路由：
  - `/api/auth/login` (登录接口)
  - `/api/posts/token/*` (远程发帖 API，使用独立 Token 认证)

#### 场景：静态资源访问
- **当** 用户请求静态资源（CSS、JS、图片）
- **那么** 中间件允许访问，不受认证限制

### 需求：认证中间件必须处理 Session 异常
系统应当妥善处理 Session 解析失败、过期等异常情况，提供友好的错误处理。

#### 场景：Session 解析失败
- **当** Session Cookie 损坏或格式错误
- **那么** 中间件视为未登录，返回 401 或重定向到登录页

#### 场景：Session 存储不可用
- **当** Session 存储（MemoryStore）出现异常
- **那么** 中间件记录错误日志，返回 500 状态码

#### 场景：并发请求的 Session 状态
- **当** 同一 Session ID 发起多个并发请求
- **那么** 中间件正确处理每个请求，保持 Session 状态一致

### 需求：认证中间件必须记录审计日志
系统应当记录所有认证相关的事件，用于安全审计和问题排查。

#### 场景：登录成功
- **当** 用户成功登录
- **那么** 系统记录日志：`[AUTH] 登录成功 - IP: xxx.xxx.xxx.xxx`

#### 场景：登录失败
- **当** 用户登录失败（密码错误）
- **那么** 系统记录日志：`[AUTH] 登录失败 - IP: xxx.xxx.xxx.xxx - 原因：密码错误`

#### 场景：未授权访问
- **当** 未登录用户尝试访问受保护资源
- **那么** 系统记录日志：`[AUTH] 未授权访问 - IP: xxx.xxx.xxx.xxx - 路径：/api/xxx`
