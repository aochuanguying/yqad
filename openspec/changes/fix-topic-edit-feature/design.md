## 上下文

当前 Web 管理界面的主题管理功能支持：
- 创建新主题（POST /api/topics）
- 删除主题（DELETE /api/topics/:id）
- 重置主题状态（POST /api/topics/:id/reset）
- 单独修改复用次数（PATCH /api/topics/:id/max-use-count）

但缺少直接编辑主题全部字段的能力，用户需要删除后重新创建才能修改主题信息。

## 目标 / 非目标

**目标：**
- 在主题卡片上添加编辑按钮，用户可一键打开编辑表单
- 支持编辑所有主题字段：标题、方向、提纲、最大复用次数、素材
- 后端 API 支持在单个 PUT 请求中更新所有字段
- 保持 API 向后兼容，不影响现有调用
- 新增完整的单元测试覆盖

**非目标：**
- 不修改主题的状态管理逻辑（status、useCount 等）
- 不改变主题的存储结构
- 不引入新的依赖库

## 决策

### 1. 前端编辑表单实现方式

**决策**：复用现有的新建主题表单结构，新增独立的编辑函数 `showEditTopicForm()`、`submitEditTopic()`、`cancelEditTopicForm()`

**理由**：
- 与现有代码风格一致
- 复用素材选择器组件
- 无需重构现有代码，降低风险

**替代方案**：
- 方案 A：统一新建和编辑表单 → 需要较大重构，风险高
- 方案 B：使用模态框 → 与现有 UI 风格不一致

### 2. 后端 API 设计

**决策**：在现有 PUT /api/topics/:id 接口中增加 `maxUseCount` 参数支持

**理由**：
- 最小化 API 变更
- 符合 RESTful 规范（PUT 用于全量更新）
- 向后兼容，不影响现有调用

**替代方案**：
- 方案 A：新增专用编辑接口 → API 冗余
- 方案 B：使用 PATCH 进行部分更新 → 需要实现更复杂的部分更新逻辑

### 3. 编辑按钮位置

**决策**：在主题卡片右上角按钮组的第一个位置添加 ✏️ 编辑按钮

**理由**：
- 符合用户操作习惯（从左到右：编辑 → 设置 → 重置 → 删除）
- 图标语义清晰
- 与其他操作按钮视觉一致

## 风险 / 权衡

### 风险 1：浏览器缓存导致前端代码不生效
**缓解措施**：强制刷新页面（Ctrl+Shift+R），或在开发环境中禁用缓存

### 风险 2：编辑表单数据验证不足
**缓解措施**：前端验证必填字段（标题、方向），后端依赖现有验证逻辑

### 风险 3：并发编辑冲突
**缓解措施**：当前为单用户场景，暂不处理。未来可考虑添加乐观锁机制

### 权衡：代码复用 vs 独立性
- 选择独立函数而非统一表单，牺牲了一定代码复用性
- 换取更低的实现风险和更清晰的代码边界
