## 修改需求

### 需求：系统必须获取至少 100 个热门话题作为候选池

系统必须在话题匹配前，分页获取至少 5 页（每页 20 个）热门话题，确保候选话题池达到 100 个，以提升话题匹配的准确性和相关性。

#### 场景：成功获取 5 页话题
- **当** 系统调用 `fetchHotTopics` 函数
- **那么** 系统循环调用 API 5 次，每次获取 1 页（20 个）话题
- **那么** 系统累计获取至少 100 个热门话题
- **那么** 系统记录每页获取的进度日志

#### 场景：某页获取失败时的降级处理
- **当** 某一页的 API 调用失败
- **那么** 系统记录警告日志
- **那么** 系统继续获取下一页
- **那么** 系统返回已成功获取的话题列表（可能少于 100 个）

#### 场景：话题总数不足 100 个
- **当** API 返回的话题总数少于 100 个（例如只有 80 个）
- **那么** 系统返回所有已获取的话题
- **那么** 系统不进行错误处理，继续后续的话题匹配流程

#### 场景：所有页都获取失败
- **当** 5 页话题全部获取失败
- **那么** 系统返回空数组
- **那么** 话题匹配流程检测到空候选列表时跳过匹配
- **那么** 发帖流程以无话题方式继续
