## 上下文

当前车辆监控告警系统通过 `alert-service.ts` 实现，依赖 `telecom-client.ts` 提供短信和电话告警功能。告警配置存储在数据库的 `vehicle_monitor_config` 表中，包含 `alert_phone` 字段。

用户需求：
- 增加 Bark 推送作为新的告警渠道
- 手机号和 Bark 键可以为空，不为空时才执行对应告警
- 支持单独或组合使用两种告警方式

## 目标 / 非目标

**目标：**
- 新增 Bark 告警渠道，与现有手机告警并行
- 支持灵活配置：仅手机、仅 Bark、两者组合
- 保持告警冷却机制和日志记录
- 各渠道独立执行，单一失败不影响其他渠道

**非目标：**
- 不替换现有的电信 API 告警
- 不改变告警冷却时间和历史记录机制
- 不支持其他推送服务（如 PushPlus、Server 酱等）

## 决策

### 1. Bark 客户端实现方式

**决策：** 新建独立的 `bark-client.ts` 模块，仿照 `telecom-client.ts` 的架构模式。

**理由：**
- 保持代码组织清晰，职责分离
- 与现有电信客户端保持一致的代码风格
- 便于未来扩展其他推送渠道

**替代方案：**
- 在 `alert-service.ts` 中直接实现 Bark API 调用 → 会导致文件臃肿
- 合并到 `telecom-client.ts` → 职责不清晰，Bark 不属于电信业务

### 2. 告警配置存储

**决策：** 在 `vehicle_monitor_config` 表中新增 `bark_key` 和 `bark_server` 字段。

**理由：**
- 与现有 `alert_phone` 字段保持一致的存储方式
- 支持自定义 Bark 服务器地址（可选）
- 数据库迁移简单，向后兼容

### 3. 告警执行策略

**决策：** 在 `alert-service.ts` 中并行执行所有配置的告警渠道。

**理由：**
- 提高告警响应速度
- 各渠道独立，失败互不影响
- 使用 `Promise.allSettled` 收集所有结果

### 4. Bark 消息格式

**决策：** 标题固定为"车辆告警"，正文包含时间戳、异常类型、位置信息。

**理由：**
- 与现有短信格式保持一致
- Bark 推送标题简洁，正文详细
- 支持点击推送跳转到地图（可选）

## 风险 / 权衡

### [风险] Bark 服务器不可用

**缓解措施：**
- 设置合理的超时时间（10 秒）
- 记录错误日志但不影响其他渠道
- 支持自定义服务器地址，用户可切换

### [风险] Bark 键泄露

**缓解措施：**
- 日志中脱敏 Bark 键（显示前 8 位）
- 数据库字段加密存储（可选）
- 建议用户定期轮换 Bark 键

### [风险] 告警延迟

**缓解措施：**
- 并行执行所有渠道
- 设置超时时间避免阻塞
- 异步执行不阻塞主监控流程

## 迁移计划

### 数据库迁移

1. 执行 SQL 迁移脚本，添加 `bark_key` 和 `bark_server` 字段
2. 默认值为空字符串，不影响现有配置
3. 更新默认配置插入语句

### 代码部署

1. 新增文件：`bark-client.ts`
2. 修改文件：`alert-service.ts`、`vehicle-monitor-storage.ts`
3. 无需重启服务（支持热重载）

### 回滚策略

- 数据库迁移可逆（DROP COLUMN）
- 代码回滚只需恢复旧版本文件
- 不影响现有告警功能
