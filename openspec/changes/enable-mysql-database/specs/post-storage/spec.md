## 新增需求

### 需求:帖子数据 MySQL 存储

系统必须使用 MySQL 数据库存储帖子数据，包括帖子内容、元数据、状态等信息。

#### 场景:创建帖子
- **当** 会员创建新帖子时
- **那么** 系统必须在 posts 表中插入一条新记录
- **那么** 系统必须生成唯一的帖子 ID（UUID）
- **那么** 系统必须关联会员 ID（外键约束）
- **那么** 系统必须设置默认状态为 draft
- **那么** 系统必须记录创建时间
- **那么** 系统必须初始化计数为 0（浏览、点赞、评论）

#### 场景:查询帖子详情
- **当** 根据帖子 ID 查询时
- **那么** 系统必须返回完整的帖子信息
- **那么** 系统必须包含作者信息（JOIN members 表）
- **那么** 系统必须包含统计计数（浏览、点赞、评论）
- **那么** 系统必须增加浏览计数（如果是查看详情）

#### 场景:更新帖子内容
- **当** 作者修改帖子时
- **那么** 系统必须更新 title 和 content 字段
- **那么** 系统必须记录更新时间
- **那么** 系统必须验证操作者是帖子作者
- **那么** 系统必须返回更新后的帖子信息

#### 场景:发布帖子
- **当** 作者发布草稿时
- **那么** 系统必须将状态改为 published
- **那么** 系统必须设置 published_at 为当前时间
- **那么** 系统必须验证帖子内容符合规范
- **那么** 系统必须更新会员发帖计数

#### 场景:删除帖子
- **当** 作者删除帖子时
- **那么** 系统必须将状态改为 deleted（软删除）
- **那么** 系统必须级联删除所有评论
- **那么** 系统必须保留帖子内容用于审计
- **那么** 系统必须减少会员发帖计数

#### 场景:查询帖子列表
- **当** 查询帖子列表时
- **那么** 系统必须支持分页查询
- **那么** 系统必须支持按发布时间排序
- **那么** 系统必须支持按浏览数排序
- **那么** 系统必须支持按点赞数排序
- **那么** 系统必须支持按会员筛选
- **那么** 系统必须支持按状态筛选（draft/published/deleted）

### 需求:帖子状态管理

系统必须管理帖子的生命周期状态，支持草稿、发布、删除等状态转换。

#### 场景:状态转换
- **当** 帖子从 draft 变为 published 时
- **那么** 系统必须验证内容不为空
- **那么** 系统必须验证作者身份
- **那么** 系统必须记录发布时间
- **当** 帖子从 published 变为 deleted 时
- **那么** 系统必须验证操作权限
- **那么** 系统必须记录删除时间

#### 场景:状态查询过滤
- **当** 普通用户查询时
- **那么** 系统必须只返回 published 状态的帖子
- **当** 作者查询自己的帖子时
- **那么** 系统必须返回所有状态的帖子
- **当** 管理员查询时
- **那么** 系统必须返回所有帖子（包括 deleted）

#### 场景:定时发布
- **当** 设置定时发布时
- **那么** 系统必须保存 scheduled_at 字段
- **那么** 系统必须在到达时间时自动发布
- **那么** 系统必须支持取消定时发布

### 需求:帖子统计管理

系统必须统计帖子的浏览、点赞、评论等数据，支持实时更新和缓存。

#### 场景:浏览计数
- **当** 用户查看帖子详情时
- **那么** 系统必须增加 view_count 计数
- **那么** 系统必须防止重复计数（同一用户短时间内）
- **那么** 系统必须异步更新（不阻塞响应）

#### 场景:点赞计数
- **当** 用户点赞帖子时
- **那么** 系统必须增加 like_count 计数
- **那么** 系统必须记录点赞者 ID（防止重复点赞）
- **当** 用户取消点赞时
- **那么** 系统必须减少 like_count 计数

#### 场景:评论计数
- **当** 添加评论时
- **那么** 系统必须增加 comment_count 计数
- **当** 删除评论时
- **那么** 系统必须减少 comment_count 计数
- **那么** 系统必须保证计数准确（定期校准）

#### 场景:统计缓存
- **当** 高频访问时
- **那么** 系统必须使用 Redis 缓存计数
- **那么** 系统必须定期同步到 MySQL
- **那么** 系统必须支持手动刷新缓存

### 需求:帖子数据完整性

系统必须保证帖子数据的完整性和一致性，使用外键约束和事务。

#### 场景:外键约束
- **当** 删除会员时
- **那么** 系统必须级联删除该会员的帖子
- **当** 删除帖子时
- **那么** 系统必须级联删除该帖子的评论
- **那么** 系统必须禁止删除还有评论的帖子（如果未设置级联）

#### 场景:事务一致性
- **当** 发帖并更新统计时
- **那么** 系统必须使用事务保证原子性
- **那么** 系统必须在失败时回滚所有操作
- **那么** 系统必须保证数据最终一致性

#### 场景:数据验证
- **当** 创建帖子时
- **那么** 系统必须验证标题长度（最多 200 字符）
- **那么** 系统必须验证内容不为空
- **那么** 系统必须验证会员 ID 存在
- **当** 验证失败时
- **那么** 系统必须返回相应的错误信息
