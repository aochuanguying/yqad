## 新增需求

### 需求:评论数据 MySQL 存储

系统必须使用 MySQL 数据库存储评论数据，包括评论内容、关联关系、状态等信息。

#### 场景:创建评论
- **当** 用户发表评论时
- **那么** 系统必须在 comments 表中插入一条新记录
- **那么** 系统必须生成唯一的评论 ID（UUID）
- **那么** 系统必须关联帖子 ID（外键约束）
- **那么** 系统必须关联会员 ID（外键约束）
- **那么** 系统必须支持父评论 ID（用于回复，可选外键）
- **那么** 系统必须设置默认状态为 approved
- **那么** 系统必须记录创建时间

#### 场景:查询评论列表
- **当** 查询帖子评论时
- **那么** 系统必须按帖子 ID 筛选
- **那么** 系统必须按创建时间排序
- **那么** 系统必须包含评论者信息（JOIN members 表）
- **那么** 系统必须支持分页查询
- **那么** 系统必须只返回 approved 状态的评论

#### 场景:查询评论详情
- **当** 根据评论 ID 查询时
- **那么** 系统必须返回完整的评论信息
- **那么** 系统必须包含评论者信息
- **那么** 系统必须包含帖子信息
- **那么** 系统必须包含父评论信息（如果是回复）

#### 场景:更新评论内容
- **当** 评论者修改评论时
- **那么** 系统必须更新 content 字段
- **那么** 系统必须记录更新时间
- **那么** 系统必须验证操作者是评论作者
- **那么** 系统必须返回更新后的评论信息
- **那么** 系统必须标记为已编辑（edited_at）

#### 场景:删除评论
- **当** 删除评论时
- **那么** 系统必须将状态改为 deleted（软删除）
- **那么** 系统必须级联删除所有子评论（回复）
- **那么** 系统必须保留评论内容用于审计
- **那么** 系统必须减少帖子评论计数

#### 场景:审核评论
- **当** 管理员审核评论时
- **那么** 系统必须支持将状态改为 approved
- **那么** 系统必须支持将状态改为 rejected
- **那么** 系统必须记录审核时间和审核者
- **那么** 系统必须通知评论者审核结果

### 需求:评论层级关系管理

系统必须管理评论的层级关系，支持多级回复和嵌套展示。

#### 场景:一级评论
- **当** 评论没有父评论时
- **那么** 系统必须将 parent_id 设为 NULL
- **那么** 系统必须在列表顶部展示
- **那么** 系统必须支��查看其下所有回复

#### 场景:回复评论
- **当** 回复某条评论时
- **那么** 系统必须设置 parent_id 为被回复评论的 ID
- **那么** 系统必须验证被回复评论存在
- **那么** 系统必须支持多级回复（最多 N 级，可配置）
- **那么** 系统必须记录回复链路径（用于快速查询）

#### 场景:查询评论树
- **当** 查询评论树时
- **那么** 系统必须返回一级评论列表
- **那么** 系统必须嵌套返回每个评论的回复列表
- **那么** 系统必须限制嵌套深度（最多 3 层）
- **那么** 系统必须按时间排序

#### 场景:删除评论树
- **当** 删除一级评论时
- **那么** 系统必须级联删除所有子评论
- **那么** 系统必须使用递归查询或路径枚举
- **那么** 系统必须保证删除完整性

### 需求:评论状态管理

系统必须管理评论的生命周期状态，支持待审核、通过、拒绝、删除等状态。

#### 场景:状态转换
- **当** 创建评论时
- **那么** 系统必须设置默认状态为 approved
- **当** 需要审核时
- **那么** 系统必须设置状态为 pending
- **当** 审核通过时
- **那么** 系统必须将状态改为 approved
- **当** 审核拒绝时
- **那么** 系统必须将状态改为 rejected
- **当** 删除评论时
- **那么** 系统必须将状态改为 deleted

#### 场景:状态查询过滤
- **当** 普通用户查询时
- **那么** 系统必须只返回 approved 状态的评论
- **当** 评论者查询自己的评论时
- **那么** 系统必须返回所有状态的评论
- **当** 管理员查询时
- **那么** 系统必须返回所有评论（包括 deleted）

#### 场景:自动审核
- **当** 启用自动审核时
- **那么** 系统必须检查敏感词
- **那么** 系统必须检查垃圾内容
- **那么** 系统必须自动设置状态为 approved 或 pending

### 需求:评论数据完整性

系统必须保证评论数据的完整性和一致性，使用外键约束和事务。

#### 场景:外键约束
- **当** 删除帖子时
- **那么** 系统必须级联删除该帖子的所有评论
- **当** 删除会员时
- **那么** 系统必须级联删除该会员的所有评论
- **当** 删除父评论时
- **那么** 系统必须级联删除所有子评论
- **那么** 系统必须禁止删除还有子评论的评论（如果未设置级联）

#### 场景:事务一致性
- **当** 创建评论并更新计数时
- **那么** 系统必须使用事务保证原子性
- **那么** 系统必须在失败时回滚所有操作
- **那么** 系统必须保证数据最终一致性

#### 场景:数据验证
- **当** 创建评论时
- **那么** 系统必须验证内容不为空
- **那么** 系统必须验证内容长度（最多 1000 字符）
- **那么** 系统必须验证帖子 ID 存在
- **那么** 系统必须验证会员 ID 存在
- **那么** 系统必须验证父评论 ID 存在（如果有）
- **当** 验证失败时
- **那么** 系统必须返回相应的错误信息

#### 场景:防止重复提交
- **当** 用户快速重复提交相同评论时
- **那么** 系统必须检测重复内容
- **那么** 系统必须限制提交频率（如 1 分钟最多 3 条）
- **那么** 系统必须返回友好的错误提示
