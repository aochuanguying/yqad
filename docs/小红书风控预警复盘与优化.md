### 导致预警的可能原因

#### 1. **双重调用风险** 🔴

**问题**：
- ✅ **Node.js 服务** (`npm start`) 正在运行
- ✅ **Python 脚本** (`export_xiaohongshu_audi.py`) 也在运行
- ✅ **两个独立的调用源**使用同一个 Cookie

**证据**：
- TypeScript 服务：`src/services/internet-search/xiaohongshu-search.ts`
- Python 脚本：`scripts/export_xiaohongshu_audi.py`
- 两者都调用小红书 API

**风险**：
- 短时间内请求频率翻倍
- 请求模式不一致（Node.js vs Python）
- 小红书可能检测到异常

---

#### 2. **请求频率过快** 🔴

**原配置（高风险）**:

**TypeScript 服务**:
```typescript
requestDelayMin: 1000,    // 1 秒
requestDelayMax: 3000,    // 3 秒
pageDelayMin: 3000,       // 3 秒
pageDelayMax: 5000,       // 5 秒
maxRequestsPerHour: 10,   // 每小时 10 次
```

**Python 脚本**:
```python
REQUEST_DELAY_MIN = 1  # 1 秒
REQUEST_DELAY_MAX = 3  # 3 秒
PAGE_DELAY_MIN = 3     # 3 秒
PAGE_DELAY_MAX = 5     # 5 秒
MAX_PAGES = 2
MAX_DETAILS = 5
```

**实际行为**：
- 每页获取 2-3 个详情
- 每个详情之间只延迟 1-3 秒
- 2 页 5 个详情总时间：**10-15 秒**
- **短时间内连续调用 API 10+ 次**

**对比真实用户**：
- 真实用户：浏览一篇笔记平均需要 30-60 秒
- 脚本速度：1-3 秒一篇
- **速度快了 20-30 倍** ⚠️

---

#### 3. **请求模式过于规律** 🟡

**固定流程**：
```
搜索 → 延迟 → 详情 1 → 延迟 → 详情 2 → 延迟 → 详情 3 → 翻页
```

**问题**：
- ❌ 每次都执行相同的流程
- ❌ 固定的延迟范围
- ❌ 缺少真实用户的随机行为

---

#### 4. **缺少用户行为模拟** 🟡

**当前实现**：
- ❌ 只有纯 API 调用
- ❌ 没有模拟页面加载
- ❌ 没有模拟滚动、点击
- ❌ 没有停留时间

**小红书检测**：
- ✅ 会检测用户行为特征
- ✅ 纯 API 调用缺少必要的行为数据
- ✅ 容易被判定为自动化脚本

## 🎯 后续建议

### 1. **分时段执行** （强烈推荐）

**策略**：
```
不要一次性获取所有数据
分成多个小批次，间隔数小时

例如：
批次 1: 10:00 - 获取第 1 页，2 个详情
批次 2: 14:00 - 获取第 2 页，2 个详情
批次 3: 19:00 - 获取第 3 页，2 个详情
```

**效果**：
- ✅ 分散请求时间
- ✅ 降低 80% 风控风险
- ✅ 最安全的方案
