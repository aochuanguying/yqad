## 新增需求

### 需求:帖子去重分析
系统必须持久化已分析帖子的 ID 集合，每次执行分析前必须过滤掉已分析过的帖子，不得对同一帖子重复执行分析。

#### 场景:过滤已分析帖子
- **当** `analyze()` 被调用，从 API 获取到一批帖子
- **那么** 系统必须从结果中移除 ID 存在于 `analyzedIds` 中的帖子，只对剩余的新帖子执行分析

#### 场景:首次运行无历史记录
- **当** `data/analysis.json` 不存在或 `analyzedIds` 字段缺失
- **那么** 系统必须将 `analyzedIds` 初始化为空数组，并对所有拉取到的帖子去重后全部分析

#### 场景:持久化新分析的帖子 ID
- **当** 一批新帖子分析完成
- **那么** 系统必须将这批帖子的 ID 追加到 `analyzedIds` 的头部（最新在前），并将更新后的 `analyzedIds` 与摘要一并写入 `data/analysis.json`

### 需求:去重后全部分析
系统必须在每次定时抓取 `analysis.postCount` 条帖子后，对去重后的所有新帖子执行分析，不得设置额外的条数截断。

#### 场景:所有拉取的帖子均为重复
- **当** 从 API 拉取的全部帖子 ID 均已存在于 `analyzedIds` 中
- **那么** 系统必须跳过分析步骤，返回已持久化的上一次摘要，不得发起 AI 调用

#### 场景:存在新帖子
- **当** 过滤重复后存在至少一条新帖子
- **那么** 系统必须对所有去重后的新帖子执行完整分析

### 需求:缓存条数上限裁剪
系统必须通过配置项 `analysis.maxCacheCount` 限制 `analyzedIds` 的最大记录条数。每次写入时，若 `analyzedIds` 数组长度超过 `maxCacheCount`，系统必须从数组末尾（最旧记录）裁剪，直到长度等于 `maxCacheCount`。

#### 场景:ID 记录超过上限
- **当** 写入 `data/analysis.json` 时，合并后的 `analyzedIds` 长度超过 `maxCacheCount`
- **那么** 系统必须保留数组头部（最新）的 `maxCacheCount` 条记录，删除尾部多余记录

#### 场景:ID 记录未超过上限
- **当** 合并后的 `analyzedIds` 长度小于或等于 `maxCacheCount`
- **那么** 系统必须完整保留所有 ID 记录，不得删除任何记录

### 需求:移除时间缓存失效逻辑
系统禁止使用时间戳判断分析结果是否过期。`getSummary()` 必须直接返回已持久化的摘要（存在即返回），不存在时才触发 `analyze()`，不得以时间作为触发重新分析的条件。

#### 场景:摘要文件已存在
- **当** `data/analysis.json` 存在且包含有效的摘要数据
- **那么** `getSummary()` 必须直接返回持久化的摘要，不得检查时间戳，不得触发重新分析

#### 场景:摘要文件不存在
- **当** `data/analysis.json` 不存在
- **那么** `getSummary()` 必须调用 `analyze()` 执行完整分析并返回结果
