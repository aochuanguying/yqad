## 新增需求

### 需求:客户端必须能够回调确认发帖成功
系统必须提供回调端点，允许客户端在实际发布成功后调用，以触发主题使用次数的扣减。

#### 场景:客户端成功发布后回调
- **当** 客户端调用 `POST /api/posts/confirm` 端点，传递有效的 `taskId` 和 `success: true`
- **那么** 系统扣减对应主题的使用次数，返回剩余可用次数

#### 场景:客户端发布失败后回调
- **当** 客户端调用 `POST /api/posts/confirm` 端点，传递 `taskId` 和 `success: false`
- **那么** 系统不扣减次数，仅记录失败日志

#### 场景:回调不存在的任务
- **当** 客户端调用回调端点，传递的 `taskId` 在内存存储中不存在
- **那么** 系统返回 404 错误，提示任务不存在或已过期

#### 场景:重复回调同一任务
- **当** 客户端第二次调用同一 `taskId` 的回调端点
- **那么** 系统返回 404 错误，提示任务已被处理

### 需求:系统必须存储待确认的发帖记录
系统必须在内存中存储已生成但未确认的发帖记录，包含主题 ID、内容、生成时间等信息。

#### 场景：生成发帖内容时创建记录
- **当** 远程 API 调用 `POST /api/posts/generate` 生成内容成功
- **那么** 系统在内存中创建 `PendingPost` 记录，返回 `taskId` 给客户端

#### 场景：批量发帖时创建记录
- **当** 远程 API 调用 `POST /api/posts/batch` 批量生成内容成功
- **那么** 系统为每篇生成的内容创建独立的 `PendingPost` 记录，每个记录返回独立的 `taskId`

#### 场景:记录过期自动清理
- **当** 发帖记录生成时间超过 30 分钟
- **那么** 系统自动删除该记录，且不扣减主题次数

### 需求:回调端点必须使用 API Token 鉴权
系统必须对回调端点实施与生成端点相同的 API Token 鉴权机制。

#### 场景:使用有效 Token 调用回调
- **当** 请求头包含有效的 `Authorization: Bearer <token>`
- **那么** 系统允许调用回调端点

#### 场景:使用无效 Token 调用回调
- **当** 请求头包含无效或缺失的 Token
- **那么** 系统返回 401 未授权错误

## 修改需求

### 需求：生成发帖内容不立即扣减次数
**修改说明：** 原逻辑在生成内容后立即扣减主题次数，现改为仅创建待确认记录，等待客户端回调。

#### 场景：远程 API 生成内容（单个）
- **当** 客户端调用 `POST /api/posts/generate` 生成内容
- **那么** 系统生成内容并存储待确认记录，**不扣减**主题使用次数

#### 场景：远程 API 生成内容（批量）
- **当** 客户端调用 `POST /api/posts/batch` 批量生成内容
- **那么** 系统为每篇内容创建待确认记录，**不扣减**主题使用次数，等待客户端逐个回调
