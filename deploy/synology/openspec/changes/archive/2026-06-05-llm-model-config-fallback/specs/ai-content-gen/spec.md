## 修改需求

### 需求:大模型调用必须支持配置和容错
系统必须支持配置多个 LLM 提供商（优先级有序），当主力提供商失败时自动降级到备用提供商；仅当所有提供商均失败时，才跳过本次生成任务。

#### 场景:可配置的模型参数
- **当** 读取配置文件时
- **那么** 系统必须支持通过 `ai.providers` 数组配置多个提供商，每个提供商包含 `apiKey`、`baseUrl`、`model`、`temperature`、`maxTokens` 等参数；同时必须向后兼容旧版顶层 `ai.*` 单一配置格式

#### 场景:主力提供商调用失败时自动降级
- **当** 主力 LLM 提供商调用失败（超时、限流、服务不可用、认证失败）
- **那么** 系统必须自动切换到下一个配置的备用提供商重新发起调用，并记录 fallback 切换日志；不得直接跳过本次生成

#### 场景:所有提供商均失败时的处理
- **当** 所有配置的 LLM 提供商均已尝试并失败
- **那么** 系统必须记录聚合错误日志，并跳过本次生成任务（行为与原"最多重试2次后跳过"一致）

#### 场景:支持多种兼容 API
- **当** 配置了非 OpenAI 的兼容服务（如通过 NAS 网关代理的 HiGPT、Azure OpenAI、本地模型）
- **那么** 系统必须通过统一的 OpenAI 协议格式正常调用并获取响应，所有提供商均为标准 `baseUrl + apiKey` 接入，无需额外的连接参数
