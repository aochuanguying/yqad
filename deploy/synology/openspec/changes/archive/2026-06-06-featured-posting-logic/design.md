## 上下文

系统当前自动发帖链路位于 [AutoPostService](file:///Users/mac/Library/CloudStorage/SynologyDrive-我的云盘/中转站/yqad/src/services/auto-post.ts)，其核心目标是“定时可用地发出去”，缺少对精华帖的规则化约束与自检；同时发帖生成与素材选择存在明显的降级路径（图片不足会直接发纯文字、话题匹配失败会直接无话题）。

本次变更以官方“精华帖创作攻略”（链接：`https://audi2c.faw-vw.com/h5/#/informationShare?id=1813131516815790082`）为规则来源，将其提炼为可执行的“精华帖规则/准备度评估”，并将该评估嵌入自动发帖流程，形成“优先精华、无法达标则降级普通”的闭环。

从攻略截图可提炼的关键规则（不保证穷尽）：
- 账号要求：注册用户、认证车主会员并绑车；头像/昵称清晰有效（系统不可自动化完成，只能作为运行前置条件提示）。
- 内容要求：原创真实、语句通顺逻辑清晰、不违规；与奥迪品牌及选车/购车/用车/售后/精品等相关且与账号绑车信息一致；排版清晰分层。
- 文案门槛：正文 ≥ 250 字。
- 图片门槛：图片 ≥ 4 张；原创真实高清、无第三方水印；与文案一致，疑似搬运/抄袭会被剔除。
- 创作方向参考：深度干货、用车体验等（用于引导生成内容结构与题材，而不是硬性门槛）。

## 目标 / 非目标

**目标：**
- 自动发帖优先生成“精华候选帖”，并在发布前做可解释的 readiness 判定（通过/未通过 + 原因）。
- 候选帖尽量满足：正文 ≥ 250 字、图片 ≥ 4 张、结构化排版（分段/编号/小标题）、与奥迪用车相关。
- 当关键门槛无法满足（如图片不足、生成内容过短/质量不达标）时，自动降级为普通帖发布，保持系统可用性。
- 关键门槛与重试策略可配置，便于在不同账号/素材库规模下调参。

**非目标：**
- 不承诺“必定获得精华”或“积分必定到账”（精华评选具有人工/规则/风控的不确定性）。
- 不做第三方水印识别、原创性图像取证等复杂能力（仅做流程与约束，素材真实性仍依赖用户素材库）。
- 不自动完成“认证车主/绑车/设置头像昵称”等账号侧前置条件（仅提示与可观测）。

## 决策

### 决策 1：引入“精华规则（policy）+ readiness 评估”作为独立模块

**选择：**
- 新增独立的 Featured Posting Policy（例如 `FeaturedPostPolicy`/`evaluateFeaturedReadiness`），将“规则”和“可解释原因”从 AutoPostService 中抽离。

**理由：**
- 规则来源于外部文档，后续会变更；独立模块便于维护与测试。
- readiness 需要给出“不达标原因”，用于日志、Web UI 展示与调参。

**替代方案：**
- 直接在 AutoPostService 内写 if/else 校验：实现快但会快速膨胀，且难测试/难复用。

### 决策 2：将“精华候选”视为一套可降级的生成策略，而不是单一硬模式

**选择：**
- 引入发帖生成模式：`featured`（精华候选）与 `normal`（普通帖）。
- 先跑 `featured`：更严格的 Prompt + 更高的图片数量目标 + 结构化要求；失败则切到 `normal`。

**理由：**
- 精华门槛有刚性项（字数、图片数），而系统现实约束（素材不足、上传失败、LLM 输出波动）会导致偶发失败；必须有稳定降级路径。

**替代方案：**
- 强制只发精华：一旦素材/网络波动会导致当日“发不出去”，不符合系统稳定性的基本目标。

### 决策 3：图片数量的“达标”以“上传成功的图片 URL 数量”为准，并采用补传重试

**选择：**
- 精华候选需要 `minImages`，以最终 `imageUrls.length` 判定。
- 若第一次上传成功数不足，则从素材库补充新的图片继续上传，直到达到门槛或耗尽重试/素材。

**理由：**
- 本地图片数量 ≠ 上传成功数量（有尺寸、格式、网络等失败因素）。

**替代方案：**
- 只看本地图片数量：会出现“本地>=4但上传<4”的误判，导致发布后不达标。

### 决策 4：字数门槛采用“正文字符数”近似，而非 NLP 分词/字数算法

**选择：**
- 使用 `content.length`（字符数）作为“≥250字”的近似；并在 Prompt 里明确“≥250字”以减少偏差。

**理由：**
- 规则易理解、实现简单、与当前系统的长度约束机制一致（[content-generator.ts](file:///Users/mac/Library/CloudStorage/SynologyDrive-我的云盘/中转站/yqad/src/ai/content-generator.ts) 已使用 length）。

**替代方案：**
- 按中文“字”统计或去除空白/标点：更接近规则，但复杂度更高且易引入争议；可作为后续增强点。

## 风险 / 权衡

- **[无法保证精华评选]** → 将“达标”定义为“满足可验证门槛 + 结构化写作约束”，并明确这只是提高概率。
- **[素材真实性/水印不可控]** → 把“无水印/原创”作为素材库使用约束与运行提示，不做自动识别。
- **[图片上传不稳定导致频繁降级]** → 引入补传重试与可配置阈值；同时在日志中输出失败原因，便于定位网络/图片问题。
- **[LLM 输出过短或结构不佳]** → 精华模式下增加生成重试次数与更明确的输出格式（小标题/编号/分段）。
- **[与现有 post 内容长度限制冲突]** → 现有 `contentLimits.post.max` 可能过低导致难写出结构化 250-400 字；需要在实现中协调默认值与精华目标范围。

## 迁移计划

- 增加配置默认值（精华策略开启但可随时关闭）。
- 兼容旧配置：未配置精华策略时行为保持当前普通发帖逻辑。
- 若上线后发现命中率/风控异常，可通过配置一键关闭精华策略回滚到普通模式（无需代码回滚）。

## 开放问题

- 精华攻略中的“与账号绑车信息一致”的校验能否通过现有 `member/info` 或其它接口获取车型信息并做弱校验？（目前系统全局人设可配置车型，但未强制一致）
- 是否需要将 readiness 结果展示到 Web 管理界面（最近一次发帖：精华候选/普通、失败原因）以便调参？
- “排版清晰分层”的可验证指标采用什么最小可行近似（例如：至少 3 段、至少 1 个编号列表/小标题）？
