## 为什么

当前素材库只有单一路径配置与扫描索引能力，发帖直接消费“原始素材”。这带来三个问题：发帖前需要人工反复处理图片（格式/体积/适配）、素材信息难检索（缺少元数据与内容说明）、以及无法稳定实现“只处理新增素材”的增量流水线。

## 变更内容

- 素材库目录拆分为“原始目录”和“处理后目录”，发帖与素材浏览默认使用“处理后目录”
- 新增“素材梳理”能力：支持每日定时与手工触发，对新增原始素材执行格式转换与压缩、元数据提取、图片内容识别与介绍生成，并将结果写入处理后目录
- 梳理过程采用增量策略：仅处理新增素材；原始素材删除不影响已生成的处理后素材与其梳理信息

## 功能 (Capabilities)

### 新增功能
- `material-processing`: 对原始素材进行增量梳理（转换/压缩、元数据提取、内容识别与介绍生成），并将处理后素材及其可查询信息持久化到处理后目录；支持定时与手工触发

### 修改功能
- `post-materials`: 素材库配置从单一路径调整为原始/处理后双路径；素材列表与发帖素材选择默认基于处理后目录；素材刷新从“全量扫描”扩展为支持“仅扫描新增原始素材并生成处理后产物”
- `scheduler`: 增加“素材梳理任务”的调度配置项与默认策略（每日定时触发），并纳入现有热重载与随机化机制（如适用）

## 影响

- 配置：需要新增 materials.rawPath / materials.processedPath（命名以现有项目配置风格为准），并可能需要扩展 Web UI 配置与展示
- 文件系统：需要定义处理后目录的结构、命名规则、以及梳理信息（元数据/识别结果/介绍）的落盘格式
- API：可能需要新增/扩展接口用于手工触发梳理与查询梳理信息（与现有 /api/materials 体系保持一致）
- 依赖与算力：内容识别/介绍生成可能依赖现有 LLM/内容分析能力；需要考虑离线运行、失败重试与降级策略
