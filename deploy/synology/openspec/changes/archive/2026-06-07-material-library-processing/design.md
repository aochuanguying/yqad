## 上下文

当前系统的素材能力基于 `materials.basePath` 进行扫描与索引，并通过 Web API / UI 提供浏览；发帖在选图与读图时同样以该目录为根目录。这意味着素材“是否适合发帖”（格式、体积、方向、清晰度、是否有可检索的说明）完全依赖人工处理与记忆。

本变更希望把“素材准备”从发帖时的即时工作前置到每日梳理任务中，并把素材库拆分为：

- 原始目录：用于接收/收集素材（不要求格式、体积一致）
- 处理后目录：用于发帖与检索（统一格式/压缩，并附带结构化信息）

系统约束：

- 以文件系统为主（NAS 挂载场景常见），尽量避免引入额外数据库作为必选依赖
- 处理应增量化，仅处理新增原始素材
- 原始素材删除不会反向影响处理后素材与其信息

## 目标 / 非目标

**目标：**

- 目录拆分：提供 `materials.rawPath` 与 `materials.processedPath`（最终命名以现有配置风格校验器为准），并使发帖与素材浏览默认消费处理后目录
- 每日定时梳理：新增“素材梳理任务”，按计划扫描原始目录的新增素材并生成处理后产物
- 手工梳理：提供手工触发入口（HTTP API 或 Web UI 按钮），可在不等待定时任务的情况下对新增素材执行同样的梳理流程
- 可查询：为处理后素材保存可机器读取的梳理信息（元数据 + 识别结果 + 介绍），便于发帖时筛选与复用

**非目标：**

- 不追求对所有图片格式的完美转换与预览一致性（例如 heic/heif 在不同运行环境的兼容差异）
- 不实现复杂的素材去重/相似图聚类（若需要，仅在后续迭代中基于现有 analysis-dedup 能力扩展）
- 不强制引入数据库；若未来规模增长，再评估引入 SQLite/嵌入式索引

## 决策

### 决策 1：用“处理后目录”作为发帖与浏览的唯一默认来源

**选择：**所有 `/api/materials` 相关浏览与 `selectImages/resolveMaterialPaths` 的根目录默认切换到 `materials.processedPath`。  
**原因：**发帖阶段的确定性最重要（格式、体积、可用性），而处理后目录由流水线保证“可用”。

**备选方案：**发帖仍读取原始目录，仅在发帖时做临时转换与压缩。  
**不选原因：**发帖路径会变得不可控（耗时/失败率/资源占用），也无法沉淀可检索信息。

### 决策 2：增量策略采用“原始文件指纹 → 处理记录”清单（Manifest）

**选择：**在处理后目录保存一个清单文件（例如：`<processedPath>/.materials/manifest.json`），记录每个已处理原始素材的指纹与输出信息。  
指纹建议包含：`sourceRelativePath` + `size` + `mtime`（必要时可扩展 `sha256`）。  
**原因：**无需额外数据库；可在原始素材删除后仍保留处理记录；实现“只处理新增”更直接。

**备选方案：**每张图片旁边落一个 sidecar JSON，并以此判断是否已处理。  
**不选原因：**目录遍历成本更高、更新原子性更差；但仍可作为辅助（例如同时落 sidecar 便于就地查看信息）。

### 决策 3：梳理信息落盘格式采用 JSON（可选再生成人读摘要）

**选择：**对每个处理后图片保存结构化 JSON（可放入 `manifest.json` 或额外 `index.jsonl`），字段至少包括：基础元数据（尺寸/方向/格式/体积）、EXIF（如可获取）、内容识别标签、介绍文本、处理时间、来源映射。  
**原因：**便于 API 返回与 Web UI 展示；对后续检索/过滤友好。

**备选方案：**只生成一段介绍文本（纯文本）。  
**不选原因：**缺少结构化字段会限制检索与过滤能力。

### 决策 4：图片转换/压缩优先采用 Node 生态库（必要时允许 CLI 降级）

**选择：**优先引入如 `sharp`（转换/压缩/读取尺寸）+ `exifr` 或 `exiftool-vendored`（元数据提取）等 Node 依赖；在运行环境不支持时，允许降级为“只复制原图 + 仅基础信息”。  
**原因：**系统当前为 Node/TypeScript，保持跨平台一致性；并可在失败时做可控降级，不阻塞整体任务。

**备选方案：**依赖 ImageMagick/ffmpeg 等系统级 CLI。  
**不选原因：**部署与可移植性差；容器/NAS 环境可用性不稳定。

### 决策 5：内容识别与介绍生成复用现有 OpenAI/网关能力

**选择：**在配置允许的情况下，使用现有 OpenAI 客户端（或 openai-compatible-gateway）进行图片内容识别与介绍生成；失败则仅保留元数据与空的识别字段。  
**原因：**减少重复基础设施；同时通过降级保证任务可用。

## 风险 / 权衡

- [处理耗时与资源占用] → 控制每日任务的并发数与单次处理上限；支持断点续跑（基于 manifest）；必要时把“内容识别”拆成独立可选步骤
- [外部 LLM 不可用/成本波动] → 默认可关闭识别；对识别步骤做失败不阻塞的降级；结果缓存避免重复识别
- [不同图片格式兼容性] → 明确支持矩阵与降级策略；对无法转换的格式保留原文件并在索引中标记 `processable=false`
- [原始文件被修改导致重复处理] → 指纹包含 mtime/size；若需要严格一致可引入 `sha256`（代价：读取全文件）
- [清单文件损坏] → 采用写入临时文件后原子 rename；定期备份（如 `.bak`）

## 迁移计划

- 配置层：保留 `materials.basePath` 一段过渡期（如存在则作为 `processedPath` 默认值），并在日志中提示用户迁移到新字段
- 数据层：首次启用时，若 `processedPath` 为空目录，可选择一次性“全量梳理”或仅从新增开始（由配置开关决定）
- 回滚：回滚到旧版本时仍可继续使用原始目录；处理后目录与 manifest 作为额外产物不影响旧逻辑（若旧逻辑仍指向 basePath）

## 待定问题

- 配置命名是否使用 `materials.rawPath/materials.processedPath`，或继续沿用 `materials.basePath` 并新增 `materials.processedPath`（需与现有 config-validator 风格一致）
- “处理后目录”的目录结构是否与原始目录完全一致（建议一致），以及输出格式统一为 jpeg/webp 的优先级
- 手工梳理入口形态：仅 API（更轻）还是同时提供 Web UI 按钮
- 素材梳理信息的查询形态：是否需要新增 `/api/materials/info?path=` 或在列表接口中直接携带摘要字段
