## 修改需求

### 需求: 系统必须提供素材库管理能力
系统必须支持配置“原始素材目录”和“处理后素材目录”。系统必须以处理后素材目录作为素材库浏览与发帖的默认数据源。

#### 场景: 配置素材库路径
- **当** 用户在配置中设置 materials.rawPath 与 materials.processedPath
- **那么** 系统必须将 rawPath 作为原始素材输入根目录
- **并且** 系统必须将 processedPath 作为处理后素材输出与消费根目录
- **并且** 系统必须支持 rawPath 与 processedPath 使用绝对路径或相对路径

#### 场景: 兼容旧版单路径配置
- **当** 用户仅在配置中设置 materials.basePath
- **那么** 系统必须将该路径视为 materials.processedPath 的默认值
- **并且** 系统必须记录一次迁移提示日志，建议用户改用 rawPath/processedPath

#### 场景: 扫描素材目录
- **当** 系统启动或收到素材刷新请求
- **那么** 系统必须递归扫描处理后素材目录下的所有图片文件（jpg、jpeg、png、gif、webp、heic、heif），并建立索引

#### 场景: 处理后素材目录不存在
- **当** 配置的处理后素材目录不存在或不可访问
- **那么** 系统必须记录警告日志，素材列表返回空数组，禁止影响其他功能正常运行

### 需求: 系统必须提供素材浏览 API
系统必须通过 HTTP API 暴露处理后素材目录的浏览能力。

#### 场景: 获取素材列表
- **当** 客户端发送 GET /api/materials 请求
- **那么** 系统必须返回处理后素材目录中所有图片的列表，包含文件名、相对路径、文件大小
- **并且** 系统必须返回稳定排序的 items：先按 directory 升序排序（根目录视为最小值），再按 filename 升序排序
- **并且** 系统必须将以下扩展名视为图片素材并纳入列表：jpg、jpeg、png、gif、webp、heic、heif

#### 场景: 按目录筛选素材
- **当** 客户端发送 GET /api/materials?dir=travel 请求
- **那么** 系统必须仅返回指定子目录下的素材列表
- **并且** 返回 items 的排序规则必须与“获取素材列表”场景一致

#### 场景: 获取素材图片
- **当** 客户端发送 GET /api/materials/file/:path 请求
- **那么** 系统必须返回处理后素材目录中对应素材文件的二进制内容，并设置正确的 Content-Type（至少支持 image/jpeg、image/png、image/gif、image/webp、image/heic、image/heif）

### 需求: Web UI 必须提供素材库浏览界面
系统必须在 Web 管理界面中提供处理后素材目录的可视化浏览功能。

#### 场景: 素材库 Tab 页展示
- **当** 用户点击"素材库"Tab
- **那么** 页面必须以网格缩略图方式展示所有可用素材
- **并且** 页面在“全部目录”模式下必须按素材的 directory（目录）分组展示（每个分组展示目录名称，根目录显示为“根目录”）
- **并且** 每个目录分组内素材必须按 filename 升序展示
- **并且** 对浏览器无法直接预览的素材格式（例如部分环境下的 heic/heif），页面必须提供可用的降级方式（例如展示可点击的“打开文件”入口），禁止导致整页素材区域不可用

#### 场景: 刷新素材索引
- **当** 用户点击"刷新"按钮
- **那么** 系统必须重新扫描处理后素材目录并更新素材列表
