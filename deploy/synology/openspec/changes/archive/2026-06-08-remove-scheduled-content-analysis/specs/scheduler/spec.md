## 修改需求

### 需求: 系统必须提供灵活的定时任务调度能力
系统必须支持配置签到、自动评论、自动发帖和素材梳理任务的执行时间、频率和优先级，统一管理自动化任务的调度。调度器必须支持运行时配置热重载，并且禁止注册内容分析定时任务。

#### 场景: 配置签到任务时间
- **当** 系统启动时读取调度配置
- **那么** 系统必须按配置的cron表达式（默认每天8:00-9:00随机时间）触发签到任务

#### 场景: 配置评论任务时间
- **当** 系统启动时读取调度配置
- **那么** 系统必须按配置的cron表达式（默认每天10:00-20:00区间内随机分散）触发评论任务

#### 场景: 配置发帖任务时间
- **当** 系统启动时读取调度配置
- **那么** 系统必须按配置的cron表达式（默认每天12:00-18:00随机时间）触发发帖任务

#### 场景: 配置素材梳理任务时间
- **当** 系统启动时读取调度配置
- **那么** 系统必须按配置的cron表达式触发素材梳理任务

#### 场景: 禁止注册内容分析定时任务
- **当** 系统启动并创建调度器
- **那么** 调度器禁止注册名为“内容分析”的定时任务

#### 场景: 配置热重载后重新调度
- **当** 系统收到配置变更事件且调度相关配置发生变化
- **那么** 系统必须取消当前所有已注册的定时任务，使用新配置重新创建并注册签到、自动评论、自动发帖和素材梳理任务，禁止丢失当天已执行的任务记录

#### 场景: 热重载忽略内容分析配置
- **当** 系统收到包含内容分析调度字段的历史配置或外部配置
- **那么** 调度器必须忽略该字段，禁止创建或恢复内容分析定时任务

### 需求: 调度器必须支持随机化以模拟人工行为
系统必须在配置的时间窗口内引入随机偏移，避免固定执行模式。

#### 场景: 执行时间随机偏移
- **当** 任务到达配置的触发时间
- **那么** 系统必须在配置的偏移范围内（默认±30分钟）添加随机延时后再执行

#### 场景: 系统启动后立即检查遗漏任务
- **当** 系统启动或重启
- **那么** 系统必须检查当天是否有应执行但未执行的签到、自动评论、自动发帖和素材梳理任务，如有则立即执行，并且禁止补执行内容分析任务
