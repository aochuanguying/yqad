## 上下文

当前素材库页面通过 `GET /api/materials` 拉取全部素材后，`renderMaterials()` 函数一次性生成所有素材卡的 HTML 并插入 DOM。对于 150+ 张素材，这会生成 750-1200 个 DOM 节点，同时触发 150+ 个图片请求，导致：
- 首屏渲染时间长（DOM 构建 + 图片解析）
- 滚动卡顿（浏览器需要维护和绘制大量节点）
- 内存占用高（所有节点和图片数据驻留内存）

用户主要在"滚动浏览时"感受到卡顿，说明问题集中在渲染性能和图片加载策略上。

## 目标 / 非目标

**目标：**
- 实现素材列表分页加载，初始仅渲染 24 个素材
- 使用 Intersection Observer API 增强图片懒加载
- 显示加载进度，提供"加载更多"交互
- 首屏渲染时间减少 80%+，滚动帧率显著提升
- 保持现有 API 不变，仅优化前端渲染逻辑

**非目标：**
- 不修改后端 API（`GET /api/materials` 仍返回全部素材）
- 不引入虚拟滚动（复杂度高于分页，且分页已足够解决问题）
- 不改变素材分组展示逻辑（仍按目录分组）
- 不修改素材选择器、发帖等其他功能

## 决策

### 1. 前端分页 vs 后端分页
- **方案 A：后端分页** - API 支持 `?page=1&limit=24` 参数
  - 优点：传输数据量小，适合超大规模素材库
  - 缺点：需要修改后端 API，增加复杂度
- **方案 B：前端分页**（本次选择）- API 仍返回全部素材，前端分批渲染
  - 优点：实现简单，后端零改动，150 张素材的 JSON 体积小（约 30KB）
  - 缺点：首次仍需传输全部数据（但对 150 张规模影响不大）
  - **理由**：当前素材规模（100-500 张）下，前端分页性价比最高

### 2. 分页大小
- **选择：24 个/页**
  - 网格布局：每行 2-6 个（响应式），24 个约 4-12 行
  - 首屏可见：约 12 个（6 列 × 2 行）
  - 缓冲区：2 倍首屏，平衡性能和用户体验
  - 可调整：后续可根据实际数据优化

### 3. 图片懒加载方案
- **方案 A：原生 `loading="lazy"`**（已有）
  - 优点：浏览器原生支持，无需额外代码
  - 缺点：控制粒度粗，无法自定义触发时机
- **方案 B：Intersection Observer API**（本次新增）
  - 优点：精确控制加载时机，支持预加载缓冲区
  - 缺点：需要额外代码实现
  - **理由**：与分页配合，可进一步优化图片���载节奏

### 4. 进度显示策略
- **选择：显示"已显示 X/Y 个素材"**
  - 位置：右上角，与素材总数统计并列
  - 更新时机：每次加载后更新
  - 加载完毕：显示"共 Y 个素材"，隐藏"加载更多"按钮

## 风险 / 权衡

**[风险] 前端分页在素材极多时（1000+）仍有性能问题**
→ **缓解**：当前规模 100-500，分页已足够；未来如超 1000 张，再考虑后端分页或虚拟滚动

**[风险] Intersection Observer 兼容性**
→ **缓解**：现代浏览器均支持，降级方案仍使用 `loading="lazy"`

**[风险] 分组渲染 + 分页逻辑复杂**
→ **缓解**：按目录分组后，每组独立分页或整体分页？选择整体分页（跨组），简化逻辑

**[权衡] 首次仍传输全部数据**
→ **接受**：150 张素材 JSON 约 30KB，传输时间可忽略；分页主要优化渲染而非传输
