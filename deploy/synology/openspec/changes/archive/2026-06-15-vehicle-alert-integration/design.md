## 上下文

**当前状态：**
- 车辆监控系统已实现异常检测功能（车窗未关、车辆移动、安全距离超标等）
- 报警功能仅记录日志，无实际通知能力
- 已有 Android Telecom API 服务可在 root 的 Android 设备上发送短信和拨打电话
- Web 管理界面缺少告警配置功能

**约束条件：**
- Android Telecom API 部署在独立的 Android 设备上，需通过网络访问
- 需要 API Token 认证确保安全性
- 告警通知需及时可靠，避免漏报

**利益相关者：**
- 车主：需要及时接收车辆异常报警
- 系统管理员：需要配置和管理告警通知

## 目标 / 非目标

**目标：**
1. 实现 Android Telecom API 的完整集成
2. 提供 Web 界面配置 API 地址和 Token
3. 实现短信 + 电话的双重通知机制
4. 记录告警历史便于追溯

**非目标：**
1. 替换现有的异常检测逻辑
2. 支持多个告警接收人（当前仅支持单个手机号）
3. 告警通知的语音合成（使用系统默认语音）

## 决策

### 1. Telecom API 客户端设计

**决策：** 创建独立的 `telecom-client.ts` 服务模块

**理由：**
- 职责分离：将 API 调用逻辑与监控逻辑分离
- 便于测试：可独立测试 API 客户端
- 易于维护：API 变更时仅需修改客户端模块

**考虑过的替代方案：**
- 直接在 `vehicle-monitor-service.ts` 中调用：会导致代码耦合，不利于维护
- 使用第三方 HTTP 客户端库：项目已有 axios，无需新增依赖

### 2. 通知策略：短信优先，电话跟进

**决策：** 异常发生时，先发送短信，间隔 5 秒后拨打电话

**理由：**
- 短信可作为书面记录，方便用户查看异常详情
- 电话提供即时提醒，确保用户不会遗漏重要报警
- 5 秒间隔避免 API 请求过于密集

**考虑过的替代方案：**
- 仅发送短信：可能无法及时提醒用户
- 仅拨打电话：用户可能错过电话，且无文字记录
- 同时发送：可能造成资源浪费，且电话可能打断短信接收

### 3. 配置存储方式

**决策：** 使用 YAML 配置文件存储 API 地址和 Token

**理由：**
- 与现有配置系统保持一致（`config/default.yaml`）
- 支持热重载，修改配置后无需重启服务
- 配置文件可版本控制（敏感信息除外）

**考虑过的替代方案：**
- 数据库存储：增加复杂度，配置变���需要数据库连接
- 环境变量：修改后需要重启服务
- Web 界面配置 + 文件存储：最佳方案，用户友好且易于管理

### 4. 告警历史记录

**决策：** 使用内存数组存储最近的告警记录，通过 API 提供查询接口

**理由：**
- 实现简单，无需数据库依赖
- 性能优秀，无 I/O 开销
- 保留最近 100 条记录，满足追溯需求

**考虑过的替代方案：**
- 文件存储：增加 I/O 开销，可能影响告警实时性
- 数据库存储：增加依赖和复杂度
- 仅日志记录：不便于查询和展示

### 5. 错误处理与重试

**决策：** API 调用失败时记录错误日志，不自动重试

**理由：**
- 告警通知对实时性要求高，重试可能延误后续通知
- 连续告警周期会自动触发新的通知
- 简化实现，避免复杂的重试逻辑

**考虑过的替代方案：**
- 自动重试 3 次：可能延误告警，增加系统负担
- 切换到备用通知渠道：当前无其他可用渠道

## 风险 / 权衡

**[风险] Android Telecom API 服务不可用**
→ **缓解措施：** 在 Web 界面显示 API 连接状态，定期检查服务可用性

**[风险] 网络延迟导致通知延误**
→ **缓解措施：** 设置合理的超时时间（10 秒），失败时记录日志

**[风险] Token 泄露**
→ **缓解措施：** 配置文件设置适当权限（chmod 600），Web 界面不直接显示完整 Token

**[风险] 短信/电话费用**
→ **权衡：** 仅在实际异常时触发，冷却时间 30 分钟避免频繁通知

**[风险] Android 设备电量或网络问题**
→ **缓解措施：** 建议将 Android 设备保持充电状态并连接稳定 WiFi
