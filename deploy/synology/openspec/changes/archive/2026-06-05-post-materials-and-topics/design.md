## 上下文

系统已有一个 Web 管理界面（Express + 内联 HTML），通过 Basic Auth 保护。当前自动发帖功能完全依赖 AI 从社区热点自由生成内容。用户需要更精细的发帖控制：预设主题方向、配图素材，使内容更有针对性。系统运行在内网环境，认证和脱敏属于不必要的复杂度。

现有技术栈：TypeScript、Express、yaml、winston、openai、node-cron。

## 目标 / 非目标

**目标：**
- 移除 Web 认证和字段脱敏，简化内网使用
- 实现素材库管理（扫描 NAS 路径下的图片、提供浏览 API 和 UI）
- 实现发帖主题配置（CRUD、素材关联、使用状态追踪）
- 改造发帖流程为"主题优先、自由生成兜底"模式

**非目标：**
- 不实现图片上传功能（用户直接将文件放入 NAS 路径）
- 不实现图片编辑或裁剪
- 不实现主题的自动推荐或智能排序
- 不实现素材的标签分类系统（仅按目录组织）

## 决策

### 1. 素材存储方案：文件系统直接读取

**选择**：直接读取配置路径下的文件系统
**替代方案**：数据库存储素材元数据、对象存储服务
**理由**：用户已有 NAS 基础设施，将图片直接放入目录是最自然的工作流。无需额外的上传流程或存储服务。启动时扫描建立内存索引，按需刷新即可。

### 2. 主题数据存储：JSON 文件

**选择**：`data/topics.json` 存储主题配置和状态
**替代方案**：SQLite、YAML 配置文件、独立数据库
**理由**：主题数据量小（预计几十到几百条），JSON 文件足够。与现有数据存储模式一致（token.json、summaries）。读写简单，无需引入数据库依赖。

### 3. 素材图片访问：Express 静态文件服务

**选择**：通过 Express 路由代理访问素材文件，路径为 `/api/materials/file/:path`
**替代方案**：直接 express.static 挂载素材目录、nginx 代理
**理由**：NAS 路径可能在项目外部（如 `/mnt/nas/materials`），不适合用 static 中间件。路由代理方式可以控制访问范围，防止路径穿越攻击。

### 4. 主题选择策略：顺序消费 + FIFO

**选择**：按创建时间顺序消费未使用的主题（先创建先使用）
**替代方案**：随机选取、按权重选取、用户指定优先级
**理由**：简单可预测。用户按发帖计划顺序创建主题，按顺序消费符合直觉。避免引入额外的排序逻辑。

### 5. 认证移除：直接删除中间件

**选择**：移除 `basicAuth` 中间件引用，移除 `web.auth` 配置项
**替代方案**：保留认证但设为可选（enabled: false）
**理由**：用户明确表示内网无需认证。保留可选开关只是无用的复杂度。彻底移除使代码更清晰。

## 风险 / 权衡

- **[NAS 路径不可达]** → 启动时仅记录警告，素材列表返回空。不阻塞其他功能。用户可通过 Web UI 查看是否有素材来确认连通性。
- **[素材文件过多导致扫描慢]** → 使用内存缓存索引，仅在手动刷新或启动时扫描。对于几千张图片的场景延迟可接受（< 1s）。
- **[topics.json 并发写入]** → 复用现有的内存写入锁模式。单用户场景下风险极低。
- **[路径穿越攻击]** → 素材文件路由对请求路径做 `path.resolve` 后校验是否在 basePath 范围内，拒绝越界访问。
- **[移除认证后的安全风险]** → 用户确认内网部署，风险由网络层隔离承担。
