## 新增需求

### 需求: 系统必须提供配置管理的 HTTP API
系统必须通过 RESTful API 暴露所有配置项的读取和修改能力，支持按分组查询和更新。

#### 场景: 获取全部配置
- **当** 客户端发送 GET /api/config 请求
- **那么** 系统必须返回当前所有配置项的 JSON 格式数据，按分组结构组织

#### 场景: 获取单个配置分组
- **当** 客户端发送 GET /api/config/:group 请求（如 /api/config/ai）
- **那么** 系统必须仅返回该分组下的配置项

#### 场景: 更新配置项
- **当** 客户端发送 PUT /api/config/:group 请求并携带有效的 JSON 请求体
- **那么** 系统必须验证配置值的合法性，将更新持久化到 config/default.yaml，并返回更新后的配置

#### 场景: 配置验证失败
- **当** 客户端提交的配置值不符合约束（如 timeout 为负数、cron 表达式格式错误）
- **那么** 系统必须返回 400 状态码和具体的验证错误信息，禁止写入无效配置

### 需求: 系统必须提供配置管理的 Web 页面
系统必须提供一个可视化的单页面应用，以表单方式展示和编辑所有配置项。

#### 场景: 访问配置页面
- **当** 用户通过浏览器访问根路径 /
- **那么** 系统必须返回配置管理的 HTML 页面，页面必须按配置分组展示所有可编辑字段

#### 场景: 分组展示配置
- **当** 页面加载完成
- **那么** 页面必须将配置项按以下分组展示：API设置、用户认证、AI模型、签到、评论、发帖、内容分析、调度、日志、内容长度约束

#### 场景: 编辑并保存配置
- **当** 用户修改表单字段并点击保存按钮
- **那么** 系统必须将修改的配置发送到后端 API 并显示保存成功或失败的提示

#### 场景: 敏感信息脱敏展示
- **当** 页面展示密码或 API Key 等敏感字段
- **那么** 系统必须以密码输入框（type=password）方式展示，禁止明文显示

### 需求: 配置变更必须触发服务热重载
系统必须在配置通过 Web UI 保存后，通知相关模块重新加载配置，无需重启服务。

#### 场景: 配置保存后热重载
- **当** 配置通过 API 成功持久化到文件
- **那么** 系统必须发出配置变更事件，所有订阅该事件的模块必须使用新配置重新初始化

#### 场景: 热重载失败回滚
- **当** 模块使用新配置重新初始化失败
- **那么** 系统必须回滚到上一个有效配置，并在 API 响应和日志中报告错误

### 需求: Web 服务器必须具备基本安全防护
系统必须对 Web 管理界面提供基本的访问控制。

#### 场景: 未授权访问拒绝
- **当** 请求未携带有效的认证信息访问 API
- **那么** 系统必须返回 401 状态码，禁止访问配置数据

#### 场景: 配置简单认证凭据
- **当** 系统启动时读取 Web 服务器配置
- **那么** 系统必须使用配置的用户名和密码进行 HTTP Basic Auth 验证
