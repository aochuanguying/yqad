## 上下文

当前项目已有完整的自动发帖系统（AutoPostService）和远程发帖 API，但存在以下缺失：
1. 用户无法在 Web 管理页面手动触发发帖，只能通过定时任务或远程 API
2. 发帖后没有完整的日志记录，用户无法追溯历史发帖记录
3. 缺乏发帖统计和审计能力

现有架构：
- **服务层**：`AutoPostService` 负责发帖逻辑，`PendingPostService` 负责待确认记录管理
- **API 层**：`posts-routes.ts` 提供生成、批量生成、确认回调等接口
- **调度层**：Scheduler 负责定时触发发帖任务
- **UI 层**：Web 管理页面提供配置管理，但无手动发帖入口

## 目标 / 非目标

**目标：**
1. 实现手工立即发帖功能，参考立即评论的实现方式
2. 实现发帖日志功能，记录完整发帖信息并支持页面查看
3. 保持与现有自动发帖逻辑的兼容性
4. 支持日志持久化和分页查询

**非目标：**
1. 不改变现有的自动发帖调度逻辑
2. 不改变远程发帖 API 的回调机制
3. 不涉及发帖内容生成算法的优化
4. 不实现日志的导出或统计分析功能（仅基础查看）

## 决策

### 1. 手工发帖 API 设计

**决策**：新增 `POST /api/posts/execute` 接口，同步执行发帖任务

**理由**：
- 参考立即评论接口（`POST /api/comment/execute`）的设计，保持一致性
- 同步执行，用户可立即获得发帖结果
- 使用 Session 登录状态鉴权，与立即评论保持一致

**替代方案**：
- 异步执行 + 轮询：增加复杂度，用户体验不佳
- 复用现有生成接口：无法区分手动/自动触发，且逻辑耦合

### 2. 发帖日志数据结构

**决策**：采用扁平化结构，包含以下字段：
```typescript
interface PostLog {
  id: string;              // 日志 ID（UUID）
  timestamp: number;       // 发帖时间戳
  triggerType: 'auto' | 'manual';  // 触发方式
  postType: 'topic' | 'free';      // 发帖类型：主题发帖/自由发帖
  mode: 'normal' | 'featured';     // 发帖模式
  topicId?: string;        // 主题 ID（如为主题发帖）
  topicName?: string;      // 主题名称（如为主题发帖）
  title: string;           // 帖子标题
  content: string;         // 帖子内容
  imageUrls: string[];     // 图片 URL 列表
  status: 'success' | 'failed';    // 发帖状态
  errorMessage?: string;   // 错误信息（如失败）
  taskId?: string;         // 任务 ID
  createdAt: number;       // 记录创建时间
}
```

**理由**：
- 包含所有关键信息，支持完整追溯
- 扁平化结构便于查询和展示
- 使用数字时间戳便于排序和筛选

### 3. 日志持久化方案

**决策**：使用 JSON 文件持久化（`data/post-logs.json`），类似 `pending-posts.json`

**理由**：
- 与现有数据持久化方案保持一致
- 实现简单，无需引入数据库依赖
- 支持原子写入和定期清理

**替代方案**：
- SQLite：增加依赖复杂度，对于日志量不大的场景过度设计
- 内存存储：重启后丢失，无法追溯历史

### 4. 日志查询 API 设计

**决策**：提供两个接口
- `GET /api/posts/logs?page=1&limit=20&triggerType=all`：分页查询日志列表
- `GET /api/posts/logs/:id`：查询单条日志详情

**理由**：
- 分页查询避免大数据量传输
- 支持按触发方式筛选
- 详情接口支持快速定位特定日志

### 5. 日志记录时机

**决策**：
- **自动发帖**：在 `AutoPostService.performDailyPosts()` 中每条发帖完成后记录
- **手工发帖**：在 `performSinglePost()` 完成后记录
- **远程 API 发帖**：在回调接口 `/api/posts/confirm` 中记录最终状态

**理由**：
- 确保所有发帖路径���被记录
- 回调时记录可确保状态准确（成功/失败）

### 6. UI 布局设计

**决策**：
- 在"发帖"标签页的"保存配置"按钮左侧添加"立即发帖"按钮（与立即评论布局一致）
- 新增"📝 发帖日志"标签页，与"评论"标签页平行
- 日志页面支持：时间范围筛选、触发方式筛选、分页、查看详情

**理由**：
- 与立即评论保持一致的交互模式，降低学习成本
- 独立标签页提供完整的日志管理功能

## 风险 / 权衡

**风险 1**：日志文件随时间增长可能变大

**缓解措施**：
- 实现定期清理机制（保留最近 1000 条或 30 天）
- 日志文件采用追加写入 + 定期压缩策略
- 未来可支持日志归档

**风险 2**：手工发帖与自动发帖可能并发执行

**缓解措施**：
- 使用 `isPostTaskRunning` 标志位防止并发
- 手工发帖时检查当前是否有发帖任务在执行
- 提示用户"发帖任务正在执行中，请稍后再试"

**风险 3**：日志持久化失败可能影响发帖主流程

**缓解措施**：
- 日志记录采用异步非阻塞方式
- 日志失败不影响发帖主流程，仅记录警告日志
- 提供日志服务健康检查

**权衡**：日志查询性能 vs 实现复杂度

**说明**：当前采用 JSON 文件存储，查询性能随数据量增长下降。但考虑到发帖频率（每日 1-3 条），30 天最多 90 条数据，JSON 文件完全够用。如未来数据量增长，可迁移至 SQLite。
