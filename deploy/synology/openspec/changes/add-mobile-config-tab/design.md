## 上下文

当前项目已有 AutoJS 配置功能（独立页面），但缺少手机短信和未接电话的记录管理。项目采用 Node.js + Express + TypeScript 技术栈，使用 MySQL 存储结构化数据，已有成熟的 API 路由和认证机制（API Token）。

**约束：**
- 保持与现有 API 风格一致（RESTful，统一响应格式）
- 复用现有的 MySQL 连接管理和认证中间件
- 前端使用原生 JavaScript + Tailwind CSS，无需构建
- 数据库配置已存在于 config/default.yaml

## 目标 / 非目标

**目标：**
- 在首页添加"手机配置"页签，整合 AutoJS 配置
- 实现短信和未接电话记录的 API 接口
- 创建 MySQL 数据表存储记录
- 复用现有 AutoJS 配置功能

**非目标：**
- 不提供短信和未接电话的 Web 管理界面（仅 API）
- 不修改 AutoJS 配置的核心功能
- 不涉及数据同步或推送机制

## 决策

### 1. 数据库表设计
**决策**：创建 `mobile_sms` 和 `missed_calls` 表，采用与现有表一致的命名和结构规范。

**理由**：保持数据库设计一致性，便于维护。

**替代方案**：
- 合并到一张表：❌ 短信和电话是不同实体，分开更清晰
- 使用 Redis 存储：❌ 需要持久化查询，MySQL 更合适

### 2. API 路由设计
**决策**：采用 RESTful 风格，路径为 `/api/posts/mobile/sms` 和 `/api/posts/mobile/calls/missed`。

**理由**：与现有 API 风格保持一致（如 `/api/posts/autojs/config`）。

**替代方案**：
- `/api/mobile/sms`：❌ 不符合现有路由组织方式
- RPC 风格：❌ 项目已采用 RESTful 规范

### 3. 前端页面组织
**决策**：首页导航新增"手机配置"页签，点击后显示包含三个子功能的页面。

**理由**：保持单页应用风格，与现有页面结构一致。

**替代方案**：
- 独立页面：❌ 增加导航复杂度
- 模态框：❌ 不适合多子功能的场景

### 4. 认证方式
**决策**：使用现有的 `apiTokenMiddleware` 进行认证。

**理由**：需求明确要求使用 API Token，与 AutoJS 配置保持一致。

## 风险 / 权衡

**数据量增长** → 定期清理旧数据（可在后续版本添加自动清理机制）

**API 滥用** → 依赖现有的频率限制中间件（已在其他接口启用）

**前端代码复用** → AutoJS 配置页面需要小幅改造以适配页签结构，但改动最小化
