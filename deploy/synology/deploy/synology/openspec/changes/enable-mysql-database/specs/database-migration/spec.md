## 新增需求

### 需求:数据库迁移脚本管理

系统必须使用 SQL 脚本管理数据库表结构变更，支持版本控制、回滚和幂等执行。

#### 场景:创建迁移脚本
- **当** 需要创建新表时
- **那么** 系统必须创建带版本号的 SQL 脚本（如 001_create_members_table.sql）
- **那么** 系统必须包含 Up 迁移（创建表）
- **那么** 系统必须包含 Down 迁移（删除表）
- **那么** 系统必须使用 IF NOT EXISTS 保证幂等性

#### 场景:执行迁移
- **当** 应用启动时
- **那么** 系统必须检查 migration 表记录当前版本
- **那么** 系统必须按版本号顺序执行未执行的迁移脚本
- **那么** 系统必须记录已执行的迁移版本
- **那么** 系统必须在一个事务中执行每个迁移（如果支持）

#### 场景:迁移回滚
- **当** 需要回滚迁移时
- **那么** 系统必须执行 Down 迁移脚本
- **那么** 系统必须更新 migration 表记录
- **那么** 系统必须保证回滚不影响其他表
- **那么** 系统必须记录回滚日志

#### 场景:迁移状态查询
- **当** 管理员查询迁移状态时
- **那么** 系统必须返回已执行的迁移列表
- **那么** 系统必须返回每个迁移的执行时间
- **那么** 系统必须返回待执行的迁移列表

#### 场景:迁移失败处理
- **当** 迁移执行失败时
- **那么** 系统必须回滚当前迁移
- **那么** 系统必须记录详细的错误日志
- **那么** 系统必须阻止应用继续启动
- **那么** 系统必须保留现场用于调试

### 需求:表结构初始化

系统必须初始化所有必要的表结构、索引和约束。

#### 场景:会员表初始化
- **当** 执行会员表迁移时
- **那么** 系统必须创建 members 表
- **那么** 系统必须创建主键索引（id）
- **那么** 系统必须创建唯一索引（username, email）
- **那么** 系统必须创建普通索引（member_level, expires_at）
- **那么** 系统必须设置默认值和自增字段

#### 场景:帖子表初始化
- **当** 执行帖子表迁移时
- **那么** 系统必须创建 posts 表
- **那么** 系统必须创建主键索引（id）
- **那么** 系统必须创建外键约束（member_id → members.id）
- **那么** 系统必须创建索引（member_id, status, published_at）
- **那么** 系统必须设置枚举类型（status）

#### 场景:评论表初始化
- **当** 执行评论表迁移时
- **那么** 系统必须创建 comments 表
- **那么** 系统必须创建主键索引（id）
- **那么** 系统必须创建外键约束（post_id → posts.id, member_id → members.id, parent_id → comments.id）
- **那么** 系统必须创建索引（post_id, member_id, parent_id）
- **那么** 系统必须设置级联删除策略

#### 场景:迁移记录表初始化
- **当** 首次执行迁移时
- **那么** 系统必须创建 schema_migrations 表
- **那么** 系统必须记录迁移版本号
- **那么** 系统必须记录执行时间
- **那么** 系统必须记录执行状态（success/failed）

### 需求:数据种子

系统必须支持初始化基础数据，如管理员账号、默认配置等。

#### 场景:管理员账号初始化
- **当** 首次部署时
- **那么** 系统必须创建默认管理员账号
- **那么** 系统必须设置强密码（环境变量配置）
- **那么** 系统必须设置会员等级为 vip
- **那么** 系统必须设置永不过期

#### 场景:基础配置初始化
- **当** 首次部署时
- **那么** 系统必须插入基础配置数据
- **那么** 系统必须插入默认分类
- **那么** 系统必须插入默认标签

#### 场景:测试数据初始化
- **当** 测试环境部署时
- **那么** 系统必须可选择性地插入测试数据
- **那么** 系统必须插入测试会员账号
- **那么** 系统必须插入测试帖子和评论

### 需求:迁移版本控制

系统必须管理迁移脚本的版本，支持升级、降级和版本查询。

#### 场景:版本升级
- **当** 执行 migrate:up 命令时
- **那么** 系统必须执行所有未执行的 Up 迁移
- **那么** 系统必须按版本号升序执行
- **那么** 系统必须更新当前版本号

#### 场景:版本降级
- **当** 执行 migrate:down 命令时
- **那么** 系统必须回滚最后一个迁移
- **那么** 系统必须执行对应的 Down 迁移
- **那么** 系统必须更新当前版本号
- **那么** 系统必须支持回滚指定版本

#### 场景:版本查询
- **当** 执行 migrate:status 命令时
- **那么** 系统必须显示当前数据库版本
- **那么** 系统必须显示已执行的迁移列表
- **那么** 系统必须显示待执行的迁移列表

#### 场景:版本重置
- **当** 执行 migrate:reset 命令时
- **那么** 系统必须回滚所有迁移
- **那么** 系统必须从第一个迁移重新执行
- **那么** 系统必须清除 schema_migrations 表

### 需求:迁移安全性

系统必���保证迁移过程的安全性，防止数据丢失和误操作。

#### 场景:生产环境保护
- **当** 在生产环境执行迁移时
- **那么** 系统必须要求二次确认
- **那么** 系统必须备份相关数据
- **那么** 系统必须在低峰期执行
- **那么** 系统必须支持快速回滚

#### 场景:破坏性操作保护
- **当** 执行 DROP TABLE 操作时
- **那么** 系统必须要求显式确认
- **那么** 系统必须先导出数据
- **那么** 系统必须提供恢复方案

#### 场景:数据备份
- **当** 执行重大迁移前
- **那么** 系统必须自动备份相关表
- **那么** 系统必须验证备份完整性
- **那么** 系统必须保留备份到迁移成功后 N 天

#### 场景:权限控制
- **当** 执行迁移时
- **那么** 系统必须使用有足够权限的数据库账号
- **那么** 系统必须禁止普通用户执行迁移
- **那么** 系统必须记录操作者信息
