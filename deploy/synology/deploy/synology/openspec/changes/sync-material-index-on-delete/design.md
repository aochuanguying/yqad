## 上下文

当前素材整理流程在 `processMaterials()` 结束时统一重建 `.materials/index.json`，并将素材元数据拆分存放到 `.materials/manifest.json` 与 `.materials/info/**/*.json`。Web 端素材列表通过 `materials-service` 优先读取索引，AI 选图通过 `image-selector` 直接依赖索引项中的 `searchableText`、`tags` 和 `intro` 做匹配。

现状中存在两个一致性缺口：

1. 处理后素材一旦在整理任务之外被删除，索引和关联记录不会自动清理。
2. 浏览和 AI 检索默认信任索引内容，可能继续返回已经不存在的素材路径。
3. Web 页面当前只有“整理素材”按钮，缺少用户可主动触发的索引诊断与修复入口。

这个变更横跨素材处理、Web 素材服务和 AI 选图模块，属于跨模块一致性修复。

## 目标 / 非目标

**目标：**
- 为“删除整理后素材”提供统一的一致性更新流程
- 保证索引、manifest、info 记录与处理后素材文件保持同步
- 保证素材浏览和 AI 检索不会返回失效路径
- 在发生人工删除或历史脏数据时提供自愈能力
- 在素材页面提供显式“重建索引”入口，支持用户主动修复异常数据

**非目标：**
- 不重构现有素材整理算法或图片转换流程
- 不引入数据库或外部索引服务
- 不处理原始素材目录的删除联动
- 不改变 AI 选图的评分算法本身

## 决策

### 1. 使用统一的“删除并同步清理”服务作为主路径

**决策：** 为处理后素材新增统一删除服务，集中完成文件删除、manifest 清理、info 清理与索引重建或增量更新。

**理由：**
- 避免删除逻辑散落在路由或 UI 层，降低脏写风险
- 可以复用现有 `material-processing` 中对路径安全和索引结构的理解
- 为后续增加批量删除、回收站或审计日志保留扩展点

**替代方案：**
- 仅在路由中直接删文件：实现快，但容易遗漏 manifest/info 清理
- 仅依赖后续整理任务重建索引：窗口期过长，无法保证浏览和 AI 检索正确

### 2. 读取索引时增加“失效项过滤 + 自愈触发”作为兜底

**决策：** 在 `materials-service` 和 `image-selector` 读取索引时校验目标文件是否存在；若发现失效项，当前请求直接过滤，并触发一次索引修复或标记需要修复。

**理由：**
- 能覆盖用户手工删除文件、历史脏数据、异常中断等非标准路径
- 即使删除入口遗漏调用同步服务，也能避免向上层返回坏数据
- 将错误影响限制在单次请求内，优先保证结果正确性

**替代方案：**
- 完全信任索引：性能最好，但错误结果会直接暴露给用户和 AI
- 每次都全量扫描文件系统：一致性最好，但读取成本高，且会破坏当前索引设计

### 3. 索引修复优先采用基于 manifest 的重建

**决策：** 检测到失效项后，优先复用 `rebuildMaterialIndex()`，并在重建前清理 manifest 中已失效的记录及对应 info 文件。

**理由：**
- 现有索引生成逻辑已经稳定，复用成本最低
- 相比手写增量补丁，更不容易出现遗漏字段
- 对当前项目规模来说，全量重建一次索引的复杂度可接受

**替代方案：**
- 直接对 `index.json` 做增量删改：性能略好，但需要同时维护 manifest/info 一致性，复杂度更高

### 4. 把“不可返回失效路径”作为接口级约束

**决策：** 无论是素材列表接口还是 AI 选图函数，都必须在返回前保证路径存在；当候选全部失效时返回空结果。

**理由：**
- 这是对调用方最明确的契约，便于测试
- 能防止 UI 缩略图加载失败、AI 结果引用坏路径等连锁问题

**替代方案：**
- 允许返回路径，由调用方自行判断：会把一致性责任转嫁给多个上游模块

### 5. 提供独立的“重建索引”接口和页面按钮

**决策：** 在素材管理接口中新增独立的索引修复端点，并在 `src/web/public/index.html` 的“整理素材”按钮旁增加“重建索引”按钮，点击后执行一致性检测和修复。

**理由：**
- 用户能在发现素材列表异常时主动修复，而不必等待下一次整理任务
- “整理素材”和“重建索引”语义不同，分离入口更容易理解
- 可以复用同一套修复逻辑供读取时自愈和手工触发共同使用

**替代方案：**
- 将修复逻辑混入“整理素材”按钮：会让按钮语义过重，也不利于单独排查索引问题
- 仅依赖后台自动修复：用户缺少可控入口，排障体验差

## 风险 / 权衡

- **删除后立即重建索引会增加单次删除耗时** -> 当前素材索引规模有限，优先保证正确性；若后续规模扩大，再演进为批量删除后合并重建
- **人工直接删除文件仍可能绕过主路径** -> 通过读取时过滤和自愈机制兜底
- **manifest 与 info 清理规则写错可能误删记录** -> 只按处理后相对路径和 manifest 关联关系删除，并保留详细日志
- **并发删除与整理任务可能相互影响** -> 复用现有素材处理中的串行保护思路，必要时为删除服务增加互斥控制
- **手动重建索引可能与整理任务并发执行** -> 共享修复服务和互斥控制，避免同时修改 `.materials` 元数据

## 迁移计划

1. 新增统一的处理后素材删除/修复服务，封装路径校验、记录定位和清理逻辑。
2. 新增独立的索引检测/修复服务，并提供重建索引接口。
3. 在 Web 素材管理页面的“整理素材”按钮旁接入“重建索引”按钮，展示执行结果并刷新素材列表。
4. 在 `materials-service` 与 `image-selector` 中加入失效索引过滤与自愈触发。
5. 增加针对“手工删文件后浏览”和“手工删文件后 AI 选图”的验证用例。

**回滚策略：**
- 若删除同步逻辑出现异常，可临时关闭删除入口，仅保留现有整理与浏览能力。
- 若重建索引入口引发副作用，可临时隐藏该按钮并保留后台修复逻辑。
- 若读取时自愈逻辑引发副作用，可回退到“过滤失效项但不自动修复”，仍能保证不返回坏路径。

## Open Questions

- 删除整理后素材时，是否需要同时删除对应原始素材，还是只维护处理后目录的一致性？本次设计默认仅处理整理后目录。
- 自愈触发应当同步执行还是异步执行？本次设计倾向同步过滤、同步或轻量即时修复，避免继续暴露失效结果。
