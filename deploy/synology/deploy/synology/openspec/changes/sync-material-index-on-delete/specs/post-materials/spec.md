## 新增需求

### 需求: 删除整理后素材时系统必须同步更新索引记录
系统在删除已经整理完成的素材时，必须同步更新 `.materials/index.json`、`.materials/manifest.json` 和对应的 `.materials/info/**/*.json` 记录，禁止保留指向已删除文件的失效索引项。

#### 场景: 通过系统删除整理后素材
- **当** 用户通过系统提供的删除入口删除一个已经整理完成的素材文件
- **那么** 系统必须删除对应的处理后素材文件
- **那么** 系统必须移除该素材对应的索引项、manifest 记录和 info 记录
- **那么** 后续素材列表和目录统计中禁止继续返回该素材

#### 场景: 删除不存在的整理后素材
- **当** 用户请求删除一个不存在的整理后素材文件
- **那么** 系统必须返回明确的失败结果或幂等结果
- **那么** 系统禁止写入新的脏索引记录

### 需求: 素材浏览与 AI 检索必须忽略失效索引项
当索引文件中仍存在指向已删除文件的历史记录时，素材浏览和 AI 检索必须忽略这些失效索引项，并必须触发刷新或自愈流程，确保后续结果与文件系统一致。

#### 场景: 浏览素材时遇到失效索引项
- **当** 系统读取素材索引并发现某个索引项对应的处理后文件已经不存在
- **那么** 系统必须禁止将该索引项返回给素材浏览接口或 Web 界面
- **那么** 系统必须记录索引失效的日志
- **那么** 系统必须在本次或下一次刷新中清理该失效索引项

#### 场景: AI 选图时遇到失效索引项
- **当** AI 选图或素材检索流程读取索引并命中某个已经不存在的素材文件
- **那么** 系统必须跳过该索引项并继续匹配其他可用素材
- **那么** 系统禁止返回不存在的文件路径
- **那么** 若没有其他可用素材，系统必须返回空结果而不是失效路径

### 需求: Web 页面必须提供手动重建索引入口
系统必须在素材管理页面的“整理素材”按钮旁提供“重建索引”按钮，用于主动检测处理后素材文件、索引、manifest 与 info 记录之间的不一致，并执行索引修复。

#### 场景: 用户手动触发重建索引
- **当** 用户点击素材页面中“整理素材”旁边的“重建索引”按钮
- **那么** 系统必须启动一次索引检测与修复流程
- **那么** 系统必须检查处理后素材文件、`.materials/index.json`、`.materials/manifest.json` 和 `.materials/info/**/*.json` 的一致性
- **那么** 系统必须修复发现的失效索引项或缺失关联记录

#### 场景: 重建索引完成后刷新界面
- **当** 重建索引流程执行完成
- **那么** 系统必须向页面返回成功或失败结果
- **那么** 页面必须刷新当前素材列表
- **那么** 页面必须向用户展示重建结果摘要
