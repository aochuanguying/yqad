## 上下文

当前项目从零开始，尚未使用任何数据库存储。关键业务数据（主题可用次数、敏感词库、API Token、任务缓存）如果仅使用内存 Map 或 JSON 文件存储，存在以下问题：
1. **数据易失性**：应用重启导致内存数据丢失
2. **文件锁竞争**：JSON 文件读写存在并发锁问题
3. **性能瓶颈**：文件 IO 性能低于内存数据库
4. **环境隔离缺失**：测试和生产环境共用同一份文件，容易数据混乱

项目已有 Redis 容器部署在 `bridge` 网络中，但应用尚未使用 Redis 进行数据存储。

**约束条件**：
- 项目从零开始，没有使用任何数据库（MySQL、ChromaDB 都是后续步骤）
- 测试环境使用宿主机 IP `192.168.50.50` 通信
- 生产环境使用容器名通信（需加入 bridge 网络）
- 测试和生产使用不同的 Redis 数据库编号防止数据混乱
- 必须配置持久化确保重启后数据不丢失

## 目标 / 非目标

**目标：**
1. 从零实现 Redis 存储系统（主题可用次数、敏感词库、API Token、任务缓存）
2. 实现测试/生产环境隔离的配置管理
3. 配置 Redis 持久化（RDB + AOF）确保重启后数据不丢失
4. 保持向后兼容，不影响现有业务逻辑
5. 提供降级方案（Redis 不可用时回退到内存/文件存储）

**非目标：**
1. 使用 MySQL 存储（第二步才引入）
2. 使用 ChromaDB 向量数据库（第三步才引入）
3. 迁移现有数据（从零开始，没有历史数据）
4. 修改业务逻辑（仅变更存储层）
5. 性能优化（保持现有性能或更好）

## 决策

### 1. Redis 数据结构选型

**决策**：根据数据类型选择最优 Redis 数据结构

| 数据类型 | 原结构 | Redis 结构 | 理由 |
|---------|-------|-----------|------|
| 主题可用次数 | `Map<string, number>` | **String** 或 **Hash** | 简单整数计数，String 足够；如需扩展元数据可用 Hash |
| 敏感词库 | `Set<string>` | **Set** | 天然去重，O(1) 查询复杂度，支持批量操作 |
| API Token | `string` | **String** | 简单键值对，支持加密存储 |
| 任务缓存 | `Map<string, AsyncTask>` | **Hash** | 结构化存储，支持字段级操作，带 TTL 自动过期 |

**考虑过的替代方案**：
- 主题可用次数使用 Sorted Set：过度设计，不需要排序功能
- 敏感词库使用 String + JSON 序列化：查询效率低，无法原子操作
- 任务缓存使用 String + JSON：无法字段级更新，序列化开销大

**选择理由**：选择最简单且满足需求的数据结构，降低复杂度。

### 2. 环境配置管理

**决策**：使用环境变量区分测试和生产环境

```yaml
# config/default.yaml
redis:
  # 测试环境配置（IP 通信）
  test:
    host: 192.168.50.50
    port: 6379
    db: 0
    keyPrefix: "test:"
  
  # 生产环境配置（容器内通信）
  production:
    host: redis
    port: 6379
    db: 1
    keyPrefix: "prod:"
```

**环境变量**：
- `NODE_ENV`: 区分环境（test / production）
- `REDIS_HOST`: 覆盖默认主机（可选）
- `REDIS_PORT`: 覆盖默认端口（可选）
- `REDIS_DB`: 覆盖默认数据库编号（可选）

**考虑过的替代方案**：
- 使用不同配置文件：维护成本高，容易混淆
- 使用配置中心：过度设计，增加复杂度
- 硬编码在代码中：缺乏灵活性

**选择理由**：配置文件 + 环境变量覆盖提供最佳灵活性，符合 12-Factor App 原则。

### 3. Redis 持久化策略

**决策**：同时启用 RDB 和 AOF 持久化

```yaml
# Redis 配置
rdb:
  enabled: true
  saveRules:
    - "900 1"    # 900 秒内至少 1 个键变化
    - "300 10"   # 300 秒内至少 10 个键变化
    - "60 10000" # 60 秒内至少 10000 个键变化
  filename: "dump.rdb"

aof:
  enabled: true
  fsync: "everysec"  # 每秒同步一次，性能和安全性平衡
  filename: "appendonly.aof"
```

**考虑过的替代方案**：
- 仅 RDB：可能丢失最后一次快照后的数据
- 仅 AOF：重启恢复慢，文件体积大
- 无持久化：重启数据丢失，不满足需求

**选择理由**：RDB 提供快速恢复，AOF 提供数据完整性，两者结合最佳。

### 4. 键命名空间设计

**决策**：使用分层命名空间防止键冲突

```
{env}:{module}:{data}
```

**示例**：
- `test:topic:uses:{topicId}` - 测试环境主题可用次数
- `prod:sensitive:words` - 生产环境敏感词库
- `test:api:token` - 测试环境 API Token
- `prod:task:{task_id}` - 生产环境任务缓存

**考虑过的替代方案**：
- 不使用前缀：测试和生产数据容易混淆
- 仅使用数据库编号：缺乏语义，调试困难
- 使用 Redis Hash 嵌套：增加操作复杂度

**选择理由**：前缀提供清晰语义，配合数据库编号双重隔离。

### 5. 降级策略

**决策**：实现 Redis 不可用时的优雅降级

```typescript
// 伪代码示例
class TopicUsesStorage {
  private useRedis: boolean = true;
  
  async getUses(topicId: string): Promise<number> {
    try {
      if (this.useRedis) {
        return await redis.get(...);
      }
    } catch (error) {
      logger.warn('Redis 不可用，降级到内存存储', error);
      this.useRedis = false;
    }
    // 降级到内存 Map
    return this.memoryStore.get(topicId) || 0;
  }
}
```

**考虑过的替代方案**：
- 直接抛出异常：影响业务可用性
- 仅使用 Redis：单点故障风险
- 自动重试：可能加重 Redis 负担

**选择理由**：降级到内存存储保证业务连续性，同时记录告警日志。

### 6. 初始化策略

**决策**：启动时初始化 Redis 连接和存储结构

```typescript
// 启动流程
1. 加载环境配置（NODE_ENV、REDIS_HOST 等）
2. 创建 Redis 连接管理器
3. 测试 Redis 连接
4. 初始化各存储模块（topic-uses, sensitive-words, api-token, task-cache）
5. 继续正常业务流程
```

**考虑过的替代方案**：
- 懒加载连接：首次使用时才连接，延迟问题发现
- 手动初始化：增加运维复杂度

**选择理由**：启动时初始化可以尽早发现问题，避免运行时故障。

## 风险 / 权衡

### [风险] Redis 单点故障
**风险**：Redis 宕机导致存储层不可用

**缓解措施**：
- 实现降级策略，回退到内存/文件存储
- 生产环境配置 Redis Sentinel 或 Cluster（未来）
- 健康检查 + 自动告警

### [风险] 内存溢出
**风险**：Redis 存储过多数据导致内存溢出

**缓解措施**：
- 设置 Redis maxmemory 限制（建议 512MB）
- 配置 LRU eviction 策略
- 为键设置合理的 TTL（任务缓存 30 分钟，主题可用次数 7 天）
- 定期监控内存使用率

### [风险] 持久化性能影响
**风险**：RDB/AOF 持久化影响写入性能

**缓解措施**：
- AOF 使用 `everysec` 策略（每秒同步一次）
- RDB 在子进程执行，不阻塞主进程
- 监控写入延迟，调整持久化频率

### [风险] 网络延迟
**风险**：容器间网络延迟影响性能

**缓解措施**：
- 使用 Docker bridge 网络（低延迟）
- 启用 Redis 连接池（复用连接）
- 批量操作减少网络往返（如 MGET、Pipeline）
- 监控 P99 延迟

### [风险] 数据一致性
**风险**：Redis 和备份数据不一致

**缓解措施**：
- Redis 作为主存储
- 定期备份到文件
- 定期一致性检���任务

### [权衡] 增加系统复杂度
**权衡**：引入 Redis 增加架构复杂度和运维成本

**理由**：
- 换取数据持久性和可靠性
- 换取更好的读写性能
- 换取环境隔离能力
- 复杂度在可控范围内（仅 4 个数据类型）

## 回滚策略

### 回滚触发条件
- Redis 连接成功率 < 95%
- P99 延迟 > 100ms
- 数据错误率 > 1%

### 回滚步骤
1. **快速回滚**：切换环境变量 `USE_REDIS=false`
2. **数据导出**：从 Redis 导出数据到 JSON 文件
3. **应用回滚**：部署上一个稳定版本
4. **验证**：确认业务恢复正常

### 回滚时间目标
- 检测时间：< 5 分钟
- 决策时间：< 5 分钟
- 执行时间：< 10 分钟
- **总时间**：< 20 分钟
