## 新增需求

### 需求：系统必须提供敏感词库的 Redis 存储功能

系统必须从零开始使用 Redis Set 存储敏感词库，支持快速查询、批量操作和热更新。

**约束**：
- 项目从零开始，没有历史数据需要迁移
- 只使用 Redis 存储，不涉及 MySQL 备份（第二步才引入）
- 使用 Set 数据结构（天然去重，O(1) 查询）

#### 场景：添加敏感词
- **当** 调用 `addWord(word)` 方法时
- **那么** 系统必须使用 `SADD` 命令将敏感词添加到 Set
- **并且** 必须返回添加是否成功

#### 场景：删除敏感词
- **当** 调用 `removeWord(word)` 方法时
- **那么** 系统必须使用 `SREM` 命令从 Set 中删除敏感词
- **并且** 必须返回删除是否成功

#### 场景：检查敏感词
- **当** 调用 `contains(word)` 方法时
- **那么** 系统必须使用 `SISMEMBER` 命令检查是否存在
- **并且** 必须返回布尔值结果

#### 场景：获取所有敏感词
- **当** 调用 `getAllWords()` 方法时
- **那么** 系统必须使用 `SMEMBERS` 命令获取所有敏感词
- **并且** 必须返回数组格式的结果

#### 场景：批量导入敏感词
- **当** 调用 `importWords(words[])` 方法时
- **那么** 系统必须使用 Pipeline 批量执行 `SADD` 命令
- **并且** 必须返回成功和失败的数量统计

### 需求：系统必须支持敏感词库热更新

系统必须支持在不重启应用的情况下更新敏感词库。

#### 场景：监听词库变更
- **当** 敏感词库发生变更时
- **那么** 系统必须通过 Redis Pub/Sub 接收通知
- **并且** 必须重新加载词库到内存缓存

#### 场景：发布更新通知
- **当** 调用 `reloadWordLibrary()` 方法时
- **那么** 系统必须发布 `sensitive:words:reload` 消息
- **并且** 所有实例必须收到并重新加载
