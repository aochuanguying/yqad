## 上下文

当前 AutoJS API 配置页面 (`autojs-config.html`) 仅提供基础配置功能（服务器地址、脚本名称），用户无法查看 AutoJS 服务器上有哪些脚本，也无法灵活执行其他脚本。

**当前状态**：
- AutoJS API 服务已部署并提供完整的 RESTful API（`/api/scripts`、`/api/execute`）
- 后端已有 `AutoJsApiClient` 客户端类支持调用 API
- 配置页面使用纯前端实现（HTML + 原生 JavaScript + Tailwind CSS）

**约束**：
- 保持纯前端实现，不增加后端依赖
- 使用现有的 API Token 认证机制
- 保持与现有配置页面的 UI 风格一致

## 目标 / 非目标

**目标：**
- 在配置页面展示 AutoJS 服务器上的所有脚本列表
- 支持一键执行任意脚本
- 提供执行状态反馈（执行中、成功、失败）
- 保持轻量级实现，不引入额外依赖

**非目标：**
- 不支持脚本编辑功能
- 不支持脚本上传/删除管理
- 不支持脚本执行历史记录（依赖 AutoJS 服务自身日志）

## 决策

### 1. 前端实现方案

**决策**：在现有 `autojs-config.html` 中增加脚本列表区域，使用原生 JavaScript 实现

**理由**：
- 保持与现有页面技术栈一致
- 零依赖，加载速度快
- 易于维护和部署

**替代方案**：
- 使用 Vue/React 等框架 → 过度设计，增加构建复杂度
- 创建独立的新页面 → 用户体验割裂，增加导航成本

### 2. 脚本列表获取时机

**决策**：页面加载时自动获取一次脚本列表，提供手动刷新按钮

**理由**：
- 脚本列表不会频繁变化
- 减少不必要的 API 请求
- 用户可随时手动刷新获取最新列表

### 3. 执行状态反馈

**决策**：使用按钮状态变化（禁用 + 加载动画）+ Toast 提示展示执行结果

**理由**：
- 与现有 Toast 通知机制保持一致
- 简洁直观的用户反馈
- 不需要复杂的状态管理

**替代方案**：
- 模态对话框 → 过于重量级
- 状态列表 → 实现复杂，对异步执行不友好

### 4. API 调用方式

**决策**：前端直接调用 AutoJS API，通过本服务后端代理（可选）

**理由**：
- 简单直接，减少中间层
- 利用现有 API Token 认证
- 避免 CORS 问题（同域或通过后端代理）

## 风险 / 权衡

**风险 1**：跨域问题（CORS）
- **缓解**：AutoJS API 服务已配置 CORS，或��过本服务后端代理请求

**风险 2**：脚本执行超时
- **缓解**：设置合理的超时时间（30 秒），异步执行不阻塞 UI

**风险 3**：脚本列表为空或获取失败
- **缓解**：友好的错误提示，提供重试按钮

**权衡**：不支持脚本执行进度实时查看
- 原因：需要 WebSocket 或轮询机制，增加复杂度
- 未来可通过增强 AutoJS API 服务实现
