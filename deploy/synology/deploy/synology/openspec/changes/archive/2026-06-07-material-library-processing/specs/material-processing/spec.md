## 新增需求

### 需求: 系统必须支持对原始素材执行增量梳理并生成处理后素材
系统必须以 `materials.rawPath` 作为原始素材输入目录，以 `materials.processedPath` 作为处理后素材输出目录，并且仅对新增的原始素材执行梳理。

#### 场景: 增量梳理仅处理新增素材
- **当** 系统执行素材梳理且在原始目录中发现图片文件
- **那么** 系统必须基于“原始文件指纹”（至少包含相对路径、mtime、size）判断该文件是否已处理
- **并且** 系统必须只对未处理过的原始素材执行后续梳理步骤

#### 场景: 原始素材删除不影响处理后素材
- **当** 某个原始素材已完成梳理并在处理后目录生成对应产物
- **并且** 该原始素材随后被删除或不可访问
- **那么** 系统必须禁止删除或破坏已生成的处理后素材与其梳理信息

### 需求: 系统必须对新增原始素材执行格式转换与压缩
系统必须将新增原始素材转换为适合发帖的图片格式并进行压缩，生成的图片文件必须保存到处理后目录。

#### 场景: 成功转换与压缩
- **当** 系统对新增原始素材执行梳理
- **那么** 系统必须生成一份处理后的图片文件，禁止覆盖同名但来源不同的已存在处理后文件
- **并且** 系统必须记录处理后文件的输出相对路径、输出格式与输出体积

#### 场景: 转换失败时降级
- **当** 系统对某个原始素材执行格式转换与压缩失败
- **那么** 系统必须记录告警日志
- **并且** 系统必须将该素材标记为“不可处理”或“处理失败”并写入梳理信息
- **并且** 系统必须禁止因单个文件失败导致整次梳理任务中断

### 需求: 系统必须提取并保存图片元数据
系统必须提取处理后图片的基础元数据（至少包含宽、高、格式、体积），并在可用时提取 EXIF 等扩展元数据，并以结构化格式保存到处理后目录。

#### 场景: 保存结构化元数据
- **当** 系统完成某张图片的处理后文件生成
- **那么** 系统必须保存该图片的结构化元数据
- **并且** 元数据必须可通过后续查询接口读取

### 需求: 系统必须支持图片内容识别与介绍生成（可配置）
系统必须支持对处理后图片进行内容识别并生成用于检索与发帖参考的介绍文本，并且该能力必须支持通过配置开关启用或禁用。

#### 场景: 启用识别时生成介绍
- **当** 配置启用了图片内容识别
- **并且** 系统完成某张图片的处理后文件生成
- **那么** 系统必须生成该图片的识别结果与介绍文本并写入梳理信息

#### 场景: 识别失败时降级
- **当** 内容识别或介绍生成失败
- **那么** 系统必须记录告警日志
- **并且** 系统必须保留已生成的处理后图片文件
- **并且** 系统必须在梳理信息中标记识别失败原因（可脱敏/摘要）

### 需求: 系统必须提供手工触发素材梳理的 API
系统必须提供 HTTP API 以便用户在任意时间手工触发素材梳理，并且该 API 必须执行增量梳理策略。

#### 场景: 手工触发梳理
- **当** 客户端发送 POST /api/materials/process 请求
- **那么** 系统必须启动一次素材梳理任务（增量）
- **并且** 系统必须返回本次任务的执行结果摘要（至少包含处理数量与失败数量）

### 需求: 系统必须提供梳理信息查询能力
系统必须提供 HTTP API 用于查询处理后素材对应的梳理信息，以便发帖时检索与使用。

#### 场景: 查询单个素材梳理信息
- **当** 客户端发送 GET /api/materials/info/:path 请求（path 为处理后目录的相对路径）
- **并且** 系统存在该 path 对应的梳理信息
- **那么** 系统必须返回该素材的结构化梳理信息（至少包含元数据、识别结果、介绍文本、处理时间与来源映射）

#### 场景: 梳理信息不存在
- **当** 客户端发送 GET /api/materials/info/:path 请求
- **并且** 系统不存在该 path 对应的梳理信息
- **那么** 系统必须返回 404
