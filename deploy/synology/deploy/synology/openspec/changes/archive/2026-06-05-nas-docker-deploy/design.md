## 上下文

本项目是一个在群晖 NAS 上 Docker 部署的自动化系统。当前 `docker-compose.yml` 已映射了 `config`、`data`、`logs` 三个目录，但 `data/materials` 素材库没有独立挂载点，宿主机上管理素材不够直观。同时 Web 管理界面（端口 3000）已在运行，但发帖主题配置素材时需手动输入路径字符串。

## 目标 / 非目标

**目标：**
- 为素材库提供独立的 Docker volume 映射，支持通过 `MATERIALS_PATH` 环境变量指定宿主机目录
- 实现后端文件列表 API（`GET /api/materials/files`），扫描素材库并返回可用文件
- 在 Web 管理界面提供文件选择器组件，替代手动输入素材路径
- 更新部署文档，增加 NAS 素材路径映射说明

**非目标：**
- 不实现素材文件的上传/删除功能（用户在宿主机直接操作文件系统）
- 不实现递归目录树浏览（仅列出素材根目录下直接文件，简化实现）
- 不变更现有的 AI 内容生成逻辑和调度机制

## 决策

### 1. 素材库独立 volume 挂载

**选择**: 在 `docker-compose.yml` 中新增一个独立的 volume 条目  
**理由**: 
- 与 `./data:/app/data` 的映射并存，给用户更大灵活性——NAS 上素材可能存储在 SSD volume 或共享文件夹中
- 使用 `${MATERIALS_PATH:-./data/materials}:/app/data/materials` 的形式，默认行为与现有一致

**替代方案（放弃）**:
- 仅使用 `./data` 映射覆盖全部数据——过于笼统，NAS 用户可能想将素材放在不同的 volume

### 2. 文件列表 API 设计

**选择**: `GET /api/materials/files` 返回 `{ files: [{name, path, size, mtime}] }`  
**理由**:
- 复用项目现有的 Express Web 框架（已有 `web.enabled` 及端口配置）
- 只读操作，安全风险低
- `path` 字段为相对路径（如 `images/banner.jpg`），前端选择后直接存入配置

**替代方案（放弃）**:
- WebSocket 实时推送文件变化——过度设计，素材文件变动频率低
- 返回 base64 缩略图——增加复杂度，列表页只需文件名即可

### 3. 前端文件选择器实现

**选择**: 弹窗式文件列表 + 复选框选择  
**理由**:
- 用户点击「选择素材」按钮 → 弹出模态框 → 请求 API 获取文件列表 → 勾选后确认
- 简单直观，不需要额外的树组件依赖
- 适配已有的 Web 管理界面 UI 风格

**替代方案（放弃）**:
- 类似操作系统的文件管理器（drag-and-drop）——UI 复杂度过高
- HTML5 `<input type="file">` 原生选择——这是上传场景而非服务端文件浏览

### 4. 路径存储策略

**选择**: 配置中只存储相对路径（相对于 `materials.basePath`）  
**理由**:
- 容器内路径固定为 `/app/data/materials`，无论宿主机映射到哪都不影响配置
- 迁移或更换 NAS 挂载路径时无需修改已保存的主题配置

## 风险 / 权衡

| 风险 | 缓解措施 |
|------|----------|
| NAS 用户权限问题导致容器读不了素材文件 | 文档中说明 UID/GID 需与容器内用户一致，建议 `chmod 755` 素材目录 |
| 素材目录文件量过大时 API 返回缓慢 | 限制单次返回 500 条，超过时提示用户分子目录管理 |
| 容器内 `/app/data/materials` 被两个 volume 覆盖（独立映射覆盖了 `./data` 映射的子目录） | Docker 的挂载机制天然支持此行为（子路径挂载优先），无需额外处理 |
