## 上下文

已抓包获取一汽奥迪APP的核心API（登录、签到、帖子列表、评论、发帖），需要在群晖NAS上部署自动化脚本系统。本阶段所有API以Mock方式实现，后续切换到真实API只需更换配置。

系统需要具备：定时调度能力、大模型内容生成能力、稳定的执行日志。

## 目标 / 非目标

**目标：**
- 搭建可在群晖NAS上稳定运行的自动化脚本框架
- 实现Mock API层，模拟所有已抓包接口
- 对接ChatGPT生成高质量评论与帖子内容
- 实现定时调度（签到、评论、发帖）
- 提供清晰的执行日志和状态报告

**非目标：**
- 不处理真实API的反爬/风控对抗（后续阶段）
- 不实现UI管理界面
- 不做多账号支持（当前为单账号）
- 不处理APP版本更新导致的API变动

## 决策

### 1. 技术栈选择：TypeScript + Node.js

**选择**：TypeScript/Node.js
**替代方案**：Python + requests
**理由**：
- 群晖NAS对Node.js支持良好（Docker或直接安装）
- TypeScript提供类型安全，适合管理复杂的API请求/响应结构
- npm生态丰富，有成熟的HTTP客户端（axios）和调度库（node-cron）
- 便于后续扩展为Web服务

### 2. 项目结构：模块化单仓

**选择**：单一仓库，按功能模块组织
**理由**：
- 功能间耦合度适中（共享token、配置等）
- 部署简单，适合NAS环境
- 目录结构：
  ```
  src/
    api/          # API客户端（Mock实现）
    services/     # 业务逻辑（签到、评论、发帖）
    ai/           # 大模型集成
    scheduler/    # 定时任务调度
    utils/        # 工具函数（日志、配置）
  config/         # 配置文件
  logs/           # 运行日志
  ```

### 3. Mock API实现方式

**选择**：本地Mock模块，通过接口抽象切换
**替代方案**：独立Mock Server（如json-server）
**理由**：
- 直接在代码中实现Mock，无需额外进程
- 通过接口/策略模式，后续替换为真实HTTP调用只需切换实现
- 开发调试更方便，无网络依赖

### 4. 大模型集成方案

**选择**：OpenAI API（GPT-4/3.5），通过配置支持其他兼容API
**理由**：
- OpenAI API成熟稳定
- 通过baseUrl配置可兼容其他OpenAI协议兼容的大模型服务
- 使用提示词工程生成符合社区风格的内容

### 5. 调度方案

**选择**：node-cron（进程内调度）
**替代方案**：系统crontab
**理由**：
- 进程内调度便于管理任务状态和依赖关系
- 支持更灵活的调度策略（随机延时、避免固定模式）
- 日志统一管理

### 6. 内容生成策略

**选择**：先分析后生成的两阶段策略
- 阶段一：获取最新帖子和评论，分析热点话题、语言风格、互动模式
- 阶段二：基于分析结果，通过大模型生成贴合社区氛围的内容
**理由**：
- 生成内容更自然、更贴近社区风格
- 避免生成与社区内容脱节的通用文本

## 风险 / 权衡

| 风险 | 缓解措施 |
|------|----------|
| 大模型API调用成本 | 控制每日调用次数，使用较低cost的模型(gpt-3.5-turbo)做日常评论 |
| 生成内容质量不稳定 | 设计Prompt模板，加入风格约束和质量检查 |
| NAS性能有限 | 脚本轻量化，避免常驻高CPU任务 |
| Mock与真实API行为差异 | Mock严格按抓包数据建模，记录请求/响应格式 |
| 定时任务执行失败 | 实现重试机制，失败通知（日志 + 可选推送） |
| 行为模式过于规律被检测 | 引入随机延时和执行时间波动 |
