## 上下文

当前系统架构：
- **前端**: 纯 HTML + Tailwind CSS + Vanilla JavaScript，通过 fetch API 与后端通信
- **后端**: Express.js 服务器，提供 RESTful API
- **服务层**: `AutoCommentService` 封装了评论业务逻辑，`performDailyComments()` 方法执行评论任务
- **调度器**: 基于 Cron 的定时任务调度，在 `src/index.ts` 中初始化

现状问题：
- 评论功能仅能通过定时调度器自动执行
- Web 界面只能配置评论参数，无法手动触发
- 没有即时反馈机制，用户无法实时查看执行结果

约束：
- 保持现有 API 鉴权机制（Session + API Token）
- 复用现有的 `AutoCommentService.performDailyComments()` 方法
- 前端保持 Vanilla JS 风格，不引入框架

## 目标 / 非目标

**目标：**
1. 在 Web 界面评论配置页添加"立即评论"按钮
2. 提供执行状态反馈（执行中、成功、失败）
3. 显示执行结果摘要（成功数、失败数）
4. 后端 API 支持手动触发评论任务
5. 保持与定时任务相同的执行逻辑

**非目标：**
1. 不修改定时任务的执行逻辑
2. 不支持并发执行（手动触发时定时任务应暂停或跳过）
3. 不提供评论历史查看功能（属于独立功能）
4. 不修改现有的评论配置表单结构

## 决策

### 1. API 设计

**决策**: 新增 `POST /api/comment/execute` 端点

**理由**:
- RESTful 风格，`execute` 动词明确表示执行操作
- 使用 POST 而非 GET，因为这是状态变更操作
- 复用现有的 `AutoCommentService`，不在 API 层重复业务逻辑

**替代方案**:
- `POST /api/tasks/comment` - 更符合任务模式，但需要统一的任务管理框架
- `PUT /api/comment/run` - PUT 通常用于资源更新，不如 POST 合适

### 2. 异步 vs 同步执行

**决策**: 采用异步执行 + 轮询查询模式

**理由**:
- 评论任务可能耗时较长（多条评论，每条之间有延时）
- HTTP 请求有超时限制，不适合长连接
- 可复用现有的任务查询模式（参考 `posts-routes.ts` 中的异步任务实现）

**实现方案**:
1. POST `/api/comment/execute` 立即返回任务 ID
2. 后台异步执行 `performDailyComments()`
3. GET `/api/comment/tasks/:id` 查询任务状态
4. 前端轮询查询直到任务完成

**替代方案**:
- 同步执行：简单但用户体验差，可能超时
- WebSocket 推送：复杂度高，需要额外基础设施

### 3. 前端 UI 位置

**决策**: 在评论配置页底部，"保存配置"按钮左侧添加"立即评论"按钮

**理由**:
- 用户明确指定的位置
- 符合操作逻辑：配置完成后立即执行
- 与"保存配置"按钮并列，视觉层次清晰

**UI 组件**:
- 按钮：蓝色主题，带图标（⚡ 立即评论）
- 状态显示：按钮右侧或上方显示执行状态
- 结果提示：Toast 通知 + 模态框显示详细结果

### 4. 并发控制

**决策**: 简单锁定机制，同一时间只允许一个评论任务执行

**理由**:
- 避免重复评论同一帖子
- 防止资源竞争和 API 限流
- 实现简单，维护成本低

**实现方案**:
- 内存标志位：`isCommentTaskRunning`
- API 检查标志位，如正在运行则返回 409 Conflict
- 任务完成后清除标志位

### 5. 错误处理

**决策**: 统一错误格式，前端根据错误类型显示不同提示

**错误类型**:
- 409 Conflict: 已有任务正在运行
- 401 Unauthorized: 未登录或 Token 失效
- 500 Internal Error: 执行异常
- 200 但 success=false: 业务逻辑失败（如 API 调用失败）

## 风险 / 权衡

### 风险 1: 并发执��冲突

**风险**: 用户手动触发时，定时任务也可能同时执行，导致重复评论

**缓解措施**:
- 使用同一锁定标志，手动任务和定时任务共享
- 定时任务检测到锁定时跳过执行
- 在日志中记录跳过原因

### 风险 2: 长时间任务阻塞

**风险**: 评论任务耗时过长，用户失去耐心或频繁刷新

**缓解措施**:
- 前端显示进度提示（已执行/总数）
- 支持后台执行，用户可离开页面
- 任务状态持久化，刷新页面后可继续查询

### 风险 3: API 限流

**风险**: 频繁手动触发导致 API 限流

**缓解措施**:
- 前端按钮防抖（执行中禁用按钮）
- 后端速率限制（参考现有 rateLimitMiddleware）
- 日志记录异常频率

### 权衡：任务持久化

**当前方案**: 内存存储任务状态（简单但不持久）

**权衡**:
- 优点：实现简单，性能好
- 缺点：服务重启后任务状态丢失

**未来改进**: 如需生产级可靠性，可引入 Redis 或数据库存储任务状态

## 迁移计划

不适用 - 此为新增功能，不涉及数据迁移或破坏性变更

## Open Questions

1. **是否需要支持取消任务？**
   - 当前方案：不支持（评论任务一旦开始应执行完成）
   - 未来可扩展：添加 `DELETE /api/comment/tasks/:id` 取消端点

2. **是否需要任务历史记录？**
   - 当前方案：仅显示当前任务结果
   - 未来可扩展：持久化任务历史，支持查询统计

3. **是否需要批量操作？**
   - 当前方案：单次执行全部评论任务
   - 未来可扩展：支持选择特定帖子评论
