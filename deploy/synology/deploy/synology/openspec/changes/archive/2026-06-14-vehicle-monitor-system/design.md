## 上下文

**当前状态：**
- 已有完善的 iOS Scriptable 小组件代码（`aujie-widget.js`），实现了奥捷智行车辆的登录、Token 管理、车辆信息获取、OBD 车况获取等功能
- 已有 Home Assistant 设备位置 API 文档，可通过 REST API 获取手机设备的位置信息
- 缺少自动化的车辆监控系统，无法实现 24 小时不间断的异常状态检测和报警

**约束条件：**
- 复用现有 `aujie-widget.js` 中的核心逻辑（登录、Token 管理、车辆信息获取等）
- 复用 Home Assistant API 文档中的设备位置获取方式
- 需要支持 Node.js 环境运行（现有脚本为 iOS Scriptable 环境）
- Token 和配置信息需要安全存储

**利益相关者：**
- 车主：需要及时的车辆异常报警
- 系统维护者：需要简单可靠的监控方案

## 目标 / 非目标

**目标：**
1. 实现车辆登录认证和 Token 自动管理（登录、刷新、持久化）
2. 实现定时监控机制，按可配置间隔自动获取车辆状态
3. 实现异常状态判定（车辆移动、门窗未关、设防状态等）
4. 实现手机位置获取（通过 Home Assistant API）
5. 实现安全距离计算和报警判定
6. 提供电话报警接口（预留实现）

**非目标：**
1. 不修改现有 iOS 小组件功能
2. 不提供图形化管理界面（后续可扩展）
3. 不支持多车辆监控（首期仅支持单车）
4. 不实现复杂的历史轨迹分析（仅实时监控）

## 决策

### 1. 运行环境选择

**决策：** 集成到现有 `audi-app-auto-tasks` 项目中

**理由：**
- 当前项目已有完善的调度器框架（`src/scheduler/index.ts`）
- 复用现有的日志系统（Winston）、配置系统（YAML）、重试工具
- 已有 Web API 框架（Express），可快速添加监控 API
- 已有 Token 存储机制（`data/token.json`），可直接复用
- 避免维护多个独立项目，降低运维成本

**集成方式：**
- 新增服务模块：`src/services/vehicle-monitor-service.ts`
- 在调度器中注册车辆监控定时任务
- 在配置文件中添加 `vehicleMonitor` 配置块
- 复用现有的 `node-cron` 调度框架

**考虑过的替代方案：**
- **独立 Node.js 项目：** 需要重复建设日志、配置、调度等基础设施
- **定时脚本（cron）：** 状态管理复杂，错误恢复能力弱
- **iOS 快捷指令自动化：** 受限于 iOS 后台能力，无法实现 24 小时监控

### 2. Token 存储方案

**决策：** 复用现有 Token 存储机制（`data/vehicle-token.json`）

**理由：**
- 与现有 `data/token.json` 保持一致的存储方式
- 本地 JSON 文件，便于管理和备份
- 配置文件存储非敏感配置（监控间隔、安全距离、设备 ID 等）
- 密码进行 MD5 加密存储（复用现有 MD5 工具函数）

**文件结构：**
```
data/
  ├── token.json              # 奥迪 APP Token（现有）
  ├── vehicle-token.json      # 奥捷智行 Token（新增）
  └── vehicle-config.json     # 车辆监控配置（新增，可选）
```

**考虑过的替代方案：**
- **Keychain（macOS）：** 安全但依赖 macOS 特定 API，降低可移植性
- **数据库存储：** 过度设计，增加依赖和复杂度
- **纯环境变量：** 配置变更需要重启服务，不够灵活

### 3. 异常判定逻辑

**决策：** 复用并扩展 `aujie-widget.js` 中的异常判定逻辑

**异常类型：**
1. **车辆移动：** 比较当前位置与上次位置的经纬度差异，超过阈值（如 50 米）判定为移动
2. **门窗未关：** 复用 `obd.anyDoorOpen` 和 `obd.anyWindowOpen` 判定
3. **设防状态异常：** `obd.isDefence === false` 且车辆处于熄火状态
4. **电压异常：** 电池电压低于阈值（如 11.5V）
5. **离线状态：** `carInfo.isOnline === false` 或 `obd.isOnline === false`

**判定策略：**
- 每次监控周期进行全量异常检查
- 任一异常触发即进入报警流程
- 记录异常发生时间和类型，便于追溯

### 4. 监控间隔设计

**决策：** 可配置的固定间隔监控

**配置项：**
- `monitorIntervalMinutes`: 监控间隔（默认 15 分钟）
- `quickMonitorIntervalMinutes`: 异常状态下的快速监控间隔（默认 5 分钟）

**理由：**
- 固定间隔实现简单，资源消耗可控
- 异常状态下自动缩短间���，提高报警及时性
- 用户可根据实际需求调整间隔

**考虑过的替代方案：**
- **动态间隔：** 根据时间段、车辆状态动态调整，实现复杂
- **事件驱动：** 依赖车辆主动上报，需要后端支持

### 5. 安全距离计算

**决策：** 使用 Haversine 公式计算两点间球面距离

**公式：** 复用 Home Assistant API 文档中提供的 `calculate_distance` 函数

**判定逻辑：**
```
if (车辆状态异常) {
  获取手机位置
  计算车辆与手机的距离
  if (距离 > 安全距离阈值) {
    触发电话报警
  }
}
```

**配置项：**
- `safeDistanceMeters`: 安全距离阈值（默认 100 米）
- `alertPhone`: 报警电话号码

### 6. 电话报警接口

**决策：** 提供抽象接口，预留实现

**接口设计：**
```javascript
async function makePhoneCall(phoneNumber, reason) {
  // 预留实现
  // 可选方案：Twilio API、阿里云语音服务、本地 SIP 等
  console.log(`[ALERT] 准备拨打 ${phoneNumber}, 原因：${reason}`)
}
```

**理由：**
- 电话报警服务依赖外部 provider，需用户自行配置
- 抽象接口便于后续接入不同服务商
- 首期可先用日志/通知替代，后续再接入电话服务

**推荐实现方案：**
- **Twilio API:** 国际服务，稳定可靠
- **阿里云语音服务：** 国内服务，资费合理
- **Home Assistant 集成：** 如已有 HA 电话服务，可直接调用

### 7. 日志和监控

**决策：** 文件日志 + 控制台输出

**日志内容：**
- 每次监控的时间、车辆状态、位置信息
- 异常判定结果和报警触发记录
- Token 刷新和错误信息

**日志级别：**
- `INFO`: 正常监控记录
- `WARN`: 可恢复错误（如网络超时）
- `ERROR`: 严重错误（如登录失败、API 异常）

**日志轮转：**
- 单文件最大 10MB
- 保留最近 7 天的日志

## 风险 / 权衡

### 风险 1: Token 过期导致监控中断

**风险：** Token 可能过期，导致 API 请求失败，监控中断

**缓解措施：**
- 实现自动重登机制（复用 `authedRequest` 逻辑）
- 登录失败时发送告警通知
- 支持 Token 手动刷新接口

### 风险 2: 位置精度问题

**风险：** GPS 定位存在误差，可能导致误判

**缓解措施：**
- 设置合理的移动阈值（如 50 米）
- 连续多次检测确认后再报警
- 结合车辆状态（引擎、设防）综合判断

### 风险 3: 网络不稳定

**风险：** 网络波动导致 API 请求失败

**缓解措施：**
- 实现重试机制（最多 3 次，指数退避）
- 区分网络错误和业务错误
- 记录失败日志，便于排查

### 风险 4: 误报扰民

**风险：** 异常判定过于敏感，导致频繁误报

**缓解措施：**
- 提供可配置的判定阈值
- 支持报警静默时段（如深夜）
- 首次报警后设置冷却时间（如 30 分钟内不重复报警）

### 风险 5: 隐私和安全

**风险：** 存储手机号、密码等敏感信息

**缓解措施：**
- 配置文件设置合适的文件权限（600）
- 日志中脱敏敏感信息
- 建议使用专用 API 账号，限制权限
