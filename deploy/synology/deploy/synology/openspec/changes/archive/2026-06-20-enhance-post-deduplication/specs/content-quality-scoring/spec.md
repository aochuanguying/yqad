## 新增需求

### 需求：系统必须对帖子内容进行多维度质量评分

系统必须基于完整性、原创性、多样性和吸引力四个维度对帖子内容进行综合质量评分。

#### 场景：质量评分合格
- **当** 待发布内容的综合质量评分达到或超过配置阈值（默认 60 分）
- **那么** 系统必须判定内容通过质量检查，允许继续发布流程
- **并且** 必须在合规性检查报告中记录各维度得分和总分

#### 场景：质量评分不合格
- **当** 待发布内容的综合质量评分低于配置阈值
- **那么** 系统必须拒绝发布该内容
- **并且** 返回错误信息必须包含总分和主要失分项

#### 场景：完整性评分
- **当** 系统评估内容完整性时
- **那么** 系统必须检查标题长度（10-50 字）、正文长度（100-2000 字）、图片数量（1-9 张）
- **并且** 每个指标达标得 33 分，总分 100 分，权重占综合评分的 30%

#### 场景：原创性评分
- **当** 系统评估内容原创性时
- **那么** 系统必须计算内容与历史发帖的最高相似度
- **并且** 原创性得分 = (1 - 最高相似度) × 100，权重占综合评分的 30%
- **并且** 相似度越低，原创性得分越高

#### 场景：多样性评分
- **当** 系统评估内容多样性时
- **那么** 系统必须检查词汇丰富度（不同词汇数/总词汇数）、句式变化（陈述句/疑问句/感叹句比例）、表情符号使用（0-5 个）
- **并且** 每个指标达标得 33 分，总分 100 分，权重占综合评分的 20%

#### 场景：吸引力评分
- **当** 系统评估内容吸引力时
- **那么** 系统必须检查热点关键词（包含当前热点话题关键词）、情感积极性（正面情感词汇占比）、互动引导词（包含"欢迎"、"讨论"、"分享"等词汇）
- **并且** 每个指标达标得 33 分，总分 100 分，权重占综合评分的 20%

#### 场景：可配置的评分权重
- **当** 配置文件中设置了 `contentQualityScoring.weights` 参数
- **那么** 系统必须使用配置的权重计算综合评分
- **并且** 权重配置必须包含四个维度的权重值，总和为 100%

#### 场景：可配置的质量阈值
- **当** 配置文件中设置了 `contentQualityScoring.minScore` 参数
- **那么** 系统必须使用配置的阈值作为质量合格标准（默认 60 分，范围 0-100）

### 需求：系统必须提供质量评分详情

系统必须在合规性检查报告中提供详细的质量评分信息，便于问题诊断和内容优化。

#### 场景：返回评分详情
- **当** 系统完成质量评分
- **那么** 系统必须返回每个维度的得分、权重和加权得分
- **并且** 必须提供每个维度的评估详情（如：标题长度 25 字 ✓、正文长度 350 字 ✓、图片数量 3 张 ✓）

#### 场景：提供优化建议
- **当** 内容质量评分低于阈值
- **那么** 系统必须提供针对性的优化建议
- **并且** 建议必须具体可操作（如："建议增加正文长度至 100 字以上"、"建议添加 1-2 张图片"）

### 需求：系统必须提供质量评分 API

系统必须提供 HTTP API 供外部调用质量评分功能。

#### 场景：调用质量评分 API
- **当** 客户端发送 POST 请求到 `/api/compliance/quality-score`
- **并且** 请求体包含 `title`（标题）、`content`（正文）、`images`（图片数组）
- **那么** 系统必须返回质量评分结果，包含各维度得分、总分和详情

#### 场景：批量质量评分
- **当** 客户端发送 POST 请求到 `/api/compliance/quality-score/batch`
- **并且** 请求体包含多个帖子内容的数组
- **那么** 系统必须返回每个帖子的质量评分结果
- **并且** 响应必须包含统计信息（平均分、最高分、最低分、合格率）
