## 上下文

当前发帖系统缺少内容去重和合规性检查机制，导致：
1. 重复内容触发社区敏感提示，影响账号健康度
2. 缺乏敏感词过滤，可能发布违规内容
3. 内容质量参差不齐，影响传播效果
4. 同一主题发帖过于频繁，容易被识别为机器行为

系统已有能力：
- `auto-post`: 自动发帖服务，支持定时任务和手工触发
- `remote-post-api`: 远程发帖 API，提供 `/api/posts/generate`、`/api/posts/execute` 等端点
- `post-logging`: 发帖日志记录，记录每次发帖的详细信息
- `topic-association`: 主题管理，支持预配置主题和均衡选择

约束条件：
- 需要在现有发帖流程中无缝集成合规性检查
- 相似度检测不能显著增加发帖延迟（目标 < 2 秒）
- 需要向后兼容，不影响现有功能

## 目标 / 非目标

**目标：**
- 实现内容相似度检测，与最近 7-14 天历史发帖比对，相似度阈值可配置
- 实现敏感词过滤和自动替换，支持敏感词库动态管理
- 实现内容质量评分机制，多维度评估内容质量
- 实现主题发帖间隔控制，防止同一主题过于频繁发布
- 生成合规性检查报告，记录检查结果并支持查询
- 在发帖 API 中集成合规性检查前置流程

**非目标：**
- 本阶段不包含主题多样化策略优化（后续迭代）
- 本阶段不包含素材混合策略优化（后续迭代）
- 不修改现有发帖日志结构（仅新增合规性检查报告）
- 不提供 Web UI 管理界面（仅 API 和数据层）

## 决策

### 1. 相似度检测算法选择

**决策：使用 TF-IDF + 余弦相似度**

**理由：**
- TF-IDF 能够捕捉文本的关键词特征，适合短文本相似度计算
- 余弦相似度计算简单高效，适合实时检测场景
- 相比深度学习模型（如 BERT），计算开销小，延迟低
- 可解释性强，便于调试和优化

**替代方案：**
- **MinHash + LSH**: 适合大规模文档去重，但对短文本效果不佳，且实现复杂
- **Sentence-BERT**: 语义理解更好，但计算开销大（需要 GPU 加速），延迟高（>5 秒）
- **SimHash**: 适合网页去重，但对内容改写不敏感

**实现细节：**
- 使用 `natural` 或 `node-nlp` 库进行 TF-IDF 向量化
- 历史发帖向量建立索引缓存，避免重复计算
- 相似度阈值默认 0.75（可配置），超过阈值判定为重复

### 2. 敏感词库管理策略

**决策：基于 Trie 树的多模式匹配算法（AC 自动机）**

**理由：**
- AC 自动机支持多模式匹配，一次扫描即可检测所有敏感词
- Trie 树结构支持高效的前缀匹配，适合中文敏感词检测
- 支持敏感词分级（如：禁止级、替换级、警告级）
- 支持动态加载和热更新，无需重启服务

**替代方案：**
- **正则表达式**: 实现简单，但性能差，难以维护大量敏感词
- **DFA 算法**: 性能接近 AC 自动机，但不支持多模式并行匹配
- **第三方 API**: 如阿里云内容安全，但增加外部依赖和成本

**实现细节：**
- 敏感词库存储在 `data/sensitive-words.json`，支持分级分类
- 替换策略：使用同义词替换或 `*` 号屏蔽
- 提供敏感词管理 API（增删改查）

### 3. 内容质量评分模型

**决策：多维度加权评分模型**

**评分维度：**
- **完整性（30%）**: 标题长度、正文长度、图片数量是否达标
- **原创性（30%）**: 与历史内容的相似度（越低分越高）
- **多样性（20%）**: 词汇丰富度、句式变化、表情符号使用
- **吸引力（20%）**: 包含热点关键词、情感积极性、互动引导词

**理由：**
- 加权模型简单透明，便于调整和优化
- 各维度独立计算，易于调试和改进
- 支持动态调整权重，适应不同策略需求

**替代方案：**
- **机器学习模型**: 需要标注数据和训练，维护成本高
- **规则引擎**: 灵活性差，难以综合评估

**实现细节：**
- 每个维度得分 0-100，加权计算总分
- 默认合格线 60 分（可配置）
- 提供评分详情，便于问题诊断

### 4. 发帖间隔控制策略

**决策：基于主题的最后发帖时间戳检查**

**理由：**
- 简单直接，易于实现和验证
- 支持按主题独立控制，灵活性高
- 可配置间隔天数（默认 3-7 天）

**实现细节：**
- 在主题数据结构中增加 `lastPostDate` 字段
- 发帖前检查：`currentDate - lastPostDate < minInterval` 则拒绝
- 支持紧急发帖白名单（特殊情况下可绕过）

### 5. 合规性检查报告存储

**决策：独立存储为 JSON 文件，与发帖日志关联**

**理由：**
- 不修改现有日志结构，降低耦合
- JSON 格式灵活，便于扩展和查询
- 支持按帖子 ID 索引和回溯

**实现细节：**
- 存储路径：`data/compliance-reports/:postId.json`
- 报告结构包含：检查时间、各项得分、是否通过、详细信息
- 提供 API 查询：`GET /api/compliance/reports/:postId`

### 6. 发帖流程集成点

**决策：在内容生成后、实际发布前执行合规性检查**

**流程：**
```
1. 接收发帖请求（生成或执行）
2. 生成帖子内容（标题、正文、图片）
3. 【新增】执行合规性检查
   - 相似度检测
   - 敏感词过滤
   - 质量评分
   - 间隔控制
4. 【新增】生成合规性检查报告
5. 检查通过 → 发布帖子
6. 检查不通过 → 返回错误，记录日志
```

**理由：**
- 在内容确定后检查，避免无效计算
- 在发布前拦截，防止违规内容流出
- 对现有流程侵入最小

## 风险 / 权衡

### 风险 1: 相似度检测性能问题

**风险：** TF-IDF 向量化和余弦相似度计算可能增加发帖延迟，尤其在历史数据量大时

**缓解措施：**
- 限制比对范围为最近 7-14 天（可配置），减少数据量
- 建立向量索引缓存，避免重复计算
- 使用异步检测，不阻塞主流程（但会增加总耗时）
- 设置超时机制（>3 秒跳过检测，记录警告）

### 风险 2: 敏感词误判和漏判

**风险：** 敏感词库不完善可能导致误判（正常内容被拦截）或漏判（违规内容通过）

**缓解措施：**
- 敏感词分级处理：禁止级（直接拦截）、替换级（自动替换）、警告级（仅记录）
- 提供白名单机制，允许特定词汇豁免
- 定期更新敏感词库，基于实际案例优化
- 记录误判案例，持续改进词库质量

### 风险 3: 质量评分模型不准确

**风险：** 简单的加权模型可能无法准确评估内容质量，导致优质内容被误杀

**缓解措施：**
- 初始阶段设置较宽松的阈值（如 50 分），观察效果后逐步调整
- 提供评分详情和诊断信息，便于人工复核
- 支持手动豁免，允许管理员强制发布
- 收集发布后的互动数据（点赞、评论），反向优化评分模型

### 风险 4: 历史数据存储和管理

**风险：** 需要存储大量历史发帖内容用于相似度比对，增加存储压力

**缓解措施：**
- 仅保留最近 30-90 天的发帖内容（可配置）
- 定期清理过期数据，使用 cron 任务自动执行
- 压缩存储文本内容，减少空间占用
- 考虑使用轻量级数据库（如 SQLite）而非 JSON 文件

### 风险 5: 发帖延迟增加

**风险：** 合规性检查全流程可能增加 2-5 秒延迟，影响用户体验

**缓解措施：**
- 优化算��性能，目标单次检查 < 500ms
- 并行执行各项检查（相似度、敏感词、质量评分可并行）
- 提供快速路径：低优先级帖子可跳过部分检查
- 异步执行非关键检查（如报告生成）

## 迁移计划

### 阶段 1: 数据准备（第 1 周）
- 创建敏感词库文件 `data/sensitive-words.json`
- 迁移历史发帖数据到结构化存储（支持相似度检索）
- 更新主题数据结构，增加 `lastPostDate` 字段

### 阶段 2: 核心功能开发（第 2-3 周）
- 实现内容去重服务（TF-IDF + 余弦相似度）
- 实现敏感词过滤服务（AC 自动机）
- 实现内容质量评分服务
- 实现发帖间隔控制服务
- 实现合规性检查报告服务

### 阶段 3: 集成与测试（第 4 周）
- 在发帖 API 中集成合规性检查流程
- 编写单元测试和集成测试
- 性能测试和优化
- 配置管理和文档编写

### 阶段 4: 灰度发布（第 5 周）
- 先在小范围环境（开发/测试）部署
- 监控关键指标：发帖延迟、拦截率、误判率
- 根据反馈调整参数（阈值、权重等）
- 全量发布

### 回滚策略
- 支持配置开关，可快速关闭合规性检查
- 保留旧版本代码，必要时回退到上一版本
- 数据向后兼容，回滚不影响现有功能

## 开放问题

1. **相似度阈值的最佳实践值？** 
   - 需要基于实际数据测试，建议初始值 0.75，逐步调整

2. **敏感词库的来源和维护责任？**
   - 需要运营团队提供初始词库，并建立定期更新机制

3. **质量评分权重如何设定？**
   - 需要 A/B 测试不同权重组合的效果，基于数据优化

4. **是否需要支持多语言敏感词检测？**
   - 当前仅支持中文，后续可扩展英文和其他语言

5. **合规性检查报告是否需要 Web UI 展示？**
   - 当前仅提供 API，后续可根据需求增加 UI 界面
