## 上下文

当前 `topic-matcher.ts` 中的 `fetchHotTopics` 函数只获取 1 页（20 个）热门话题，候选池较小。用户要求扩大到 5 页（100 个）话题，以提升匹配质量，同时保持最终关联的话题数量在 1-5 个。

## 目标 / 非目标

**目标：**
- 扩大热门话题候选池，从 20 个增加到 100 个
- 提升话题匹配的准确性和相关性
- 保持最终关联话题数量在 1-5 个

**非目标：**
- 不改变话题匹配的 AI 语义分析逻辑
- 不改变发帖流程的其他环节
- 不增加最终关联的话题数量（仍为 1-5 个）

## 决策

### 1. 分页获取策略

**方案：循环获取 5 页数据**
```typescript
export async function fetchHotTopics(token: string): Promise<HotTopic[]> {
  const allTopics: HotTopic[] = [];
  const totalPages = 5;
  
  for (let page = 1; page <= totalPages; page++) {
    const topics = await apiClient.getHotTopics(token, page, 20);
    if (topics.length === 0) break; // 无更多数据时提前退出
    allTopics.push(...topics);
  }
  
  return allTopics;
}
```

**理由：**
- 实现简单，代码改动最小
- 可以控制总数量（5 页 × 20 条/页 = 100 条）
- 失败降级：如果某页获取失败，不影响已获取的话题

**替代方案：**
- 并发获取 5 页：性能更好，但代码复杂度增加，且可能触发 API 限流
- 增加到 10 页：候选更多，但耗时更长，收益递减

### 2. 性能优化

**预期耗时：**
- 当前：1 页 × ~500ms = ~500ms
- 优化后：5 页 × ~500ms = ~2500ms
- 增加约 2 秒，在可接受范围内

**降级策略：**
- 如果某页获取失败，记录错误但不中断
- 返回已成功获取的话题列表
- 如果全部失败，返回空数组（现有降级逻辑处理）

### 3. AI 匹配逻辑

**无需修改**：
- `matchTopics` 函数已经限制最多返回 5 个话题
- AI 会自动从 100 个候选中选择最相关的 1-5 个
- 提示词中已有"最多选择 5 个话题"的约束

## 风险 / 权衡

**已知风险：**
- **API 耗时增加**：5 页获取可能增加 2-3 秒耗时
  - 缓解措施：每页已有 10 秒超时，总耗时可控；发帖是低频操作，用户感知不强
- **API 限流风险**：连续 5 次调用可能触发限流
  - 缓解措施：每页调用独立，失败不影响已获取的数据；发帖频率低（每天几次），限流风险低
- **内存占用略增**：100 个话题对象占用更多内存
  - 缓解措施：话题对象较小（id + name + heatDegree），100 个约几十 KB，影响可忽略

**权衡：**
- **候选数量 vs 耗时**：100 个候选比 20 个匹配更准确，增加 2 秒耗时可以接受
  - 权衡结果：优先保证匹配质量，发帖不是高频操作，2 秒增加可接受
- **并发 vs 串行**：并发获取更快，但串行更稳定
  - 权衡结果：选择串行获取，代码简单，稳定性高

## 迁移计划

**无需迁移**：
- 这是内部优化，调用方式不变
- 发帖流程无需修改
- 配置无需调整
