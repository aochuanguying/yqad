## 上下文

当前仓库已经存在 `docker-deployment` 主规范，但实际代码仍缺少主应用可直接使用的打包文件，`README` 中群晖 Docker 部署章节也还是占位状态。与此同时，素材处理链路在 HEIC/HEIF 转换失败时会退回到 macOS 专属的 `sips`，这意味着本地开发环境可以工作，但群晖 `DS218+ / NAS218+` 这类 Linux Docker 环境无法可靠复用同一条处理路径。

本次变更涉及部署文件、运行时依赖、素材处理兜底逻辑、配置项和部署文档，属于跨模块变更。目标环境明确为群晖 DSM 7 的 Container Manager，CPU 架构为 `linux/amd64`，并且需要保留现有 `sharp` 主处理链路与 JPEG 输出质量配置。

## 目标 / 非目标

**目标：**
- 为主应用提供可直接用于群晖 `DS218+ / NAS218+` 的 Docker 打包与编排文件。
- 使 HEIC/HEIF 图片在 Linux 容器中具备稳定的方案 B 兜底转换能力，即 `sharp` 主链路失败后自动调用 Linux 转换工具。
- 保留现有 JPEG 输出模式，并将图片转换质量、兜底开关和超时等参数纳入配置。
- 明确 `config`、`data`、`logs` 三类持久化目录和 `web.baseUrl` 的部署要求，保证远程图片下载链路可用。
- 提供适用于本机 Mac 构建、群晖运行的操作文档和验证步骤。

**非目标：**
- 不在本次提案中引入新的图片输出格式，例如 WebP 或 AVIF。
- 不改变素材处理任务的业务入口、调度方式或素材索引格式。
- 不为非群晖 Linux 发行版提供额外的发行版级适配矩阵。
- 不处理 ARM 群晖机型的部署兼容性。

## 决策

### 决策 1：基础镜像使用 Debian 系 Node 官方镜像

**选择：** 使用 `node:20-bookworm-slim` 或同级别 Debian slim 镜像，而不是 Alpine。

**理由：**
- 当前主项目依赖 `sharp`，并且需要稳定的 HEIC/HEIF 编解码依赖，Debian/glibc 环境对原生模块和系统工具更友好。
- 相比 Alpine/musl，Debian slim 在群晖 x86_64 场景下更容易获得可复现的运行结果，尤其适合需要叠加命令行图像工具的场景。
- 主项目已经要求 Node 18+，直接提升到 Node 20 可以与现有独立 Dockerfile 的方向保持一致，并获得更稳定的长期支持版本。

**备选方案：**
- `node:18-alpine`：体积较小，但 `sharp` 与 HEIC 相关依赖在 musl 环境下稳定性和调试体验较弱。
- `node:20-alpine`：仍然存在与 musl 相关的运行时不确定性。

**结论：** 采用 Debian slim，优先保证群晖部署成功率和原生依赖稳定性。

### 决策 2：采用方案 B 的双阶段转换链路

**选择：** 维持 `sharp` 为主转换链路；当输入格式为 HEIC/HEIF 且 `sharp` 直接解码失败时，在 Linux 容器中调用 `heif-convert` 生成临时 JPEG，再交由 `sharp` 做最终旋转和 JPEG 质量输出。

**理由：**
- 与当前代码结构最接近，演进成本低，保留现有 JPEG 输出与 EXIF 旋转处理逻辑。
- `heif-convert` 只负责 HEIC/HEIF 兜底，不干扰 JPG/PNG/WebP 等正常路径。
- 相比引入更重的 ImageMagick，全链路成本更低，问题定位也更集中。

**备选方案：**
- 仅依赖 `sharp`：实现最简单，但对目标容器环境的 HEIC 解码能力过于乐观。
- 使用 ImageMagick 作为统一兜底：通用性更强，但镜像更重、命令参数更复杂。
- 在入库前预转换 HEIC：运维简单，但会改变现有使用流程。

**结论：** 采用 `sharp + heif-convert` 的方案 B。

### 决策 3：把 HEIC 兜底参数显式纳入配置

**选择：** 在 `materials.processing` 下增加 HEIC 兜底相关配置，例如是否启用、使用的命令、单文件超时。

**理由：**
- 用户已经明确关注“图片格式转换时的配置”，仅靠硬编码不利于后续排障。
- 群晖设备性能有限，超时和是否启用兜底属于真实运维参数。
- 配置化后可在不重写逻辑的前提下支持未来替换兜底工具。

**备选方案：**
- 完全硬编码：实现最省事，但运维不可见。
- 全部转为环境变量：与项目当前以 YAML 为主的配置模式不一致。

**结论：** 以 YAML 配置为主，必要时允许环境变量补充关键值。

### 决策 4：部署形态以“本机构建 + 群晖运行”为主

**选择：** 文档主路径以 Mac 本机使用 `docker buildx --platform linux/amd64` 构建镜像，再导入或推送到群晖运行；`docker-compose.yml` 面向群晖 Container Manager 直接使用。

**理由：**
- 当前开发环境是 macOS，且群晖 `DS218+` 性能有限，本机跨平台构建更实际。
- 明确 `linux/amd64` 可以避免在 Apple Silicon 本机上错误产出 `arm64` 镜像。
- Compose 文件可以直接成为群晖导入栈的基础。

**备选方案：**
- 在群晖本机直接构建：步骤更少，但构建慢且调试成本高。
- 仅提供 Dockerfile 不提供 Compose：过于原始，不利于复用。

**结论：** 以跨平台构建 + 群晖运行作为推荐路径。

### 决策 5：部署文档必须显式覆盖外部访问与持久化

**选择：** 在部署说明中明确 `config`、`data`、`logs` 卷挂载，要求设置 `web.baseUrl` 为 NAS 可访问地址，并说明 `config/local.yaml` 的推荐用法。

**理由：**
- 当前图片 URL 生成逻辑在未设置 `web.baseUrl` 时会返回相对路径，不适合手机或远端调用。
- token、素材、日志都依赖本地文件系统，若不做卷挂载，容器重建会丢状态。
- 这些要求与代码现状强相关，必须在部署文档中显式说明。

## 风险 / 权衡

- `[Debian 镜像体积更大]` → 接受一定体积增长，换取 `sharp` 与 HEIC 依赖更高的稳定性。
- `[heif-convert 在部分镜像源中包名或依赖变化]` → 在 Dockerfile 中固定 Debian 发行版，并在文档中写明验证命令。
- `[群晖 CPU 性能有限，批量 HEIC 转换耗时较长]` → 提供单文件超时和每轮处理文件数限制，避免长时间阻塞。
- `[本机与群晖平台不一致导致镜像架构错误]` → 部署文档必须强制要求 `--platform linux/amd64`。
- `[配置项变多带来理解成本]` → 保持默认值保守可用，只暴露少量与转换质量、兜底开关、超时直接相关的参数。

## 迁移计划

1. 在根目录补齐 Docker 打包文件，并将其定位为主应用官方部署入口。
2. 扩展素材处理配置结构，加入 HEIC 兜底相关参数与默认值。
3. 调整 HEIC/HEIF 转换逻辑，使 Linux 环境不再依赖 `sips`。
4. 用增量规范更新 Docker 部署要求，并新增图片兜底转换能力规范。
5. 补齐群晖部署文档，包含构建、导入、启动、验证、故障排查与回滚说明。

**回滚策略：**
- 若 Docker 化部署验证失败，可继续沿用现有本机运行方式，不影响当前源代码业务逻辑。
- 若 Linux 兜底转换出现问题，可临时关闭 HEIC 兜底配置，保留非 HEIC 素材链路正常运行。
