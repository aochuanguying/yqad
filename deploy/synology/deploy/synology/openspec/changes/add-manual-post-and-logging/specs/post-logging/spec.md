## 新增需求

### 需求:系统记录每次发帖的完整日志
系统必须记录每次发帖的完整信息，包括发帖时间、触发方式、发帖内容、状态等。

#### 场景:自动发帖完成
- **当** 自动发帖任务完成（无论成功或失败）
- **那么** 系统创建一条日志记录，triggerType 为"auto"

#### 场景:手工发帖完成
- **当** 手工发帖任务完成（无论成功或失败）
- **那么** 系统创建一条日志记录，triggerType 为"manual"

#### 场景:远程 API 发帖回调
- **当** 远程 API 发帖调用确认接口
- **那么** 系统更新或创建日志记录，记录最终状态

### 需求：发帖日志数据结构
发帖日志必须包含以下字段：id、timestamp、triggerType、postType、mode、topicId、topicName、title、content、imageUrls、status、errorMessage、taskId、createdAt。

#### 场景：记录主题发帖
- **当** 使用预配置主题发帖成功
- **那么** 日志记录的 postType 为"topic"，topicId 和 topicName 字段包含对应的主题信息

#### 场景：记录自由发帖
- **当** 使用自由生成模式发帖
- **那么** 日志记录的 postType 为"free"，topicId 和 topicName 字段为空

#### 场景：记录发帖成功
- **当** 发帖成功完成
- **那么** 日志记录的 status 为"success"，包含完整的标题、内容和图片 URL

#### 场景：记录发帖失败
- **当** 发帖执行失败
- **那么** 日志记录的 status 为"failed"，errorMessage 字段包含具体错误信息

### 需求:日志持久化存储
系统必须将发帖日志持久化保存到磁盘文件（data/post-logs.json）。

#### 场景:保存新日志
- **当** 发帖任务完成并生成日志记录
- **那么** 系统将日志追加写入到 post-logs.json 文件

#### 场景:系统重启
- **当** 系统重启后
- **那么** 历史日志记录仍然可从 post-logs.json 读取

### 需求:日志查询 API 接口
系统必须提供日志查询接口，支持分页查询和筛选。

#### 场景:查询日志列表
- **当** 客户端发送 GET 请求到 `/api/posts/logs?page=1&limit=20`
- **那么** 系统返回分页的日志列表，包含总记录数和当前页数据

#### 场景:按触发方式筛选
- **当** 客户端请求 `/api/posts/logs?triggerType=manual`
- **那么** 系统只返回手动触发的发帖日志

#### 场景:查询日志详情
- **当** 客户端请求 `/api/posts/logs/:id`
- **那么** 系统返回指定 ID 的完整日志记录

#### 场景:查询不存在的日志
- **当** 客户端请求不存在的日志 ID
- **那么** 系统返回 404 错误

### 需求:日志查看页面
系统必须在 Web 管理页面提供发帖日志查看功能。

#### 场景:访问日志页面
- **当** 用户点击"📝 发帖日志"标签页
- **那么** 系统加载并显示最近的发帖日志列表

#### 场景:查看日志详情
- **当** 用户点击某条日志
- **那么** 系统展示该日志的完整信息，包括标题、内容、图片等

#### 场景:筛选日志
- **当** 用户选择触发方式筛选（全部/自动/手动）
- **那么** 系统刷新列表，只显示符合条件的日志

#### 场景:分页浏览
- **当** 日志数量超过每页限制
- **那么** 系统显示分页控件，支持翻页查看更多历史日志

### 需求:日志定期清理
系统必须定期清理过期的日志记录，避免文件过大。

#### 场景:清理过期日志
- **当** 日志记录超过 30 天
- **那么** 系统自动清理该记录

#### 场景:保留最近日志
- **当** 日志总数超过 1000 条
- **那么** 系统只保留最近的 1000 条记录
