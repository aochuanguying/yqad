## 修改需求

### 需求：系统必须提供配置管理的 Web 页面

**修改说明**：增加"立即发帖"按钮和"发帖日志"标签页

系统必须提供一个可视化的单页面应用，以表单方式展示和编辑所有配置项。Web 界面无需认证即可直接访问。**在发帖配置页面增加"立即发帖"按钮，并新增"📝 发帖日志"标签页用于查看历史发帖记录。**

#### 场景：访问配置页面

- **当** 用户通过浏览器访问根路径 /
- **那么** 系统必须直接返回配置管理的 HTML 页面，禁止要求任何形式的认证

#### 场景：分组展示配置

- **当** 页面加载完成
- **那么** 页面必须将配置项按以下分组展示：API 设置、🔐 登录、🔑 API Token、AI 模型、评论、发帖、调度、📷 素材库、📝 发帖主题、📝 发帖日志

#### 场景：立即发帖按钮

- **当** 用户访问"发帖"标签页
- **那么** 页面必须在"保存配置"按钮左侧显示"⚡ 立即发帖"按钮

#### 场景：点击立即发帖按钮

- **当** 用户点击"立即发帖"按钮
- **那么** 系统必须调用 `POST /api/posts/execute` 接口，并显示执行结果（成功或失败）

#### 场景：访问发帖日志页面

- **当** 用户点击"📝 发帖日志"标签页
- **那么** 系统必须加载并显示最近的发帖日志列表，包含时间、触发方式、标题、状态等信息

#### 场景：日志列表分页

- **当** 日志数量超过每页限制（20 条）
- **那么** 页面必须显示分页控件，支持翻页查看更多历史日志

#### 场景：日志筛选

- **当** 用户在日志页面选择触发方式筛选（全部/自动/手动）
- **那么** 系统必须刷新列表，只显示符合条件的日志

#### 场景：查看日志详情

- **当** 用户点击某条日志记录
- **那么** 系统必须展示该日志的完整信息，包括标题、内容、图片 URL 等
