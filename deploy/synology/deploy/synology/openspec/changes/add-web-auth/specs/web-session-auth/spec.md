## 新增需求

### 需求：用户必须通过登录认证才能访问 Web 管理界面
系统应当提供基于 Session 的登录认证机制，管理员必须使用正确的用户名和密码登录才能访问受保护的资源。

#### 场景：用户使用正确凭据登录成功
- **当** 用户在登录页面输入正确的用户名和密码
- **那么** 系统创建有效的 Session 并重定向到管理界面首页

#### 场景：用户使用错误凭据登录失败
- **当** 用户输入错误的用户名或密码
- **那么** 系统返回错误提示"用户名或密码错误"，不创建 Session

#### 场景：未登录用户访问受保护资源
- **当** 未登录用户尝试访问受保护的 API 或页面
- **那么** 系统返回 401 状态码（API）或重定向到登录页（页面）

#### 场景：已登录用户访问登录页
- **当** 已登录的用户尝试访问登录页
- **那么** 系统自动重定向到管理界面首页

#### 场景：用户登出
- **当** 已登录用户点击"退出登录"按钮
- **那么** 系统销毁 Session 并返回登录页

### 需求：Session 必须具有过期机制
系统应当为每个 Session 设置最大存活时间，超时后自动失效，用户需要重新登录。

#### 场景：Session 在有效期内
- **当** 用户在 Session 有效期内发起请求
- **那么** 系统正常处理请求并刷新 Session 有效期

#### 场景：Session 过期
- **当** Session 超过配置的最大存活时间
- **那么** 系统自动销毁该 Session，用户需要重新登录

#### 场景：Session 配置
- **当** 管理员在配置中设置 `sessionMaxAge`
- **那么** 系统使用配置的值作为 Session 最大存活时间（毫秒）

### 需求：登录页面应当提供友好的用户界面
系统应当提供简洁、直观的登录页面，支持表单验证和错误提示。

#### 场景：访问登录页面
- **当** 用户访问 `/login.html`
- **那么** 系统返回包含用户名和密码输入框的登录表单

#### 场景：表单验证
- **当** 用户提交空的用户名或密码
- **那么** 系统在客户端提示"用户名和密码不能为空"

#### 场景：登录错误提示
- **当** 登录失败时
- **那么** 系统在登录页面显示错误原因（如"用户名或密码错误"）
