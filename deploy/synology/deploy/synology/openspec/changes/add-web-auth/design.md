## 上下文

当前项目已有基于 Token 的 API 认证机制（用于远程发帖 API），但 Web 管理界面本身没有访问控制。项目使用 Express 作为 Web 框架，已有中间件架构（error-handler, json-body-parser）。配置系统支持 YAML 配置文件和环境变量。

**现状：**
- Web 界面完全公开，无任何访问限制
- 已有 API Token 机制（`src/utils/api-token.ts`），但仅用于发帖 API
- 使用 express 作为 Web 框架
- 配置系统支持 config/default.yaml 和环境变量

**约束：**
- 不能影响现有远程发帖 API 的 Token 认证机制
- 需要支持 Docker 部署环境
- 配置需要支持环境变量（便于容器化部署）
- 尽量使用成熟库，减少自行实现加密逻辑

## 目标 / 非目标

**目标：**
- 为 Web 管理界面添加基于 Session 的登录认证
- 保护所有 API 端点（除登录接口和远程发帖 API）
- 支持管理员账号配置（用户名 + 密码）
- 提供友好的登录界面和登出功能
- 支持会话过期和刷新机制

**非目标：**
- 不支持多用户/角色系统（仅单管理员账号）
- 不涉及 OAuth/SSO 等复杂认证方式
- 不修改现有远程发帖 API 的 Token 认证逻辑
- 不实现密码找回、邮箱验证等功能

## 决策

### 1. 认证方案选择：Session vs JWT

**决策：** 使用基于 Cookie 的 Session 认证

**理由：**
- Web 界面是单用户管理后台，不需要 JWT 的跨域/无状态特性
- Session 方案更简单，易于实现登出功能
- 避免 XSS 攻击导致 Token 泄露的风险
- Express 有成熟的 `express-session` 中间件支持

**考虑过的替代方案：**
- JWT：更适合多服务/跨域场景，但增加复杂性（Token 存储、刷新机制）
- Basic Auth：太简陋，无会话管理，每次请求都需传输密码

### 2. Session 存储方案

**决策：** 使用内存存储（MemoryStore）

**理由：**
- 单实例部署，无集群需求
- 简化依赖和配置
- 重启后会话失效可接受

**考虑过的替代方案：**
- Redis：支持分布式，但增加外部依赖
- 数据库：增加复杂度，无必要

### 3. 密码存储方案

**决策：** 使用 bcrypt 进行密码哈希存储

**理由：**
- 业界标准，安全性高
- 防止彩虹表攻击
- Node.js 生态成熟（`bcryptjs` 库）

**考虑过的替代方案：**
- 明文存储：绝对不可接受
- SHA256：不适合密码存储（无盐值/计算太快）

### 4. 配置方式

**决策：** 支持配置文件和环境变量两种方式

```yaml
# config/default.yaml
web:
  auth:
    enabled: true
    username: admin
    passwordHash: $2b$10$...  # bcrypt 哈希
    sessionSecret: your-secret-key  # 或使用环境变量
    sessionMaxAge: 86400000  # 24 小时
```

**理由：**
- 开发环境可用配置文件
- 生产环境（Docker）使用环境变量
- 密码以哈希形式存储，避免明文泄露风险

### 5. API 保护范围

**决策：** 默认保护所有 `/api/*` 路由，显式排除公开接口

**需要公开的接口：**
- `/api/auth/login` - 登录接口
- `/api/posts/token/generate` - 远程发帖 API（已有独立 Token 认证）
- `/api/posts/token` - Token 状态查询（已有独立 Token 认证）

**实现方式：**
- 创建 `authMiddleware` 应用于所有 API 路由
- 登录路由使用 `publicRoutes` 白名单机制

### 6. 登录界面实现

**决策：** 复用现有 Web 界面风格，新增登录 Tab 或独立登录页

**方案 A（推荐）：** 独立登录页 `/login.html`
- 未登录时访问任何受保护资源自动跳转
- 登录成功后跳回原页���

**方案 B：** 在现有界面增加登录 Tab
- 用户已习惯当前 Tab 布局
- 但登录状态判断复杂

**选择：** 方案 A，更清晰的用户体验

## 风险 / 权衡

### [风险] Session 内存存储在重启后丢失
**缓解措施：**
- 单用户场景可接受
- 重启后重新登录即可
- 如需要持久化，可切换到 Redis Store

### [风险] 密码哈希计算增加登录延迟
**缓解措施：**
- bcrypt 的 cost factor 设为 10（约 100ms）
- 仅登录时计算一次，不影响后续请求
- 可接受的安全代价

### [风险] HTTPS 缺失导致 Cookie 劫持
**缓解措施：**
- 文档中强调生产环境必须使用 HTTPS
- 配置 `cookie.secure: true` 强制 HTTPS
- 内网部署风险较低

### [风险] 暴力破解密码
**缓解措施：**
- 增加登录失败次数限制（可选）
- 使用强密码策略（文档建议）
- 配置文件权限设为 600

## 迁移计划

### 部署步骤

1. **安装依赖**
   ```bash
   npm install express-session bcryptjs
   npm install -D @types/express-session @types/bcryptjs
   ```

2. **配置管理员账号**
   - 生成密码哈希（提供脚本或在线工具）
   - 更新配置文件或设置环境变量

3. **更新代码**
   - 应用所有代码变更
   - 编译 TypeScript

4. **重启服务**
   - 停止旧版本
   - 启动新版本
   - 验证登录功能

### 回滚策略

- 保留旧版本代码
- 如登录功能异常，可临时禁用认证（`web.auth.enabled: false`）
- 回滚到无认证版本（不推荐，仅应急）

## 开放问题

1. **是否需要支持多管理员账号？**
   - 当前需求是单用户，暂不考虑

2. **是否需要登录失败锁定机制？**
   - 第一版先不实现，根据实际使用情况决定

3. **Session 过期时间设置多久？**
   - 默认 24 小时，可配置
   - 平衡安全性和便利性
